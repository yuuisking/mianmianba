# 复盘中心开发模型硬性实现要求清单

## 1. 文档目的

本文件不是讨论方向，也不是做阶段规划，而是直接规定：

> 复盘中心必须被实现成什么样，哪些事情必须做到，哪些做法不允许继续存在。

任何模型、Agent 或开发窗口在执行复盘中心相关开发时，都必须以本清单为硬约束，而不是继续把复盘中心做成“报告展示页”。

---

## 2. 最终产品定义

复盘中心必须被实现为：

> 一个围绕用户能力成长闭环构建的数据分析中台。

它必须同时具备以下能力：

1. 洞察能力：明确指出当前最重要的问题和最值得优先处理的事项。
2. 诊断能力：解释问题为什么出现、为什么反复出现、它会影响什么。
3. 证据能力：任何结论都必须能追溯到真实样本、真实回答和真实评分证据。
4. 动作能力：把问题转成明确、可执行、可验证的训练动作。
5. 改善验证能力：训练完成后，系统必须能判断问题是否真的被压下去。

如果缺少上述任一项，复盘中心都不能算完成。

---

## 3. 禁止继续采用的实现方式

以下做法禁止继续作为复盘中心的核心实现方式：

1. 只聚合旧 `Report` 的文本字段来生成洞察。
2. 只有“结论文案”，没有对应证据台。
3. 只有“问题列表”，没有根因分析。
4. 只有“建议按钮”，没有动作说明、完成标准和验证标准。
5. 只有“多 Agent 介绍卡”，没有真实 Agent 输入输出链路。
6. 只做页面展示，不沉淀中间分析实体。
7. 把 `0 轮对话`、中断样本、无报告样本和有效样本混在同一个主视图里。
8. 从复盘页跳转训练时，只传文本 topic，不传岗位、场景、目标、失败样本、训练目标等完整上下文。
9. 任何结论无法说明样本数、时间范围、覆盖范围和置信度。

---

## 4. 必须达到的页面能力

复盘中心页面必须包含以下 6 个能力区块。

## 4.1 总览区

必须回答：

1. 当前最大问题是什么。
2. 当前最明显进步是什么。
3. 今天最值得做的动作是什么。
4. 当前结论是否可信。

必须包含：

- 核心结论
- 今日行动建议
- 样本数
- 时间范围
- 置信度

不允许只有一句 headline 而没有可信度信息。

## 4.2 问题诊断区

必须展示问题树，而不是单层列表。

问题至少要拆成三层：

- 能力维度
- 根因
- 影响

每个问题卡必须包含：

- 问题名
- 严重度
- 最近出现次数
- 最近一次出现时间
- 根因解释
- 影响描述
- 推荐动作入口

## 4.3 证据区

任何问题都必须能展开到证据。

每条证据必须至少包含：

- 来源会话
- 面试类型
- 岗位 / 公司 / 轮次
- 问题标题
- 用户回答摘录
- 命中的问题标签
- 为什么命中
- 推荐改写或更优表达

没有证据的结论不允许进入最终用户视图。

## 4.4 动作区

每条建议都必须是一张任务卡，而不是一个单纯跳转按钮。

必须包含：

- 练什么
- 为什么练这个
- 推荐训练方式
- 完成标准
- 验证标准
- 动作入口

## 4.5 改善验证区

必须展示：

- 哪些问题下降了
- 哪些问题反弹了
- 哪些建议已执行
- 哪些建议有效
- 哪些建议无效

如果当前没有足够样本，也必须明确告诉用户“当前无法判断改善效果”，而不是静默不展示。

## 4.6 数据探索区

必须支持以下筛选：

- 时间范围
- 面试类型
- 岗位
- 公司
- 维度
- 样本状态

至少支持从“总览 -> 问题 -> 证据 -> 动作”逐层下钻。

---

## 5. 必须达到的数据能力

复盘中心不能继续只消费旧报表文本，必须具备独立的数据分析能力。

必须直接接入的原始数据源包括：

- `interviewSession`
- `messages`
- `report`
- `weakness`
- `learningProgress`
- `documentInterviewSession`
- `userAnswerScore`

必须做到：

1. 区分有效样本与无效样本。
2. 有效样本单独入分析池。
3. 无效样本单独归档并标注原因。
4. 复盘结论必须建立在有效样本上。

---

## 6. 必须沉淀的分析实体

复盘中心不能只在请求时临时拼 JSON，必须沉淀结构化中间结果。

至少要有以下几类实体或等价结构：

1. `ReviewSnapshot`
2. `ReviewIssue`
3. `ReviewEvidence`
4. `ReviewAction`
5. `ReviewActionExecution`

每一类实体都必须能被追踪、回查和复用，不能只存在于页面渲染层。

---

## 7. 必须实现的多 Agent 架构

复盘中心必须使用真实多 Agent 工作流，而不是概念展示。

至少必须实现以下角色：

1. `Orchestrator Agent`
2. `Signal Agent`
3. `Evidence Agent`
4. `Diagnosis Agent`
5. `Strategy Agent`
6. `Drill Agent`
7. `Progress Agent`
8. `Narrative Agent`

### 每个 Agent 必须满足的约束

每个 Agent 都必须明确：

- 输入
- 输出
- 不负责什么
- 错误降级策略

### 不允许的做法

- 一个 Agent 包揽所有事情
- 页面文案里写了 Agent 名字，但服务端没有对应执行结果
- Agent 只返回自然语言，不返回结构化结果

---

## 8. Agent 协作必须满足的顺序

复盘链路必须按以下顺序执行：

1. `Orchestrator Agent` 接收复盘请求
2. `Signal Agent` 清洗样本
3. `Evidence Agent` 抽取证据
4. `Diagnosis Agent` 生成问题树
5. `Strategy Agent` 生成动作单
6. `Drill Agent` 生成个性化训练任务
7. `Progress Agent` 判断建议有效性
8. `Narrative Agent` 组织用户视图
9. `Orchestrator Agent` 合并生成最终 dashboard snapshot

页面层不允许承担核心推理职责。

---

## 9. 复盘到训练的闭环必须成立

从复盘中心点击任意“立即补强 / 专项突破 / 去行动”时，必须带上完整上下文，而不是只带一个 topic 文本。

至少必须带上：

- 用户目标岗位
- 用户目标公司
- 用户目标级别
- 当前问题 ID
- 当前问题描述
- 推荐训练方式
- 推荐题型
- 推荐难度
- 最近失败样本摘要
- 训练目标
- 验证标准

如果进入训练页后仍提示“等待补充岗位信息”，则判定闭环未完成。

---

## 10. 历史记录必须重做

历史记录区不能继续作为简单日志列表。

必须做到：

1. 默认展示有效训练记录。
2. 无效记录单独归档。
3. 无效记录必须带原因。
4. 有效记录必须支持查看关键节点和报告入口。
5. 支持同类问题前后对比。

以下内容不允许继续直接混在主列表中：

- `0 轮对话`
- 无有效回答
- 无报告但未标注原因

---

## 11. 可信度表达必须上线

每条核心洞察都必须带：

- 样本数
- 时间范围
- 覆盖场景
- 置信等级

如果当前样本不足，系统必须明确表达：

- 当前只能给弱结论
- 当前不能给稳定结论
- 当前需要更多样本

不允许在样本不足时仍给出强结论口吻。

---

## 12. 页面交互必须达到的标准

必须支持：

1. 问题卡展开证据
2. 证据回看原始上下文
3. 从问题卡直接进入定向训练
4. 从动作卡看到完成标准
5. 从改善区查看前后对比
6. 从筛选器切换时间 / 岗位 / 类型维度

不允许只有“看板式浏览”，没有下钻能力。

---

## 13. 用户价值必须可被证明

复盘中心的交付标准不是“页面看起来高级”，而是下面这些问题能被回答：

1. 用户是否知道自己当前最大的拖分项。
2. 用户是否知道这个问题为什么会发生。
3. 用户是否能看到支持结论的真实证据。
4. 用户是否知道下一步最值得做的动作。
5. 系统是否能判断这个动作做完后有没有效果。

只要有任意一个问题答不上来，复盘中心就不算真正完成。

---

## 14. 交付验收标准

复盘中心只有同时满足以下条件才算完成：

1. 不再依赖旧 `Report` 文本聚合作为唯一分析来源。
2. 页面能展示问题树，而不是只有问题列表。
3. 任意关键问题都能展开证据台。
4. 任意建议都带完整动作说明、完成标准和验证标准。
5. 从复盘到训练的上下文链路完整打通。
6. 可展示建议是否有效的改善验证结果。
7. 多 Agent 是真实执行链，而不是概念展示。
8. 支持时间、岗位、类型等基础筛选和下钻。
9. 历史区不再把无效样本和有效样本混在一起。
10. 所有关键洞察都带样本范围和置信度。

---

## 15. 最终执行口径

模型在实现复盘中心时，不允许再把目标理解成：

- 优化文案
- 美化看板
- 补几张卡片
- 再写一层总结

必须理解成：

> 交付一个真正围绕用户成长闭环运作的数据分析中台。

如果实现结果仍然只是“更好看的报告页”，则视为未完成。
