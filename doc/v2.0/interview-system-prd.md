# 面面吧产品 PRD（V1.0）

## 1. 文档目的

本文档用于定义“面面吧”从当前版本升级为“面试能力操作系统”的产品方向、核心壁垒、关键模块、信息架构、用户路径、增长与运营策略。

文档基于以下实际体验与观察形成：

- 首页已经出现“当前薄弱点 Top 3”和直接回流动作，说明产品已具备闭环雏形。
- 专项训练支持一句话目标进入多轮追问，且报告会产出“结构化评估报告 / 多 Agent 评审团 / 下一步训练计划”。
- 复盘中心支持“查看诊断 / 开始计划”，并能把问题、岗位、公司、级别和训练目标回写到训练入口。
- 全流程面试模式不是假入口，网络层已经有 `interview-plans / stage / round / experience-tasks / parse` 等实体，说明产品天然适合做流程化、状态化的面试系统。

---

## 2. 产品一句话定义

面面吧不是“会问你面试题的 AI”，而是一个围绕目标岗位，持续完成“诊断 -> 追问 -> 证据评估 -> 纠偏训练 -> 学习补强 -> 再验证 -> Offer Ready 判断”的面试能力操作系统。

---

## 3. 当前产品判断

## 3.1 当前最有价值的能力

- 不是单轮问答，而是闭环训练。
- 不是只出报告，而是能把报告回流成行动。
- 不是抽象鼓励，而是开始有能力维度、问题画像和补强动作。
- 不是单纯训练入口，而是开始具备全流程面试计划能力。

## 3.2 当前最核心的产品优势

- **闭环感**：从模拟/专项到复盘再到行动计划，链路比普通 AI 问答工具完整。
- **训练感**：系统有追问，且追问有一定针对性，不只是同义改写。
- **可信倾向**：当无有效回答时，会拒绝伪造正式报告，这是建立信任的关键。
- **流程化潜力**：全流程模式已经有计划、阶段、轮次等实体，适合向更高壁垒产品演进。

## 3.3 当前最关键的问题

- 报告还不够“证据化”，缺少逐句引用与可追溯解释。
- 首页、专项、全流程、复盘之间的统一主线还不够强。
- 用户还看不清“我是否真的变强了”。
- 产品现在容易被误判成“高级 Prompt 产品”，而不是“面试教练系统”。
- 全流程模式在交互文案上仍然偏“专项训练感”，未完全体现多轮流程推进的独特价值。

## 4. 目标用户与扩展边界

## 4.1 第一目标用户

- 1-3 年工作经验的前端 / 后端工程师
- 典型痛点：
  - 面试不稳定，答题常常“懂一点但讲不完整”
  - 缺少真实追问场景
  - 知道自己不够强，但不知道问题到底在哪里
  - 刷题很多，但没有形成岗位化表达能力

## 4.2 第二目标用户

- 校招技术岗学生
- 典型痛点：
  - 缺真实面试场景
  - 不知道目标岗位的能力权重
  - 只会背八股，缺乏结构化表达和追问承压能力

## 4.3 当前不作为主打入口的人群

- 泛职场用户
- 非求职导向的学习型用户
- 没有明确目标岗位与目标公司的泛流量人群

### 为什么当前不能平均铺开

不是因为别的场景不重要，而是因为一开始最缺的不是“页面”，而是：

- 统一的评估标准
- 岗位能力图谱
- 追问策略
- 高质量案例库
- 训练处方系统

如果这些底层内核还没有打透，就同时做前端、后端、测试、产品、运营、数据开发，只会得到很多入口、很多页面、很多题目，但没有一个方向真正做到强。

### 例子

- 如果现在同时做“前端 + 后端 + 测试 + 产品经理”，表面上模块更多了，但每个岗位背后的能力维度都不一样，评分标准会迅速失真。
- 例如前端更强调：
  - 性能
  - 工程化
  - 浏览器机制
- 测试更强调：
  - 测试设计
  - 缺陷分析
  - 自动化策略
- 产品经理更强调：
  - 需求拆解
  - 优先级判断
  - 指标定义

如果底层不抽象出“通用能力内核 + 岗位适配器”，系统会越做越散。

### 如何判断已经准备好扩场景

当以下条件同时成立时，说明可以快速接入新场景：

- 已有稳定的通用能力内核
- 已有结构化评分标准库
- 已有可复用的追问编排器
- 已有证据抽取与校准能力
- 已有内容生产、报告生成、行动回流的完整链路

### 什么叫“通用能力内核”

指那些跨岗位都成立的基础维度：

- 结构化表达
- 证据充分性
- 问题拆解
- 方案权衡
- 结果导向
- 沟通清晰度
- 真实性

这些维度一旦成熟，新岗位接入时只需要加“岗位适配器”。

### 通用能力内核怎么具体做

不是先写一堆抽象维度名字，而是要把每个通用维度做成一套可执行的标准件。至少要落成 6 层：

- `维度字典层`
  - 明确定义每个维度到底在判断什么。
  - 例如“结构化表达”不是“说得顺”，而是是否具备背景、目标、动作、结果的组织能力。
  - 例如“证据充分性”不是“提到数据就行”，而是是否给出可验证事实、指标、过程、结果。
- `评分标准层`
  - 每个维度都要有低分、中分、高分标准。
  - 例如“问题拆解”可以定义：
  - 低分：上来直接回答，没有拆边界、没有分步骤。
  - 中分：能拆步骤，但缺少优先级与约束。
  - 高分：能拆边界、识别约束、说明优先级与取舍。
- `证据抽取层`
  - 系统不是只看整段回答感觉像不像，而是要能抽取证据片段。
  - 例如用户说“我把接口成功率从 92% 提升到 99.5%”，这就是结果导向与证据充分性的强证据。
  - 例如用户说“我做了很多优化，效果不错”，这几乎不能算有效证据。
- `追问编排层`
  - 通用内核必须能针对通用问题做追问。
  - 如果发现“真实性”不足，就追问“这部分是你主导还是参与？你的具体动作是什么？”
  - 如果发现“权衡”不足，就追问“你为什么选 A 不选 B，当时约束是什么？”
- `报告映射层`
  - 每个维度都要能映射到报告中的亮点、问题、风险、建议。
  - 否则前面评了很多，最后报告只剩一句“继续努力”，价值会断掉。
- `行动回流层`
  - 每个通用维度都要能回流成训练动作。
  - 例如“结构化表达差”进入同题重答。
  - 例如“证据不足”进入项目举证专项训练。
  - 例如“权衡不清”进入方案取舍训练。

### 例子：把“结构化表达”做成真正可复用的通用内核

如果系统想把“结构化表达”做成所有岗位都能复用的内核，不是只在文档里写一句“要有逻辑”，而是要同时落地：

- 统一定义
  - 回答是否具备清晰骨架。
- 通用识别规则
  - 是否有背景。
  - 是否有目标。
  - 是否有动作。
  - 是否有结果。
- 通用追问模板
  - “你刚才描述了做法，但背景是什么？”
  - “结果如何证明？”
  - “这里你个人起到什么作用？”
- 通用报告话术
  - “你的回答能说出动作，但背景和结果缺失，导致面试官难以快速理解价值。”
- 通用训练动作
  - 同题重答
  - STAR 改写
  - 高分案例对照学习

这样以后不管是前端、测试、产品还是运营，只要用户回答没结构，都能直接复用这一整套能力链路。

### 什么叫“岗位适配器”

就是每个新岗位只新增：

- 岗位能力权重
- 岗位题型
- 岗位案例库
- 岗位术语
- 岗位通过标准

### 岗位适配器怎么具体做

岗位适配器不是给每个岗位重新造一套产品，而是在通用能力内核之上，补一层“岗位特有配置 + 岗位特有内容 + 岗位特有标准”。

一个完整的岗位适配器，至少包含 7 个部分：

- `能力权重配置`
  - 决定这个岗位更看重哪些通用维度，哪些岗位维度。
  - 例如前端更重“技术深度、性能、工程化、表达”。
  - 产品经理更重“需求判断、用户价值、优先级、表达、权衡”。
- `岗位专属维度`
  - 通用维度不够时，新增岗位专属维度。
  - 例如测试岗位新增“缺陷定位能力”“用例覆盖意识”。
  - 例如运营岗位新增“转化漏斗理解”“活动策略判断”。
- `题型适配`
  - 同样是“拆解能力”，前端题会是“性能排查怎么拆”，产品题会是“需求方案怎么拆”，测试题会是“测试计划怎么拆”。
- `术语与语境适配`
  - 系统的题目、追问、报告话术要切到该岗位真实语境。
  - 前端说的是 LCP、缓存、工程化。
  - 产品说的是 DAU、转化、优先级、用户价值。
- `案例库适配`
  - 高分案例、低分反例、追问样本、学习材料都必须换成该岗位语境。
  - 否则用户会觉得“这套系统只适合前端，迁移到别的岗位就是硬套壳”。
- `通过标准适配`
  - 不同岗位、不同级别、不同公司，对通过线要求不同。
  - 例如校招生产品经理和 5 年经验产品经理，题型相似，但通过标准完全不同。
- `行动方案适配`
  - 即便同样都是“证据不足”，前端可能要去补“项目指标与技术方案证据”，产品可能要补“用户研究依据和指标结果”。

### 例子：同一个通用维度，如何通过适配器变成不同岗位能力

以“问题拆解”这个通用内核为例：

- 放到前端岗位
  - 题目会变成“线上页面白屏，你怎么排查？”
  - 看的是链路拆解、日志定位、前后端边界、优先级判断。
- 放到测试岗位
  - 题目会变成“一个支付接口频繁失败，你怎么设计排查与验证方案？”
  - 看的是异常分类、复现路径、测试覆盖、环境隔离。
- 放到产品岗位
  - 题目会变成“留存下降，你如何拆解原因并决定下一步动作？”
  - 看的是用户分层、漏斗定位、假设验证、指标设计。

所以真正复用的不是题目文本，而是底层判断能力；真正变化的是岗位语境、题型、证据形态与通过标准。

### 例子：如何快速接入“测试工程师”

通用能力内核仍然保留：

- 结构化表达
- 证据
- 拆解
- 权衡

然后增加测试岗位适配器：

- 缺陷定位能力
- 用例设计完整性
- 自动化策略

题目就从“前端性能排障题”切换为“接口稳定性测试设计”“回归策略设计”“线上缺陷复盘”。

通用能力内核仍然保留：

- 结构化表达
- 逻辑拆解
- 权衡

再增加产品岗位适配器：

- 需求判断
- 用户价值分析
- 优先级排序
- 指标设计

题目会切换成“新功能上线决策”“增长方案评估”“需求冲突协调”。

## 5. 产品北极星目标

帮助用户在目标岗位面试中更稳定地通过真实轮次，并且让用户能清楚看到自己的能力缺口、训练路径和提升证据。

---

## 6. 北极星指标与核心指标

## 6.1 北极星指标

- **有效训练闭环完成率**
  - 定义：用户完成一场训练/模拟后，查看报告，并进入下一步动作（改写 / 重答 / 学习 / 开始计划）的比例

## 6.2 核心体验指标

- 报告后二次动作转化率
- 同题重答提升率
- 7 日复训率
- 14 日留存
- 用户主观准确度评分
- 报告建议采纳率

## 6.3 业务指标

- 首次付费转化率
- 7 日付费留存
- 用户推荐率
- 按岗位细分的训练完成率

## 7. 产品战略总定位

面面吧的战略定位不应拆成时间表达，而应统一描述成一个由浅到深的能力体系：

- 第一层：比普通 AI 面试工具更会指出问题
- 第二层：比普通 AI 面试工具更会安排训练
- 第三层：比普通 AI 面试工具更会证明成长
- 第四层：最终成为岗位化、流程化、可持续演进的面试能力操作系统

### 统一定位表达

面面吧是一个围绕目标岗位、目标公司和真实轮次，持续完成“建模、追问、评估、复盘、训练、学习、成长证明、Offer Ready 判断”的面试能力操作系统。

### 例子

- 如果用户是“字节前端 3 年经验”：
  - 系统不是只问几道题
  - 而是先建模这个岗位
  - 再进入一面 / 二面 / 深挖 / 压力追问
  - 再输出证据型报告
  - 再安排针对性的专项训练
  - 最后告诉用户距离目标岗位 readiness 还差什么

- 如果用户以后切换为“测试工程师”：
  - 系统主框架不变
  - 变的是岗位适配器、题型、维度权重和案例库

### 为什么要用“总定位”而不是时间定位

因为你的产品不是分阶段变成另一个产品，而是在不断加深同一个系统：

- 同一个用户画像
- 同一套评估内核
- 同一条训练闭环
- 同一个成长档案
- 同一个 Offer Ready 判断

---

## 8. 不可复制的核心竞争力设计

## 8.1 结论

你的壁垒不能建立在“界面”和“Prompt”上，而必须建立在四层资产上：

- 评估标准资产
- 数据资产
- 训练路径资产
- 效果证明资产

## 8.2 四层壁垒

### 第一层：评估标准资产

沉淀每个岗位、每个级别、每个能力维度的：

- 定义
- 高分标准
- 低分反例
- 常见伪高分表现
- 面试官真实追问路径
- 分数映射规则

这是所有 Agent、报告、计划的共同底座。

### 第二层：行为与证据数据资产

沉淀以下结构化数据：

- 候选人回答原文
- 证据片段
- 维度评分
- Judge 分歧结果
- 追问分叉路径
- 同题重答前后差异
- 学习补强后效果

越多用户使用，系统越准，外部越难追。

### 第三层：训练处方资产

不是简单推荐内容，而是形成：

- 某种问题应该先练什么
- 练几轮
- 怎么判断练会了
- 应该补什么知识点
- 先做专项训练还是先看知识卡

这是产品从“报告工具”升级为“教练系统”的关键。

### 第四层：效果证明资产

用户必须能看到：

- 哪个维度提升了
- 为什么提升了
- 哪些问题还没修掉
- 当前状态距离目标岗位还有多远

只有可证明的成长，才能带来留存和品牌。

---

## 9. 产品核心主线

统一主线必须是：

`目标岗位建模 -> 基线诊断 -> 多轮训练 -> 结构化评估 -> 行动处方 -> 学习补强 -> 再验证 -> Offer Ready`

所有模块都必须为这条主线服务。

## 10. 核心模块设计

## 10.1 首页：今日最该做的动作

### 当前问题

- 首页仍然偏模块导航。
- 用户需要自己决定“先干什么”。

### 现状评分（满分 10 分）

- `6.5/10`
- 原因：已经有薄弱点和继续动作雏形，但仍偏“展示已有功能”，不够像“训练调度台”。

### 目标

把首页从“模块入口页”升级为“今日训练控制台”。

### 首页核心区域

- 今日优先补强 1 项
- 最近一次训练结论
- 下一步最优动作
- 本周能力变化
- 距目标岗位 readiness 变化

### 首页核心动作

- 8 分钟纠偏训练
- 同题重答
- 先补一张知识卡
- 开启下一轮流程面试

### 例子

用户昨天完成了一场“字节前端全流程一面”后，首页不该只展示：

- 去训练
- 去学习
- 去复盘

而应直接展示：

- 你当前最影响通过率的问题：结构化表达不稳定
- 建议先做：8 分钟同题重答
- 做完再看：React 关键性能指标案例卡
- 然后继续：下一轮项目深挖

## 10.2 发起面试：从配置页升级为岗位建模入口

### 阶段面试

适合用户主动配置：

- 岗位
- 公司
- JD
- 简历
- 风格
- 模式
- 面试方向

### 现状评分（满分 10 分）

- `7.2/10`
- 原因：配置项已经比较完整，真实感不错，但还偏“让用户填表”，缺少更强的岗位建模反馈。

### 全流程面试

根据实测，它已经具备以下雏形：

- 全流程列表
- 发起计划
- 公开面经采集
- 基于岗位、公司、JD、简历生成计划
- `planId / stageId / roundId`

### 现状评分（满分 10 分）

- `6.8/10`
- 原因：底层流程结构已经出现，产品潜力很强，但前台表达还没有完全把“流程系统”讲清楚。

### 建议升级

全流程模式不应只是另一种“发起面试页”，而应明确展示：

- 目标公司真实流程预估
- 当前处于哪个轮次
- 下一轮面试主题
- 每轮目标能力
- 轮次通过条件
- 本轮由谁来面
- 本轮评审团如何投票
- 预计节奏

### 详细解释

“全流程面试”不应该只是把普通面试配置多包一层壳，而应该让用户明确知道自己正在准备一个“流程”，不是单轮问答。

普通发起面试的关注点是：

- 我现在想练什么
- 我想问哪些题

全流程面试的关注点应该是：

- 目标公司通常有几轮
- 当前这一轮更像哪种真实轮次
- 这一轮会重点考什么
- 这一轮的“面试官画像”是什么
- 通过这一轮需要达到什么水平
- 要获得多少票才能进入下一轮
- 如果没过，最可能死在哪

### 例子：字节跳动前端 3 年经验全流程

用户创建计划后，页面不应只显示“开始面试”，而应显示：

- 目标公司：字节跳动
- 目标岗位：前端开发工程师
- 当前流程：
  - 一面：基础能力 + 项目深挖
  - 二面：复杂场景设计 + 性能与工程化
  - 三面：业务判断 + 团队协作
  - 四面：团队 leader 判断
  - 五面：HR 决策面
- 当前所处轮次：一面
- 当前面试官：
  - 团队主干 1：偏项目深挖
- 后续预置面试官链路：
  - 二面：团队主干 2
  - 三面：团队主管
  - 四面：团队 leader
  - 五面：HR
- 当前轮次目标：
  - 能解释真实项目
  - 能说清性能优化权衡
  - 能在追问下保持结构稳定
- 当前轮次通过条件：
  - 结构化表达 >= 某阈值
  - 技术深度 >= 某阈值
  - 项目真实性判断通过
- 当前投票规则：
  - 评审团 5 票
  - 至少 3 票通过才算本轮通过
- 评审团构成：
  - 技术深度评审
  - 表达结构评审
  - 项目真实性评审
  - 权衡能力评审
  - 风险校准评审
- 当前风险：
  - 你最容易在“项目真实性深挖”环节被卡住

### 发起面试后的进度条设计

用户在全流程面试里点击“发起面试”后，不应该直接白屏等待或只看到一个 loading。应该进入一个明确的初始化进度条页面，让用户知道系统正在做什么。

建议最少展示下面这些步骤：

- 岗位确认中
- 获取最新面经中
- 公司流程解析中
- 面试官阵列生成中
- 评审团规则装载中
- 流程初始化中
- 首轮问题准备中
- 即将进入面试间

### 为什么这个进度条很重要

- 它不是装饰，而是降低等待焦虑。
- 用户知道系统不是卡住了，而是在真实做准备动作。
- 每一步都在强化“这不是普通聊天，而是在初始化一场流程面试”。

### 例子：用户点击“发起面试”后的体验

用户点击 `发起面试` 后，页面进入一个全屏流程初始化态：

- `岗位确认中`
  - 系统正在读取岗位、公司、年限、目标部门。
- `获取最新面经中`
  - 系统正在拉取并筛选最新公开面经模式。
- `公司流程解析中`
  - 系统正在推断常见轮次结构。
- `面试官阵列生成中`
  - 系统正在生成团队主干 1、团队主干 2、主管、leader、HR 等角色模板。
- `评审团规则装载中`
  - 系统正在准备本轮是 3 人团还是 5 人团，以及过轮规则。
- `流程初始化中`
  - 系统正在创建 plan、stage、round。
- `首轮问题准备中`
  - 系统正在准备第一轮主问题与追问策略。
- `即将进入面试间`
  - 系统提示“本轮主面试官：团队主干 1，重点考察：项目真实性与性能优化”。

### 这个页面该怎么展示

- 顶部显示整体进度条
- 中间显示当前步骤名称
- 下方显示一句用户可读解释
- 右侧可显示本次流程概览：
  - 目标公司
  - 目标岗位
  - 预估轮次
  - 当前主面试官
- 页面不要展示复杂技术细节，只展示用户能理解的准备动作

### 为什么要预置“不同轮次的面试官”

全流程面试如果从头到尾都只是同一个 AI 面试官在问，只是题目变了，用户还是会觉得这是“多轮聊天”，而不是真实流程。

更接近真实招聘流程的方式应该是：

- 一面预置 `团队主干 1`
  - 更偏基础能力、项目真实性、细节深挖。
- 二面预置 `团队主干 2`
  - 更偏系统设计、复杂问题、方案取舍。
- 三面预置 `团队主管`
  - 更偏业务判断、协作、owner 意识。
- 四面预置 `团队 leader`
  - 更偏方向感、复杂项目判断、团队影响力。
- HR 面预置 `HR Agent`
  - 更偏动机、稳定性、沟通风险、职业逻辑。

这样用户感知到的就不只是“题目在变”，而是：

- 面我的人变了
- 每一轮关注点变了
- 每一轮的通过标准也变了

这会让全流程面试比普通 mock 更像真实公司流程。

### 为什么要有评审团投票机制

真实面试里，是否通过往往也不是某一个人一句话决定，尤其在关键轮次，通常会有多个评价视角共同决定。

所以系统里可以设计：

- `3 人评审团`
  - 适合普通轮次，追求速度与成本平衡。
- `5 人评审团`
  - 适合关键轮次，追求更稳定的判断。

规则可以非常清晰：

- 超过半数投“通过”，本轮才通过。
- 未过半则停留本轮，系统生成失败原因和补强动作。
- 如果票数接近，例如 `2:3` 或 `3:2`，系统内部可以记录为“险过”或“高风险通过”，但前端仍然只展示“通过 / 未通过”。

### 例子：一面如何通过投票决定是否进入二面

假设用户正在进行“字节前端一面”：

- 面试官是 `团队主干 1`
- 本轮结束后进入 `5 人评审团` 投票

五位评审分别从不同视角判断：

- 技术深度评审：通过
- 表达结构评审：通过
- 项目真实性评审：不通过
- 权衡能力评审：通过
- 风险校准评审：不通过

最终票数是：

- 通过 `3`
- 不通过 `2`

因为已经过半，所以用户进入二面。

但系统不会只告诉用户“你过了”，还应告诉用户：

- 你过轮了
- 加一项是“结构清晰、性能优化取舍基本成立”
- 优化点是“项目真实性和证据稳定性仍偏弱”
- 二面前建议先做一次项目举证专项训练

这样全流程面试就不仅有“流程推进”，还有“流程治理”。

### 再举一个例子：测试工程师全流程

- 一面：测试基础与用例设计
- 二面：自动化策略与质量平台
- 三面：线上故障分析与协作

虽然岗位不同，但全流程的组织方式不变，变化的是：

- 轮次目标
- 维度权重
- 题型
- 通过标准

### 产品价值

这部分是很重要的差异化，因为多数产品只做“题库式 mock”，而你可以做“流程式准备”。

## 10.3 专项训练：纠偏引擎

### 目标

把单点弱项打透，而不是泛泛问答。

### 现状评分（满分 10 分）

- `7.8/10`
- 原因：已经具备追问、报告与行动回流，是你当前最好的一块，但“同题重答对比”和“证据化纠偏”还可以更强。

### 建议训练模式

- 一句话目标训练
- 同题重答训练
- 改写训练
- 证据补强训练
- 项目深挖训练
- 高压追问训练

### 关键机制

- 每轮训练只围绕 1-2 个明确问题
- 训练后立即给改写建议
- 支持同题重答并比较前后差异

### 例子

用户问题是“结构化表达差”，专项训练不应该只是继续问一堆类似问题，而应该：

- 第一步：让用户回答原题
- 第二步：指出原回答中的结构断点
- 第三步：给出 STAR 或总分总改写版本
- 第四步：让用户同题重答
- 第五步：比较改写前后差异

### 面试间交互原则

- 不提供“思考时间按钮”或“澄清按钮”作为主交互，因为真实面试总时长本就包含思考成本。
- 如果用户在较长时间内未回答，系统应主动给出温和提示，而不是无限等待。
- 面试中的等待态必须可感知，用户需要知道系统在做什么，而不是面对空白页面。
- 技术面试中的回答与题目应支持 Markdown 与代码高亮，避免技术表达被纯文本损耗。
- 每轮回答提交后，系统可以先返回阶段性反馈或生成状态，再补全长文本结论，降低等待焦虑。

### 等待态与无响应设计

- 问题发出后记录用户首字开始输入时间。
- 用户长时间未输入时，系统提示：
  - 你可以先给出思路框架
  - 先回答你最确定的部分
  - 如果完全不会，可以直接说明你会如何排查
- AI 追问生成时应展示明确状态，例如：
  - 正在分析你的回答结构
  - 正在提取关键证据
  - 正在生成下一轮追问

### 技术表达能力设计

- 题目区支持代码块显示、行内代码高亮、表格与列表。
- 回答区支持 Markdown 输入与代码块渲染。
- 后续可扩展到真正的 coding interview，但基础的代码表达能力应先内建。

## 10.4 全流程面试：流程化能力验证系统

### 实测观察

从网络请求可以看到：

- `GET /api/v2/interview-plans?mode=FULL_FLOW`
- `GET /api/v2/interview-experience-tasks`
- `POST /api/v2/interview-plans`
- 跳转面试时携带 `planId / stageId / roundId`

这说明产品已经具备流程引擎基础。

### 应升级为

一个“可视化流程面试系统”，包含：

- 公司流程图
- 阶段任务
- 轮次结果
- 淘汰 / 通过 / 待跟进
- 各轮能力变化

### 现状评分（满分 10 分）

- `6.8/10`
- 原因：基础结构已出现，但用户还没明显感受到“我正在经历一个真实招聘流程”。

### 流程层结构建议

- 计划 Plan
- 阶段 Stage
- 轮次 Round
- 问题链 Question Chain
- 轮次报告 Round Review
- 总结报告 Final Review

### 价值

这部分是未来最容易做出壁垒的模块之一，因为它直接对应真实招聘流程，而不是抽象练习。

### 例子

如果一个用户已经完成“腾讯后台开发工程师一面”，系统不该只显示“已结束”，而应显示：

- 一面结果：通过 / 边缘 / 风险高
- 一面最强项：问题拆解
- 一面最弱项：结构化表达
- 推荐动作：先做 STAR 纠偏训练，再进入二面模拟
- 二面重点：项目深挖 + 系统设计

## 10.5 报告：从总结器升级为证据型教练报告

### 当前已有

- 总评摘要
- 表现亮点
- 风险与不足
- 多 Agent 评审团
- 下一步训练计划
- 维度评分

### 现状评分（满分 10 分）

- `7.4/10`
- 原因：已经有结构和训练感，但证据引用、分数解释、回放能力仍然不够强。

### 下一步必须升级

每个结论都要带：

- 原句证据
- 判定理由
- 分数解释
- 高分改写示例
- 下一步动作

### 例子

不能只写：

- “你的结构化表达不足”

应该写成：

- 证据：你在第 2 题里先讲结论，后面没有补背景和结果
- 判定：说明你有思路，但结构框架不稳定
- 改写示例：如果按 STAR，你应该这样回答
- 下一步动作：进入“结构化表达纠偏训练”

### 报告结构建议

#### A. 总结层

- 这场面试整体水平
- 当前最大问题
- 当前最强项

#### B. 证据层

- 证据片段引用
- 哪句话命中什么维度
- 为什么扣分 / 加分

#### C. 纠偏层

- 怎么改写
- 需要补什么知识
- 下一轮该训练什么

#### D. 进步层

- 和上一场对比
- 哪个维度提升了
- 哪个维度仍然不稳定

#### E. 回放层

- 完整题目链回放
- 每轮追问为什么出现
- 用户原回答与改写版本对照
- 本题关联的知识点、案例与补强动作

### 报告体验设计

- 报告不应只在面试结束后一次性生成，而应尽量在面试过程中完成中间评估。
- 面试结束后应优先返回：
  - 总结
  - 关键问题
  - 下一步动作
- 更细的评审细节、引用证据、长文本建议可以继续补齐。

## 10.6 复盘中心：从历史列表升级为能力雷达控制台

### 当前已有

- 当前关键问题聚合
- 查看诊断
- 开始计划

### 现状评分（满分 10 分）

- `7.0/10`
- 原因：聚合问题与行动入口已经有了，但问题稳定性、证据量、修复轨迹还不够强。

### 必须升级

- 问题按“稳定弱项 / 波动问题 / 新出现问题 / 已修复问题”分类
- 展示每个问题的修复进度
- 展示问题背后的证据样本数
- 支持从问题直接进入：
  - 改写训练
  - 同题重答
  - 学习补课
  - 项目深挖

### 例子

如果一个问题叫“未讨论 React 18 新特性对性能优化的影响”，复盘中心不该只让用户“查看”，而应允许：

- 看原回答证据
- 看为什么被判为问题
- 直接进入 React 18 训练题
- 先看一张 React 18 性能案例卡
- 再同题重答

### 用户价值

让用户知道自己“问题在哪、修到什么程度、下一步做什么”。

## 10.7 学习中心：从内容库升级为处方学习中心

### 当前问题

- 内容不错，但入口仍偏内容浏览。

### 现状评分（满分 10 分）

- `7.1/10`
- 原因：内容本身已有不错基础，但学习中心和报告、训练、成长证明之间的链路还不够强。

### 应升级为

- 报告驱动推荐
- 问题驱动学习
- 高分案例对照
- 反例解析
- 对话式追问学习
- 学完即测、测完即回流训练

### 建议内容形态

- 知识卡片
- 高分示例
- 低分反例
- 追问树
- 项目表达模板
- 行为面模板

### 学习模式设计

- 5 分钟速读：快速补概念
- 深度阅读：补原理与边界
- 对话式面试学习：围绕一个知识点连续追问，直到判断是否真正掌握

### 例子

用户在报告中暴露“对缓存与请求去重解释不具体”，学习中心不应只推荐一篇文章，而应给：

- 知识卡：SWR / React Query 核心机制
- 高分案例：别人如何回答“请求去重”
- 追问学习：系统连续追问 3 轮，确认你是否真的会用
- 再回专项训练验证

### 学习结果沉淀

- 已阅读不等于已掌握
- 学习中心应记录：
  - 是否读完
  - 是否通过追问验证
  - 是否在后续面试中成功用出来
  - 对应薄弱点是否真的下降

## 13. 增长与运营策略

## 13.1 先打穿通用能力内核，再按场景接入适配器

这里不主张“永远只做一个垂类”，而是主张：

- 前台市场可以逐步扩展
- 后台能力必须先有统一内核

### 这件事在产品上到底怎么落地

可以把整套系统理解成两层：

- `底层通用引擎`
  - 负责通用维度判断、证据抽取、追问逻辑、报告生成、行动回流、成长档案。
- `上层岗位适配层`
  - 负责岗位权重、岗位题型、岗位案例、岗位术语、岗位学习内容、岗位通过标准。

产品落地时，不是一个岗位做完再推倒重来接另一个岗位，而是按下面方式建设：

- 第一步：先把通用维度做成稳定标准件
  - 例如结构化表达、证据、拆解、权衡、真实性。
- 第二步：让所有报告都基于证据和维度产出
  - 这样以后换岗位，报告骨架不用重做。
- 第三步：让专项训练动作绑定到通用问题
  - 这样“同题重答”“证据补强”“取舍训练”都能被不同岗位复用。
- 第四步：抽象岗位适配器配置结构
  - 每个岗位都能配置：能力权重、题型、术语、案例、学习包、通过标准。
- 第五步：接入新岗位时，只替换适配层，不重做引擎层
  - 用户看到的是新岗位体验，系统内部复用的是同一条评估和训练链路。

### 一个可执行的产品结构

如果从产品设计角度把它落地，建议把后台维护成下面几个配置块：

- `通用维度库`
  - 例如结构化表达、证据、拆解、权衡、真实性。
- `通用训练动作库`
  - 例如同题重答、STAR 改写、项目举证、方案取舍训练。
- `通用报告组件库`
  - 例如亮点卡、风险卡、证据引用卡、行动建议卡。
- `岗位适配器配置中心`
  - 维护每个岗位的维度权重、题型树、术语库、案例库、通过线。
- `岗位内容生产中心`
  - 维护该岗位的高分样本、低分样本、知识卡、面经模式。

### 例子：如果接“数据开发”

当你未来接入“数据开发”时：

- 不用重做报告页
- 不用重做复盘系统
- 不用重做成长档案
- 不用重做专项训练框架

真正要新增的是：

- 数据开发岗位能力权重
- 数据链路、数仓建模、稳定性、性能相关题型
- 数据开发岗位术语库
- 数据开发高分案例与低分反例
- 数据开发的通过标准与学习内容

这就是为什么“先打穿通用能力内核，再按场景接入适配器”不是一句战略口号，而是一套具体的产品建设方式。

### 正确的扩展方式

先打穿：

- 通用维度
- 证据抽取
- 追问编排
- 报告证据化
- 行动回流
- 成长档案

然后给每个新岗位加：

- 岗位适配器
- 案例库
- 面经库
- 评分权重
- 题型策略

### 例子

如果未来要接“测试工程师 / 产品经理 / 数据开发 / 运营”：

- 不需要重做整套产品
- 只需要新增对应的岗位适配层和内容生产层

### 如何定义“已经准备好开疆扩土”

满足以下条件时可以快速接入新场景：

- 同题重答闭环稳定
- 报告证据化稳定
- 复盘到行动回流稳定
- Agent 评分一致性稳定
- 岗位适配器结构已经抽象完成
- 学习内容、案例、面经可以按标准化流程生产

## 14.1 每个模块都应有独立 Agent 团

产品层面上，不应只把“面试本身”理解成多 Agent，以下模块都应有自己的 Agent 团：

- 阶段面试 Agent 团
- 全流程面试 Agent 团
- 专项训练 Agent 团
- 复盘中心 Agent 团
- 成长档案 Agent 团
- 学习中心建设 Agent 团
- 面经采集整理 Agent 团
- Offer Ready Agent 团

### 各 Agent 团在产品层分别负责什么

#### 阶段面试 Agent 团

- 负责单场面试的出题、追问、评分、证据抽取和即时反馈
- 用户看到的是：
  - 当前问题
  - 当前追问
  - 当前报告

#### 全流程面试 Agent 团

- 负责把目标公司 / 岗位转成一套真实流程计划
- 用户看到的是：
  - 当前轮次
  - 当前是谁在面
  - 本轮目标
  - 本轮评审团投票结果
  - 下一轮风险
  - 整体流程推进

#### 专项训练 Agent 团

- 负责围绕一个弱项进行纠偏、改写、同题重答和提升验证
- 用户看到的是：
  - 我当前在补什么
  - 改写前后差异
  - 我有没有真的提升

#### 复盘中心 Agent 团

- 负责从多次会话中聚合问题、识别稳定弱项、生成行动建议
- 用户看到的是：
  - 我的问题清单
  - 哪些最值得先修
  - 为什么它重要

#### 成长档案 Agent 团

- 负责把多次训练和面试结果串成持续成长轨迹
- 用户看到的是：
  - 哪些能力稳定提升
  - 哪些能力仍在波动
  - 哪些问题已修复

#### 学习中心建设 Agent 团

- 负责生产和维护高质量知识卡、案例、反例、学习路径
- 用户看到的是：
  - 不是随机文章，而是和当前问题强相关的学习材料

#### 面经采集整理 Agent 团

- 负责采集、清洗、去重、结构化、审核公开面经
- 用户看到的是：
  - 更真实的公司流程
  - 更贴近目标岗位的题目网络

#### Offer Ready Agent 团

- 负责判断用户距离目标岗位的 readiness、轮次风险和最短补强路径
- 用户看到的是：
  - 我现在能不能投
  - 会卡在哪一轮
  - 最应该补什么

### 例子

同样是“证据”：

- 面试 Agent 团关心的是：这句话能不能支持本轮评分
- 复盘 Agent 团关心的是：这类问题是否在历史上重复出现
- 成长档案 Agent 团关心的是：这个问题是稳定弱项还是偶发波动
- 知识中心 Agent 团关心的是：应该补哪种知识内容和案例

同一个输入，不同 Agent 团的目标不同，因此应该隔离设计。

## 15. 产品成功的判断标准

如果一年后用户提起面面吧，第一反应不是：

- “这个 AI 很会聊天”

而是：

- “这个产品真的能看出我哪儿不行”
- “它能逼我练到过关”
- “它能证明我真的变强了”
- “别人是给我一份报告，它是给我一套训练系统”

那它就开始真正建立壁垒了。
