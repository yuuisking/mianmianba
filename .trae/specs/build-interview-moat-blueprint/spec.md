# 构建面面吧核心壁垒与平台总蓝图 Spec

## Why
当前面面吧已经不再是一个从零开始的 AI 面试 Demo，而是已经具备了 `全流程计划 / 多 Agent 提问链 / 多 Agent 评审链 / 公开面经采集 / 复盘中心 / 行动回流` 等关键基础能力。问题不在于“有没有功能”，而在于这些能力还没有被统一成一份平台级蓝图，因此后续实现很容易继续沿着页面需求、局部修复或单点能力增强发散，最终做成“功能很多、系统不强、壁垒不稳”的产品。

本 Spec 的目标不是替代现有执行 spec，而是定义一份高于单模块开发的总蓝图：明确面面吧未来必须围绕什么主线演进、哪些能力构成不可复制的核心竞争力、哪些模块是主链、哪些模块是配套、当前系统距离目标态还差什么、以及接下来所有实现应该如何被约束，避免平台被继续做偏。

## What Changes
- 将面面吧从“AI 模拟面试产品”重新定义为“围绕目标岗位与目标公司持续完成诊断、追问、评估、复盘、训练、学习、成长证明和 Offer Ready 判断的面试能力操作系统”。
- 明确平台的核心壁垒不建立在界面或 Prompt 上，而建立在四层可积累资产上：
  - 评估标准资产
  - 证据与行为数据资产
  - 训练处方资产
  - 效果证明资产
- 将平台主线统一为：
  - `目标岗位建模 -> 基线诊断 -> 多轮训练/多轮流程 -> 证据化评估 -> 行动处方 -> 学习补强 -> 再验证 -> 成长证明 -> Offer Ready`
- 将“通用能力内核 + 岗位适配器”定义为平台扩展的核心抽象，要求未来扩岗必须复用统一能力内核，而不是按岗位复制页面和 Prompt。
- 将全流程面试确认为平台当前阶段最重要的核心壁垒，要求其升级为真实的流程化能力验证系统，而不是多个轮次拼接的聊天体验。
- 明确阶段面试、专项训练、全流程面试、复盘中心、学习中心、成长档案、Offer Ready、面经情报网络都应有各自的模块级 Agent 团，彼此职责隔离、输入输出结构化。
- 确立“证据化报告”和“可证明成长”是平台可信度与留存的核心前提，后续所有评分、总结、建议、风险判断都必须能回到结构化证据。
- 明确当前技术落地口径为：
  - 当前态：Next.js + Prisma 单体内模块化分层
  - 目标态：能力边界清晰、模块职责隔离、长任务与实时链路分层
  - 当前阶段不以微服务化为目标，但所有新能力必须按可隔离、可拆分的边界设计
- 建立平台分阶段路线图：
  - Phase 1：打穿全流程壁垒与流程治理主链
  - Phase 2：强化证据化报告、专项纠偏、复盘回流、学习处方与成长档案
  - Phase 3：完善 Offer Ready、跨岗位适配器、模块独立运行边界与平台化评测校准能力
- **BREAKING**：后续新增能力不再允许以“单模型大 Prompt 包打所有职责”的方式进入主链。
- **BREAKING**：后续平台规划不再允许把首页、复盘、学习、全流程、专项训练当作彼此独立的功能岛进行设计，必须回到统一主线。

## Current-State Baseline
- 当前已具备的真实基础能力：
  - `InterviewPlan / InterviewPlanStage / InterviewRound` 全流程主结构已经在线
  - 主动结束、缺席淘汰、反馈深链、站内信、轮次终态已接入
  - 提问链已拆为 `Planner / Evidence / Composer / Guard`
  - 评审链已具备多 Agent 面后评审和逐轮评审结构
  - 公开网页面经采集底座已上线，具备抓取、抽取、审核、去重、写库能力
  - 复盘中心已接入真实数据，并支持问题、证据、动作和诊断弹窗
- 当前最主要的缺口：
  - 平台还缺统一的“通用能力内核”定义与可执行配置
  - 岗位适配器尚未抽象为稳定结构
  - 全流程的“真实流程感”和“面试官链路感”仍强于以前，但尚未完全成为核心体验壁垒
  - 报告与复盘仍然有结构，但证据化程度、逐句可解释性、成长证明仍不足
  - 学习中心、成长档案、Offer Ready 与现有主链还未真正连成一个平台系统
  - 技术架构上虽已有多个 service-like 模块，但长期职责边界与结构化数据契约仍未正式冻结

## Product Positioning
- 面面吧不是“更会聊天的 AI 面试官”
- 面面吧也不是“带报告的题库系统”
- 面面吧的最终定位应被定义为：
  - 面向目标岗位和目标公司
  - 围绕真实轮次和真实能力要求
  - 持续完成建模、追问、评估、复盘、训练、学习、成长证明和 readiness 判断
  - 的面试能力操作系统

## Product Principles
- 真实优先：所有关键输出优先基于真实输入、真实岗位、真实简历、真实会话、真实面经、真实证据生成。
- 闭环优先：每次面试、训练、学习都必须能回流到问题、动作、成长或 readiness 判断。
- 证据优先：没有证据支撑的强结论不得进入高置信用户视图。
- 系统优先：优先建立平台可复用内核，不靠堆页面和堆模块提升感知复杂度。
- 岗位化优先：所有问题、评分、建议、学习内容都必须服务目标岗位，而不是泛化的“面试感觉”。
- 流程优先：全流程面试必须体现计划、轮次、通过条件、面试官链路和状态推进，不允许退化成单轮聊天。
- 可隔离优先：当前虽以单体实现，但模块边界、数据契约、异步任务边界必须先设计清楚，确保能力不会互相污染。

## Strategic Layers
- 第一层壁垒：评估标准资产
  - 通用能力维度字典
  - 各维度评分标准
  - 高分/低分/伪高分标准
  - 追问触发条件
  - 训练动作映射关系
- 第二层壁垒：行为与证据数据资产
  - 回答原文
  - 证据切片
  - 维度评分
  - 评审分歧
  - 同题重答前后变化
  - 学习后是否真正改善
- 第三层壁垒：训练处方资产
  - 针对某种问题先练什么
  - 练几轮
  - 什么条件下算练会
  - 先补知识还是先重答
  - 下一轮该验证什么
- 第四层壁垒：效果证明资产
  - 哪个维度提升了
  - 为什么提升了
  - 哪些问题稳定修复
  - 当前距离目标岗位还有多远

## Architecture Principles
- 不允许一个大模型同时承担：
  - 岗位建模
  - 出题
  - 追问
  - 判分
  - 汇总
  - 计划生成
- 每个 Agent 只负责单一、可验证、可结构化输出的职责。
- 模块级 Agent 团之间通过结构化结果通信，而不是共享推理状态。
- 长任务与高耗时流程必须支持异步任务化，不得依附单次 Web 请求生命周期。
- 结构化结果优先入库，面向用户的自然语言摘要、报告、解释可作为后续渲染层。
- 评估与报告必须支持增量化，不能完全依赖“结束后一次性大生成”。

## ADDED Requirements
### Requirement: 平台统一主线
系统 SHALL 将产品所有关键模块统一收敛到一条明确主线，而不是各页面各自表达价值。

#### Scenario: 用户进入平台
- **WHEN** 用户首次或再次进入平台
- **THEN** 系统必须能围绕目标岗位、目标公司和当前能力状态，给出清晰的当前阶段、下一步动作和最终目标路径
- **AND** 首页、发起面试、专项训练、复盘中心、学习中心、成长档案与 Offer Ready 都必须服务同一主线

### Requirement: 通用能力内核
系统 SHALL 建立可复用的通用能力内核，作为跨岗位、跨模块、跨 Agent 的共同评估与训练底座。

#### Scenario: 定义通用维度
- **WHEN** 平台定义能力内核
- **THEN** 至少应包含结构化表达、证据充分性、问题拆解、方案权衡、结果导向、真实性、沟通清晰度等通用维度
- **AND** 每个维度都必须有定义、评分标准、证据规则、追问模板、报告模板和训练动作映射

#### Scenario: 复用通用内核
- **WHEN** 新岗位或新模块接入平台
- **THEN** 其评估、追问、报告与训练必须优先复用通用能力内核
- **AND** 不允许为每个岗位各自重新发明一套孤立标准

### Requirement: 岗位适配器
系统 SHALL 在通用能力内核之上构建岗位适配器，用于承载岗位差异、岗位题型、术语、案例和通过标准。

#### Scenario: 技术岗首批接入
- **WHEN** 首批岗位适配器落地
- **THEN** 平台应优先覆盖技术岗，至少能明确前端与后端技术岗的能力权重、题型树、术语、案例来源与通过标准

#### Scenario: 新岗位扩展
- **WHEN** 后续接入测试、产品、运营、数据开发等岗位
- **THEN** 系统应通过新增岗位适配器完成扩展
- **AND** 不需要重做报告、复盘、成长、训练等底层系统

### Requirement: 首页训练控制台
系统 SHALL 将首页升级为训练控制台，而不是纯导航页。

#### Scenario: 老用户返回首页
- **WHEN** 用户已有训练、面试、复盘或学习记录
- **THEN** 首页展示当前最值得先做的问题、下一步最优动作、本周变化、流程推进状态和 readiness 变化趋势

#### Scenario: 新用户进入首页
- **WHEN** 用户尚未形成完整训练档案
- **THEN** 首页仍应围绕“如何快速进入岗位建模与基线诊断”组织信息
- **AND** 不以静态 mock 数据伪装平台能力

### Requirement: 发起面试作为岗位建模入口
系统 SHALL 将发起面试从配置页升级为岗位建模入口，并明确区分阶段面试与全流程面试。

#### Scenario: 阶段面试
- **WHEN** 用户选择阶段面试
- **THEN** 系统围绕某一轮次或某一类面试目标进行建模和配置
- **AND** 输出本场目标、面试风格、关键考察维度和用户风险提示

#### Scenario: 全流程面试
- **WHEN** 用户选择全流程面试
- **THEN** 系统基于目标公司、目标岗位、JD、简历和面经情报生成完整面试计划
- **AND** 让用户清晰知道总流程、当前轮次、预估轮次、轮次目标、通过条件与风险点

#### Scenario: 全流程配置页收敛
- **WHEN** 用户进入全流程面试配置页
- **THEN** 页面不再让用户手动配置流程结构
- **AND** 流程结构应由系统根据公司、岗位、部门、JD、简历和面经情报自动推断生成
- **AND** 页面保留岗位信息配置，但必须新增“部门信息”字段，作为面经采集、流程解析和面试官画像生成的输入之一

#### Scenario: 确认配置后的初始化进度态
- **WHEN** 用户点击确认面试配置
- **THEN** 页面进入明确的初始化进度态，而不是直接跳转面试间或白屏等待
- **AND** 面向用户的步骤文案至少包含：
  - `初始化面试题目`
  - `初始化面试流程`
- **AND** 即使后台真实执行了面经采集、流程推断、计划构建等动作，也不得直接向用户暴露“正在采集面经”等实现细节
- **AND** 初始化完成后再进入正式面试间

### Requirement: 全流程面试壁垒
系统 SHALL 将全流程面试升级为真实的流程化能力验证系统，而不是多个轮次拼接的聊天体验。

#### Scenario: 创建全流程计划
- **WHEN** 用户创建目标公司和岗位的全流程计划
- **THEN** 系统生成 `InterviewPlan / InterviewPlanStage / InterviewRound`
- **AND** 每个 stage 都包含阶段目标、主面试官角色、关键维度、题型、投票规则和通过条件

#### Scenario: 流程推进
- **WHEN** 用户完成当前轮次
- **THEN** 系统根据评审团投票和阶段治理结果决定通过、淘汰、险过或补强后再试
- **AND** 下一轮必须继承上一轮暴露的问题与风险

#### Scenario: 初始化等待态
- **WHEN** 用户点击发起全流程面试
- **THEN** 平台展示明确的初始化步骤与进度反馈
- **AND** 用户可见文案应优先使用“初始化面试题目”“初始化面试流程”等产品表达
- **AND** 后台可以执行岗位确认、面经情报获取、流程解析、面试官阵列生成、评审规则装载、流程初始化、首轮问题准备等动作
- **AND** 让用户知道系统正在完成真实准备动作，而不是单纯加载

#### Scenario: 流程信息展示位置
- **WHEN** 用户查看全流程列表中的当前面试卡片
- **THEN** 卡片中应展示本次流程包含哪些环节、当前位于哪一个环节、后续还剩哪些环节
- **AND** 用户不需要在配置阶段手动维护流程信息
- **AND** 流程信息应以下钻清晰、易理解的方式展示，而不是只给出抽象状态文本

### Requirement: 模块级多 Agent 团
系统 SHALL 以模块级 Agent 团而不是单一大模型作为平台核心执行方式。

#### Scenario: 阶段面试 Agent 团
- **WHEN** 用户进行阶段面试
- **THEN** 至少存在岗位解析、提问编排、证据抽取、维度评审、漏洞交叉质询、教练建议和可信度判断等独立职责

#### Scenario: 全流程面试 Agent 团
- **WHEN** 用户进行全流程面试
- **THEN** 至少存在公司流程解析、面经检索、计划生成、轮次策略、面试官画像、主面试官、评审团投票聚合、闸门裁决、风险预测等独立职责

#### Scenario: 复盘中心 Agent 团
- **WHEN** 系统聚合多场训练和面试结果
- **THEN** 至少存在问题聚合、稳定性分析、影响评分、证据分组、动作推荐等独立职责

### Requirement: 提问链与评审链严格解耦
系统 SHALL 将“当前轮次如何问”与“当前轮次如何判”明确拆开，不允许同一链路既主导发问又主导裁决。

#### Scenario: 面试进行中
- **WHEN** 用户正在回答问题
- **THEN** 提问链只负责节奏推进、追问生成与用户可见问题
- **AND** 不直接输出最终裁决

#### Scenario: 轮次结束
- **WHEN** 当前轮次结束
- **THEN** 评审链独立消费问答证据并输出评分、票型、置信度、裁决摘要与下一步建议

### Requirement: 证据化报告
系统 SHALL 将报告升级为证据型教练报告，所有关键结论都必须能回到原句证据与判定理由。

#### Scenario: 生成轮次或阶段报告
- **WHEN** 用户完成某次阶段面试、专项训练或全流程轮次
- **THEN** 报告应包含原句证据、判定理由、维度分数解释、改写示例、推荐动作

#### Scenario: 证据不足
- **WHEN** 当前回答证据不足、Judge 分歧过大或样本不足
- **THEN** 系统必须降置信或明确返回“不足以判断”
- **AND** 不允许伪造高置信总结

### Requirement: 专项训练作为纠偏引擎
系统 SHALL 将专项训练定义为围绕单一问题连续纠偏、改写、同题重答和再验证的引擎，而不是泛泛多轮问答。

#### Scenario: 稳定弱项进入专项训练
- **WHEN** 复盘中心确认某个问题为高影响稳定弱项
- **THEN** 系统可以直接将该问题映射到专项训练
- **AND** 训练过程至少支持原回答诊断、改写脚手架、同题重答、前后差异比较

### Requirement: 复盘中心作为能力诊断中枢
系统 SHALL 将复盘中心升级为问题总览、证据回放、动作安排、修复进度和历史轨迹的能力中枢。

#### Scenario: 用户进入复盘中心
- **WHEN** 用户进入复盘中心
- **THEN** 页面展示当前关键问题、问题分类、问题趋势、动作效果、样本覆盖、历史训练记录和诊断入口

#### Scenario: 查看单个问题
- **WHEN** 用户打开问题诊断
- **THEN** 平台展示问题详情、根因、证据样本、动作链路、验证结果和关联轮次/岗位/公司信息

### Requirement: 学习中心作为处方学习系统
系统 SHALL 将学习中心从内容库升级为处方学习系统，由问题、训练和岗位目标驱动内容分发。

#### Scenario: 问题触发学习
- **WHEN** 平台识别到用户在某个知识点或能力点上反复失分
- **THEN** 学习中心推荐知识卡、高分案例、反例解析、追问树和学习路径

#### Scenario: 学完再测
- **WHEN** 用户完成某项学习内容
- **THEN** 系统应支持重新进入训练或同题验证
- **AND** 学习结果不能仅以“已阅读”作为掌握证明

### Requirement: 成长档案
系统 SHALL 形成持续的成长档案，用于区分稳定提升、波动问题、修复完成和回退风险。

#### Scenario: 累积多次记录
- **WHEN** 用户完成多场阶段面试、专项训练或全流程轮次
- **THEN** 平台能够形成维度趋势、问题状态变化、修复轨迹和成长叙事

#### Scenario: 判断是否真的提升
- **WHEN** 某个维度短期出现一次高分
- **THEN** 系统不得直接判定为“已修复”
- **AND** 需要通过连续样本和后续验证确认提升是否稳定

### Requirement: Offer Ready
系统 SHALL 提供面向目标岗位与目标公司的 readiness 判断，而不是停留在单场表现总结。

#### Scenario: 用户准备投递
- **WHEN** 用户积累了足够多的训练、流程和复盘记录
- **THEN** 平台输出 readiness 分数、各轮次风险、最大能力缺口和最短补强路径

#### Scenario: 轮次风险判断
- **WHEN** 系统输出 readiness 结果
- **THEN** 应明确说明用户最可能卡在哪一轮、为什么、以及在进入该轮次前最值得补的动作

### Requirement: 面经情报网络
系统 SHALL 将公开面经采集能力升级为公司流程模式、题型分布、岗位追问网络和风险预测的情报底座。

#### Scenario: 形成公司模式
- **WHEN** 系统积累足够多的公司/岗位/轮次面经样本
- **THEN** 能沉淀结构化的公司流程模式、轮次主题模式、岗位高频考点和风险热点

#### Scenario: 被全流程消费
- **WHEN** 用户创建某公司某岗位的全流程计划
- **THEN** 全流程计划优先消费该公司和岗位的面经情报模式
- **AND** 不允许完全脱离情报输入生成泛化流程

### Requirement: 结构化数据底座
系统 SHALL 保留结构化的计划、轮次、问题、回答、证据、评审、动作、趋势、readiness 和面经情报数据。

#### Scenario: 核心实体入库
- **WHEN** 平台执行计划生成、问答、评审、复盘、训练、学习、readiness 判断
- **THEN** 关键结果必须以结构化字段落库，而不是仅保留自然语言总结

#### Scenario: 后续平台演进
- **WHEN** 后续需要做增长分析、judge 校准、成长验证或服务化拆分
- **THEN** 当前结构化数据足以支撑分析与迁移

### Requirement: 单体内模块化隔离边界
系统 SHALL 在当前 Next.js 单体架构内明确定义模块化隔离边界，并确保不同能力在不做微服务的前提下也能稳定解耦。

#### Scenario: 当前阶段实现
- **WHEN** Phase 1 和 Phase 2 能力落地
- **THEN** 平台可以继续以单体仓库和单部署单元实现
- **AND** 各模块必须有明确的职责边界、输入输出契约、数据边界和异步任务边界

#### Scenario: 未来拆分
- **WHEN** 某模块出现高耗时、高并发、强独立扩容、强失败重试需求
- **THEN** 该模块应可逐步拆为独立 worker、独立任务处理层或独立部署单元
- **AND** 这种拆分是能力边界清晰后的结果，而不是本蓝图的先验目标

### Requirement: 评估校准与可信度守门
系统 SHALL 建立评估校准、分歧处理和可信度守门机制，避免评分漂移和伪高置信输出。

#### Scenario: Judge 分歧过大
- **WHEN** 多个 Judge 对同一问题出现较大分歧
- **THEN** 系统触发重审、降置信或交给校准机制处理

#### Scenario: 缺少证据
- **WHEN** 证据片段过少、样本不足或答案有效信息密度过低
- **THEN** Trust 机制必须阻止系统给出过强结论

### Requirement: 可观测与可回放
系统 SHALL 让核心 Agent 调用、轮次推进、证据抽取、裁决理由和报告生成过程可观测、可追踪、可回放。

#### Scenario: 线上问题排查
- **WHEN** 平台出现题干异常、裁决异常、流程状态异常或报告异常
- **THEN** 系统能够回放 request、agent 调用、输入输出 hash、延时、版本、结构化结论和状态推进链

## MODIFIED Requirements
### Requirement: 现有全流程功能
现有全流程功能 MUST 从“多轮模拟入口”升级为“平台核心壁垒主链”。

#### Scenario: 用户理解全流程
- **WHEN** 用户进入全流程相关页面
- **THEN** 应清晰感知这是目标公司与目标岗位下的真实流程准备系统
- **AND** 不再只看到“开始面试”或“继续当前轮次”这类单轮心智表达

#### Scenario: 列表卡片承接流程感知
- **WHEN** 用户返回当前面试列表
- **THEN** 当前面试卡片应明确展示流程全貌、当前环节和接下来的环节
- **AND** 让用户在列表层就能理解自己正处在哪一轮流程，而不是必须先进入配置页或面试间

### Requirement: 现有复盘中心
现有复盘中心 MUST 从“历史问题与推荐动作页”升级为“问题诊断与行动中枢”。

#### Scenario: 用户查看历史记录
- **WHEN** 用户查看历史训练与诊断
- **THEN** 平台不仅展示分数和问题，还要展示问题趋势、动作验证、样本覆盖和问题状态变化

### Requirement: 现有面经采集能力
现有公开网页面经采集能力 MUST 从“出题素材采集器”升级为“面经情报网络底座”。

#### Scenario: 平台消费采集结果
- **WHEN** 全流程计划、问题策略、知识中心和 readiness 系统读取面经数据
- **THEN** 它们优先消费结构化情报模式，而不是只消费零散 summary 文本

### Requirement: 现有报告能力
现有报告能力 MUST 从“结构化总结”升级为“证据型教练报告 + 增量评估入口”。

#### Scenario: 结束时出报告
- **WHEN** 用户结束面试或训练
- **THEN** 平台优先返回可读的骨架总结和下一步动作
- **AND** 后续继续补齐细粒度证据和长文本评审

## REMOVED Requirements
### Requirement: 单一大模型包打所有职责
**Reason**: 该方式会导致出题、追问、评分、总结、计划相互污染，难以校准、难以解释，也无法形成平台级壁垒。
**Migration**: 后续所有主链能力必须迁移到模块级 Agent 团与结构化结果驱动的架构。

### Requirement: 模块孤岛式规划
**Reason**: 将首页、发起面试、专项训练、复盘中心、学习中心、成长档案分别规划，会让产品继续偏向“功能集合”而不是“能力系统”。
**Migration**: 所有模块设计必须回到统一主线，并明确其输入、输出与回流关系。

### Requirement: 长期依赖 mock 或弱证据展示关键结论
**Reason**: mock 和弱证据会直接破坏平台可信度，削弱壁垒积累。
**Migration**: 缺数据时显示空状态或低置信提示，关键视图逐步全部切到真实结构化数据。

## Phase Roadmap
- Phase 1：全流程壁垒与流程治理
  - 统一 plan/stage/round 主骨架
  - 公司流程模式与面试官链路
  - 评审团投票与阶段闸门
  - 逐轮反馈沉淀与复盘回流
  - 面经情报对全流程策略输入
- Phase 2：证据化平台与训练闭环
  - 通用能力内核
  - 岗位适配器首版
  - 证据化报告
  - 专项纠偏引擎
  - 复盘中心强化
  - 学习中心处方化
  - 成长档案
- Phase 3：平台化壁垒与扩展
  - Offer Ready
  - 多岗位快速接入
  - Judge 校准体系
  - 目标态 service 拆分
  - 平台级评测、观测与策略实验

## Out of Scope For Immediate Execution
- 一次性完成所有岗位的广泛接入
- 一次性完成全部模块的独立部署化改造
- 一次性完成视频、语音、白板、情绪识别等所有高级多模态能力
- 一次性上线完整企业合作与服务化体系

## Success Criteria
- 用户不再把面面吧理解为“更会聊天的 AI 面试官”
- 用户能明确感知全流程面试是一个真实流程准备系统
- 报告与复盘能解释“问题在哪、为什么、怎么改、是否真的变好”
- 平台能逐步形成跨场景复用的能力内核和岗位适配器
- 后续任何模块开发都能被这份蓝图约束，而不是继续朝局部最优发散
