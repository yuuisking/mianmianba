# 面面吧 v2.0 平台迭代 Spec

## Why
当前面面吧 v1.0 已经具备“发起面试 / 专项训练 / 学习中心 / 复盘中心”的基础雏形，但距离“真正帮助用户学会知识、提升面试表现、提升拿 Offer 概率”的目标还有明显差距。核心问题不再是某一个页面不好看，而是整个平台缺少一套完整的求职训练操作系统：

- 首页还没有真正承接“进度、继续训练、薄弱点、复盘闭环”。
- 发起面试仍偏单场练习，不能模拟真实求职中的一面、二面、三面、HR 面与 Offer 流程。
- AI 面试官仍偏单 Prompt 驱动，没有拆成真正有分工的 Agent 面试评审团。
- 代码面试环节缺失，无法覆盖真实技术面中高频出现的“手撕代码”场景。
- 复盘中心还没有成为“用户能力诊断中枢”，无法系统管理薄弱点、训练记录、面试记录与改进任务。
- 数据模型仍以页面驱动为主，没有形成“用户画像 - 面试计划 - 面试执行 - 评分报告 - 薄弱点管理 - 后续训练”的统一数据闭环。

因此 v2.0 不是一次普通重构，而是一次平台级迭代：在冻结学习中心和会员管理主线的前提下，优先重做首页、发起面试、专项训练、复盘中心与代码面试环境，先把“更像一个真正懂求职者的平台”这件事做扎实。

## What Changes
- 冻结学习中心主线，不再作为 v2.0 当前阶段的主开发目标；学习中心仅保留已有能力与必要兼容。
- 基于 `UI/重构v2.0/daan-ui/daan_ai_ui_gallery_responsive.html` 对首页进行一比一复刻，视觉、模块顺序、信息组织与交互节奏必须高度对齐。
- 首页新增并打通三类核心能力：`学习/训练进度`、`继续训练`、`当前薄弱点`，所有数据必须来自真实记录，不允许长期使用前端静态 mock。
- 重构“发起面试”为两条主链路：
  - `阶段面试`：延续当前单场模式，但配置能力、候选人画像、题量/时长、强度和面试官策略要升级。
  - `全流程面试`：围绕目标公司 + 岗位 + JD + 简历生成整套面试计划，支持一面、二面、三面、HR 面、Offer 结论等阶段。
- 将当前单 AI prompt 方案升级为 `Agent 面试评审团`：至少拆分为面试计划、题目生成、实时追问、评分裁决、总结归纳、报告生成、学习建议等角色。
- 所有面试模式都要支持“代码面试环节”，并提供接近 LeetCode 的代码间体验，允许在面试流中自动跳转到代码作答页面。
- 把面试流程和出题流程抽象为状态机，确保阶段推进、打断追问、代码题切换、面试结束、报告生成、复训回流都有明确状态。
- 彻底重构复盘中心，使其变成“能力诊断与行动中枢”，支持能力维度、知识点薄弱、历史记录、成长趋势、改进任务包与推荐训练计划。
- 新增 v2.0 的统一数据底座与库表设计，系统性管理用户画像、面试计划、面试轮次、题目、回答、代码提交、评分、报告、薄弱点、学习/训练进度等核心数据。
- 支持未来多模态能力，但按迭代节奏拆层推进：先把平台骨架、数据结构、Agent 机制和页面主链打稳，再逐步增强视频、语音、情绪识别、白板等高级能力。

## Demo Research Summary
本次 spec 已详细阅读并核对以下 demo：

- `UI/重构v2.0/daan-ui/daan_ai_ui_gallery.html`
- `UI/重构v2.0/daan-ui/daan_ai_ui_gallery_responsive.html`
- `UI/重构v2.0/daan-ui/interview_code_editor.html`

已确认的关键设计结论：

- 首页不是纯导航页，而是“训练操作台”，重点承接问候语、四大核心入口、我的进度、继续训练、Top3 薄弱点。
- 发起面试不是一个简单表单，而是“先选模式，再进入不同配置链路”的入口页；`阶段面试` 和 `全流程面试` 在用户心智上必须分开。
- 候选人画像页在 v2.0 中非常关键，它是正式进入面试前的“风险识别 + 策略确认”页。
- 面试房间右侧辅助区不是装饰，而是“考察点 / 回答结构 / 失分风险”的实时支架。
- 代码间 demo 明确表明：v2.0 需要独立的代码面试空间，而不是在文本面试框里嵌个 textarea。
- 复盘中心 demo 的方向是对的：它不是简单“看分数”，而是“看问题、看趋势、看行动方案”。

  - 数据层：Prisma schema、用户进度与薄弱点聚合服务、面试计划与状态机服务

## Product Principles
- 真实优先：所有关键结果优先使用真实用户输入、真实 JD、真实简历、真实会话、真实回答，禁止伪造用户画像和训练结论。
- 闭环优先：每一场面试和训练都必须能回流到进度、薄弱点、复盘与后续行动。
- 求职导向优先：不是“问对没问对”，而是“是否更接近真实求职成功”。
- 过程数据优先：不仅保留结论，还要保留回答过程、追问过程、代码提交过程与阶段推进过程。
- 可迭代架构优先：v2.0 第一阶段先把结构、数据和体验骨架做对，再逐步增强多模态和高级 Agent 能力。

## ADDED Requirements
### Requirement: 首页 1:1 承接 v2.0 Demo
系统 SHALL 以 v2.0 demo 首页为目标蓝本，对首页进行高度一致的复刻，并保持其作为训练操作台的定位。

#### Scenario: 首页默认展示
- **WHEN** 用户进入首页
- **THEN** 页面展示问候区、四大核心入口、我的进度、继续训练、当前薄弱点 Top 3
- **AND** 布局结构、模块顺序、信息密度和视觉节奏与 demo 保持一致

#### Scenario: 首页从真实数据中读取
- **WHEN** 用户已有训练、面试或复盘记录
- **THEN** 首页中的进度、继续训练和薄弱点使用真实数据填充
- **AND** 缺少数据时展示明确空状态，而不是长期写死静态数字

### Requirement: 学习与训练进度管理中枢
系统 SHALL 建立统一的用户成长进度模型，打通首页、专项训练、面试记录与复盘中心。

#### Scenario: 训练完成后更新进度
- **WHEN** 用户完成一场面试或专项训练
- **THEN** 系统更新用户的训练次数、连续训练天数、复训率、最近训练项与薄弱点统计

#### Scenario: 首页显示继续训练
- **WHEN** 用户存在未完成、可续练或推荐复训的训练项
- **THEN** 首页展示“继续训练”卡片
- **AND** 点击后进入对应训练上下文，而不是重新从零开始

### Requirement: 薄弱点体系化管理
系统 SHALL 以知识点、能力维度、表达维度和招聘阶段维度为基础，系统性管理用户薄弱点。

#### Scenario: 薄弱点沉淀
- **WHEN** 用户在面试、专项训练或代码题中表现出低分维度或高频失误
- **THEN** 系统将其沉淀为结构化薄弱点记录
- **AND** 记录来源、出现频次、最近得分、严重程度和建议行动

#### Scenario: 薄弱点驱动推荐
- **WHEN** 首页或复盘中心展示薄弱点
- **THEN** 系统同时提供对应的专项训练、模拟面试、复训任务或学习建议入口

### Requirement: 发起面试双模式
系统 SHALL 将发起面试拆分为 `阶段面试` 和 `全流程面试` 两种模式，并在入口阶段明确区分。

#### Scenario: 阶段面试
- **WHEN** 用户选择阶段面试
- **THEN** 系统进入单轮面试配置流程
- **AND** 用户可以配置岗位、层级、语言、强度、模式、题量/时长、JD 与简历

#### Scenario: 全流程面试
- **WHEN** 用户选择全流程面试
- **THEN** 系统进入完整面试计划创建流程
- **AND** 用户可以配置目标公司、目标岗位、目标职级、JD、简历以及目标招聘流程

### Requirement: 全流程面试计划生成
系统 SHALL 支持围绕目标公司和岗位生成真实招聘链路下的多轮面试计划。

#### Scenario: 生成完整计划
- **WHEN** 用户提交公司、岗位、JD 和简历
- **THEN** 系统生成包含一面、二面、三面、HR 面、Offer 结论的候选面试计划
- **AND** 每一轮都包含轮次目标、重点考察模块、预估时长、可能的代码环节和评估重点

#### Scenario: 公司风格定制
- **WHEN** 系统具备该公司或岗位的真实面经、题型偏好或公开资料
- **THEN** 全流程面试优先参考这些资料构建问题策略
- **AND** 不允许完全脱离岗位上下文泛泛提问

### Requirement: 候选人画像与风险识别页
系统 SHALL 在进入正式面试前生成候选人画像页，用于确认岗位匹配度、风险点与本场策略。

#### Scenario: 画像确认
- **WHEN** 用户完成面试配置并进入确认页
- **THEN** 页面展示岗位匹配度、简历风险点、本场策略、简历可优化点和手动补充训练重点
- **AND** 用户确认后才正式进入面试执行阶段

### Requirement: Agent 面试评审团
系统 SHALL 将当前单一 AI 面试官升级为多 Agent 协作的评审团机制。

#### Scenario: 面试前规划
- **WHEN** 用户发起面试
- **THEN** `Planner Agent` 负责生成面试计划、轮次策略、考察维度和难度节奏

#### Scenario: 面试中提问
- **WHEN** 用户处于进行中的面试状态
- **THEN** `Interviewer Agent` 负责主问题与追问
- **AND** `Evidence Agent` 负责引用 JD、简历、历史表现、岗位要求和相关面经作为提问依据

#### Scenario: 面试后裁决
- **WHEN** 用户结束本轮面试
- **THEN** `Scoring Agent` 负责多维度评分
- **AND** `Summary Agent` 负责总结亮点与短板
- **AND** `Report Agent` 负责生成结构化复盘报告与行动建议

### Requirement: 面试中的打断与深追问
系统 SHALL 支持模拟真实面试中的打断、追问和边界挖掘，而不是只按固定题单顺序问答。

#### Scenario: 面试中打断
- **WHEN** 用户回答中存在明显漏洞、过于抽象或避重就轻
- **THEN** 系统允许面试官状态机进入打断追问分支
- **AND** 打断内容带有明确上下文依据

### Requirement: 代码面试环节
系统 SHALL 为所有合适的面试流程提供独立代码面试环节，并使用接近 LeetCode 的代码间体验。

#### Scenario: 自动进入代码环节
- **WHEN** 当前轮次计划包含算法题或编码题
- **THEN** 系统自动将用户切换到代码面试间
- **AND** 保留当前面试上下文、题号、倒计时和面试官提示

#### Scenario: 代码过程记录
- **WHEN** 用户在代码间中输入、运行、提交或请求提示
- **THEN** 系统记录代码草稿、运行结果、提交结果、提示使用情况和关键时间节点

### Requirement: 面试状态机
系统 SHALL 使用状态机管理面试计划、面试轮次、问题推进、打断追问、代码切换、报告生成和后续回流。

#### Scenario: 状态清晰推进
- **WHEN** 用户从发起面试进入正式执行
- **THEN** 系统按状态机推进 `draft -> profiled -> planned -> in_progress -> coding(optional) -> round_completed -> report_generated -> archived`

#### Scenario: 全流程轮次推进
- **WHEN** 用户完成当前轮次
- **THEN** 系统决定进入下一轮、进入 HR 面、结束流程或给出 Offer 结论
- **AND** 每一步都记录状态变更原因与时间

### Requirement: 复盘中心作为诊断中枢
系统 SHALL 将复盘中心重构为“能力诊断、问题归因、改进任务、趋势追踪”的核心中枢。

#### Scenario: 复盘首页
- **WHEN** 用户进入复盘中心
- **THEN** 页面展示训练频次、平均分、薄弱维度、复训率、高频薄弱点、历史记录和推荐补强计划

#### Scenario: 深度报告
- **WHEN** 用户打开某次面试或训练详情
- **THEN** 页面展示题目级表现、回答质量分析、薄弱点归因、代码过程分析、行动任务包和推荐复训项

### Requirement: 全量数据保留与画像沉淀
系统 SHALL 对用户的关键训练与面试数据进行体系化持久化，以便支持后续画像分析、复盘与推荐。

#### Scenario: 持久化核心记录
- **WHEN** 用户进行面试、专项训练、代码作答或复盘
- **THEN** 系统保留面试计划、轮次记录、题目记录、回答记录、代码记录、评分记录、报告记录和薄弱点记录

#### Scenario: 数据主权
- **WHEN** 用户希望查看或导出历史数据
- **THEN** 系统具备面试成长档案导出能力的设计预留

## Data Architecture Draft
v2.0 需要新增或重构以下核心数据实体：

- `UserGrowthProfile`
  - 记录用户当前阶段、目标岗位、目标公司、目标职级、当前画像摘要、最新雷达图快照
- `UserProgressSnapshot`
  - 记录训练次数、连续天数、复训率、最近训练、最近复盘、推荐行动等聚合数据
- `WeaknessDimension`
  - 统一定义薄弱点字典，如知识点、表达能力、项目深度、系统设计、代码能力、HR 沟通等
- `UserWeaknessRecord`
  - 记录用户某个薄弱点的频次、严重等级、最近出现时间、最近得分、来源会话
- `InterviewPlan`
  - 一次完整的面试计划，可对应阶段面试或全流程面试
- `InterviewStagePlan`
  - 某个计划中的轮次定义，如一面、二面、三面、HR 面、Offer Review
- `InterviewSession`
  - 用户实际进行的一次会话
- `InterviewRound`
  - 某次 session 中的轮次执行记录
- `InterviewQuestion`
  - 面试中出现的题目、考察点、触发来源、是否追问、是否代码题
- `InterviewAnswer`
  - 用户回答文本、语音转写、回答时长、是否被打断、提示次数
- `CodingSession`
  - 代码作答空间，记录语言、题目、倒计时、编辑版本和运行记录
- `CodingSubmission`
  - 单次运行或提交结果，记录测试结果、代码快照、耗时、错误信息
- `InterviewScoreCard`
  - 评分结果，按维度拆分，如技术深度、表达逻辑、系统设计、代码能力、稳定性等
- `InterviewReport`
  - 总结、亮点、问题、建议、行动任务包、推荐学习路径
- `AgentExecutionLog`
  - 记录 Planner / Interviewer / Scoring / Summary / Report 等 Agent 的输入输出摘要
- `ReplayTask`
  - 复训任务、改进任务包和后续训练建议

## Agent Architecture Draft
建议把 Agent 评审团拆成以下最小可行角色：

- `Planner Agent`
  - 读取用户目标、简历、JD、公司信息、历史弱项，输出本场或全流程计划
- `Resume / JD Analyst Agent`
  - 负责结构化简历解析、JD 解析、风险点提取、能力对齐
- `Interviewer Agent`
  - 负责主问题和实时追问
- `Evidence Agent`
  - 负责为问题提供依据，避免无来源追问和模型瞎问
- `Code Interview Agent`
  - 负责代码题发题、追问、提示策略和代码过程点评
- `Scoring Agent`
  - 负责题目级、轮次级与整场级评分
- `Summary Agent`
  - 负责人类可读的亮点与短板总结
- `Report Agent`
  - 负责最终复盘报告、任务包和后续推荐
- `Coach Agent`
  - 负责把薄弱点回流到专项训练和首页推荐

## Delivery Strategy
为了保证 v2.0 做得像“正规平台”，而不是堆需求，建议按以下顺序执行：

1. 先落数据底座和状态机骨架。
2. 再做首页 1:1 复刻与真实进度/薄弱点承接。
3. 再重做发起面试的双模式与画像确认页。
4. 再做面试计划与多轮执行框架。
5. 再接入代码面试间。
6. 再重做复盘中心。
7. 最后逐步增强多模态、语音、视频、情绪识别、白板等高阶能力。

## Non-Goals For First Milestone
- 当前里程碑不以“学习中心整体重做”为目标。
- 当前里程碑不以“会员管理体系整体重做”为目标。
- 当前里程碑不承诺一次性完成所有多模态能力的生产级上线。
- 当前里程碑不承诺一次性接入所有外部平台数据源（LeetCode / GitHub / 牛客），但需要为后续接入留出模型和事件结构。
