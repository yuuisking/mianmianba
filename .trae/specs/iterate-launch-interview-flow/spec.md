# 发起面试链路迭代 Spec

## Why
当前“发起面试”链路已经具备基本可用性，但在面试模式命名、简历必传校验、解析结果展示、无数据处理、题量/时长控制、音频面试体验和离开面试保护等关键环节存在明显产品问题。为了避免影响其他模块，本次迭代只允许改动“发起面试”菜单下的相关功能与其直接依赖的面试房间逻辑。

## What Changes
- 仅重构“发起面试”菜单下的配置、解析确认、进入面试房间和进行面试的链路，不改动其他现有模块功能。
- 保持当前已收敛的全站 UI 风格不变，不对无关页面做样式改造。
- 将原“语音 / 视频”模式合并为一个新的实时面试模式，默认以音频为主，允许用户在面试中自行开启/关闭本地视频预览。
- 为简历上传增加强制校验，没有简历时禁止继续解析和进入后续流程。
- 将简历解析结果改为“简短 Markdown 摘要 + 用户画像主展示”，不再只显示文件名，也不展示大段冗余原文。
- 严格移除发起面试链路中的假数据兜底：简历、JD、画像、技术栈、项目、差距提示、专项训练重点等任一关键数据拿不到时，必须明确提示用户，禁止自行 mock。
- 为所有模式的面试增加统一的题量或时长控制，并提供 `不限` 选项；当用户选择有限上限时，到达上限后自动结束。
- 将实时面试房间按“会议式”体验重构，参考通用会议软件的操作模型，但配色、字体和控件风格继续遵循 `brand-guidelines` 与当前系统视觉基调。
- 在所有面试过程中拦截导航离开行为；若用户点击其他菜单或离开当前页，弹窗提示“是否结束本次面试”。对于已形成有效面试会话的场景，确认结束后应写入面试记录；若刚进入面试即秒退、尚未形成有效会话，可不强制计入面试次数。
- 修复专项训练重点未正确进入 prompt 的问题，并系统性重构面试 prompt，确保不同模式、题量/时长策略、专项重点和真实解析数据都能稳定生效。

## Impact
- Affected specs: 发起面试配置、简历解析确认、实时面试房间、文字面试进度控制、专项训练 prompt、面试离开保护
- Affected code: `src/app/setup/page.tsx`、`src/app/profile/page.tsx`、`src/app/interview/page.tsx`、`src/app/api/parse/route.ts`、`src/app/api/chat/route.ts`、相关会话/报告写入逻辑、必要的客户端状态与弹窗组件

## ADDED Requirements
### Requirement: 简历强制上传
系统 SHALL 在“发起面试”链路中要求用户必须上传简历，未上传时不得继续解析或进入画像确认。

#### Scenario: 未上传简历时尝试继续
- **WHEN** 用户未上传简历就点击解析或下一步
- **THEN** 系统阻止继续
- **AND** 页面明确提示“请先上传简历”

### Requirement: 实时面试统一模式
系统 SHALL 将原有语音面试与视频面试合并为一个统一的实时面试模式，以音频交流为主，并允许用户在房间内选择是否开启本地视频。

#### Scenario: 用户选择实时面试
- **WHEN** 用户在“发起面试”中选择实时面试模式
- **THEN** 系统进入同一个实时面试房间
- **AND** 用户默认可直接进行音频交流
- **AND** 用户可以在房间内主动开启或关闭本地视频预览

### Requirement: 无 mock 数据兜底
系统 SHALL 在发起面试链路内完全禁止使用 mock 数据伪造简历解析、JD 解析、用户画像、技术栈、项目、差距预警和专项训练重点。

#### Scenario: 简历或 JD 未解析出有效信息
- **WHEN** 解析结果缺少关键字段或为空
- **THEN** 系统提示用户重新上传、补充或修改输入
- **AND** 页面不展示虚构的技术栈、项目、差距预警或其他伪造内容

### Requirement: 面试离开保护
系统 SHALL 在面试过程中拦截用户通过菜单或页面跳转离开当前面试，并在确认结束后结束当前会话。对于已经形成有效会话的面试，系统应记录本次面试；对于刚进入即离开的无效会话，可不强制记录。

#### Scenario: 面试中点击其他菜单
- **WHEN** 用户处于进行中的面试状态并点击其他导航菜单
- **THEN** 系统弹出“是否结束本次面试”确认提示
- **AND** 用户取消时停留在当前面试
- **AND** 用户确认结束时终止当前面试

#### Scenario: 有效面试会话结束
- **WHEN** 用户已经进入有效面试流程并确认结束
- **THEN** 系统写入或更新本次面试记录

#### Scenario: 刚进入即离开
- **WHEN** 用户刚进入面试房间就立即结束或跳走，尚未形成有效会话
- **THEN** 系统可以不强制记录一次面试次数

## MODIFIED Requirements
### Requirement: 发起面试解析确认页
系统 SHALL 在简历和 JD 解析完成后，以“用户画像”为主、简历摘要为辅的方式展示结果，并明确标识缺失项，而不是仅显示文件名或使用假数据填充。

#### Scenario: 解析成功
- **WHEN** 简历和 JD 成功解析出有效信息
- **THEN** 页面展示用户画像核心信息
- **AND** 页面额外展示一个简短的 Markdown 简历摘要
- **AND** 页面不展开冗长原文

#### Scenario: 解析失败或信息不足
- **WHEN** 简历、JD 或画像关键内容缺失
- **THEN** 页面提示用户解析失败或信息不足
- **AND** 页面不展示任何虚构内容

### Requirement: 所有面试模式的结束条件
系统 SHALL 允许用户在发起面试时为所有模式配置统一的题量或时长策略，并包含 `不限` 选项；当用户选择有限题量或有限时长时，到达上限后自动结束当前面试。

#### Scenario: 按题量结束
- **WHEN** 用户为任一面试模式选择固定题量（如 5 / 10 / 20 / 不限）
- **THEN** 当前面试按真实题量推进
- **AND** 达到题量上限后自动结束面试
- **AND** 当用户选择 `不限` 时，系统不因题量自动结束

#### Scenario: 按时长结束
- **WHEN** 用户为任一面试模式选择固定时长（如 10 分钟 / 20 分钟 / 不限）
- **THEN** 面试在倒计时结束后自动结束
- **AND** 当用户选择 `不限` 时，系统不因时长自动结束

### Requirement: 系统性 Prompt 生效
系统 SHALL 为“发起面试”链路建立统一、可追踪的 prompt 组装机制，确保面试模式、简历解析结果、JD 解析结果、用户画像、专项训练重点以及题量/时长策略都能真实进入 AI 面试官上下文。

#### Scenario: 面试配置真实进入 Prompt
- **WHEN** 用户完成发起面试配置并进入任意模式面试
- **THEN** 系统按统一规则组装 prompt
- **AND** prompt 只使用真实解析或真实配置数据
- **AND** 缺失数据时以提示或限制继续代替 mock 造数

#### Scenario: 专项训练重点驱动提问
- **WHEN** 用户在发起面试时选择了专项训练重点
- **THEN** AI 面试官围绕该重点进行连续提问
- **AND** 不退化为泛泛而谈的通用提问
- **AND** 不因 prompt 拼装缺失而忽略用户选择

## REMOVED Requirements
### Requirement: 语音模式与视频模式分离
**Reason**: 两者核心链路相同，差异仅在于是否展示本地视频，继续分离会增加用户理解成本和维护成本。
**Migration**: 统一合并为一个实时面试模式，在房间内提供视频开关控制。

### Requirement: 关键数据缺失时使用 mock 内容补位
**Reason**: 这会直接破坏面试结果的可信度，也是本次需求明确禁止的底线。
**Migration**: 对所有关键缺失数据改为明确报错、空状态或要求用户补充输入，不再生成伪造内容。
