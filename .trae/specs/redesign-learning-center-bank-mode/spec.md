# 学习中心重做 Spec

## Why
当前学习中心和学习管理同时存在产品形态不统一、视觉风格笨重、信息架构混乱、AI 生成链路不清晰的问题。需要基于新的产品共识重新定义前台学习中心与后台学习管理，并明确“面面吧智能助手”必须完全独立、不可被本次重构影响。

## What Changes
- 前台学习中心彻底去掉“知识库”概念，改为更接近题库/分类/题目的浏览模式。
- 前台列表页与题库页参考 `mianshiya` 的克制布局、信息密度与交互节奏，但不得直接照搬视觉细节或文案。
- 题目详情页继续保留现有“语雀式阅读工作区”方向，不推翻既有阅读模式。
- 学习中心首页、题库页与题目页进一步去“UI 味”，统一改为超宽、低装饰、以正文为中心的学习界面。
- 用户进入具体题库后不再停留在中间题库页做二次选择，而是直接进入首题文档页，并在左侧看到整本题库的目录树。
- 题目页顶部操作收敛为最少必要动作，并新增“知识点模式 / 面试模式”双阅读模式，突出学习而不是操作堆叠。
- 题库卡片去掉首字母图标，增加轻量颜色层次，但保持克制，不得做成重装饰卡片。
- 后台 Agent 生成的每篇文档必须强制包含“知识点总结 / 面试常问 / 参考答案和解析”三段结构，并带发布前强校验。
- 继续保持按细分主题拆题库，但每个题库必须显著扩题，减少知识点过度分散和用户来回查找。
- 后台学习管理重做为“AI 生成题库”的管理系统，第一阶段直接产出分类、题目、答案、解析、代码示例等完整题目内容。
- 后台 AI 生成链路采用多 Agent 协作与编排调度模式，但首版优先使用“可观测、可重试、可人工介入”的收敛实现，不做过度复杂化。
- 历史学习中心数据按新方案直接清空，不保留旧前台数据结构作为线上来源。
- “面面吧智能助手”保持完全独立，不得因学习中心或学习管理重构而不可用。

## Impact
- Affected specs: 学习中心前台信息架构、题库浏览、题目阅读、学习管理后台、AI 生成链路、旧数据清空策略
- Affected code: `src/app/learning/**`、`src/app/admin/learning/**`、`src/app/api/learning/**`、`src/app/api/admin/learning/**`、`data/learning-center/**`

## ADDED Requirements
### Requirement: 题库化学习中心首页
系统 SHALL 提供去“知识库化”的学习中心首页，首页以题库、分类、题目入口和筛选能力为中心，不再让用户感知“知识库”概念。

#### Scenario: 用户进入学习中心首页
- **WHEN** 用户打开学习中心首页
- **THEN** 页面展示题库化列表、分类筛选、搜索或切换入口
- **AND** 页面文案不出现“知识库”概念
- **AND** 页面风格保持克制、低装饰、弱气泡化

### Requirement: 题库页舒适浏览体验
系统 SHALL 提供接近面试题库产品的题库页浏览体验，强调列表清晰、层次稳定、操作直接。

#### Scenario: 用户进入某个题库页
- **WHEN** 用户从学习中心进入某个题库
- **THEN** 页面展示该题库下的分类、题目数量、学习进度和题目列表
- **AND** 页面不使用厚重卡片堆叠或明显 AI 气泡风格
- **AND** 用户可以快速切换分类并定位到具体题目

### Requirement: 学习页面采用超宽正文布局
系统 SHALL 将学习中心首页、题库页和题目页统一收口为超宽内容区，减少无意义留白，让用户视线优先落在题库和正文内容上。

#### Scenario: 用户浏览学习中心或题库
- **WHEN** 用户打开学习中心首页、某个题库或某道题目
- **THEN** 主内容区采用明显更宽的排版宽度
- **AND** 页面保留必要安全边距，但不得维持当前偏窄、偏拥挤的容器宽度
- **AND** UI 装饰弱化，正文、目录和题目内容成为视觉重心

### Requirement: 进入题库后直接进入首题文档页
系统 SHALL 在用户进入具体题库时直接打开该题库的首题文档页，而不是先停留在需要再次选择分类的中间页。

#### Scenario: 用户从题库列表进入某个题库
- **WHEN** 用户点击任一题库卡片
- **THEN** 系统直接进入该题库的首题文档阅读页
- **AND** 左侧目录展示整个题库的分类与题目树
- **AND** 用户无需先进入一个中间页再二次选择分类

### Requirement: 题目页支持知识点模式与面试模式
系统 SHALL 在题目阅读页提供两种清晰的学习视角：知识点模式和面试模式。

#### Scenario: 用户切换题目阅读模式
- **WHEN** 用户在题目页切换阅读模式
- **THEN** 知识点模式优先展示知识点总结
- **AND** 面试模式优先展示面试常问，并提供对应的查看答案入口
- **AND** 顶部操作区只保留必要动作，避免抢占正文注意力

### Requirement: 文档三段结构强制化
系统 SHALL 强制所有正式题目文档都包含“知识点总结 / 面试常问 / 参考答案和解析”三段核心结构。

#### Scenario: Agent 生成或发布一篇题目文档
- **WHEN** 后台 Agent 生成题目内容或发布前校验文档
- **THEN** 文档必须包含“知识点总结”“面试常问”“参考答案和解析”三个结构化板块
- **AND** 任一板块缺失、过短、空泛或主题不相关时，系统不得将其视为可发布内容
- **AND** 校验失败时必须要求重试、修订或人工审核

### Requirement: 细分题库保持独立但必须扩题
系统 SHALL 保持按细分方向拆分题库，但每个题库必须提供足够密度的题量和覆盖度，避免知识点过度分散。

#### Scenario: 初始化或扩充某个细分题库
- **WHEN** 系统生成或补充如 Java 集合、Java 并发、MySQL 等细分题库
- **THEN** 每个题库都应覆盖该主题的核心知识脉络与高频题
- **AND** 题量必须明显高于当前首版样例规模
- **AND** 用户应能在单个题库内完成较完整的连续学习，而不是被迫跨多个题库来回查找

### Requirement: Agent 文档质量必须可量化治理
系统 SHALL 为文档质量建立明确的生成约束、审核维度和最低发布门槛，而不是只检查是否生成成功。

#### Scenario: 后台审核一批 Agent 生成文档
- **WHEN** 系统对生成内容做质检
- **THEN** 必须检查主题相关性、信息密度、结构完整性、表达可学性和面试可用性
- **AND** 低质量文档必须被拦截，不得进入前台正式题库
- **AND** 质量策略必须能持续推动题库从“能看”提升到“能学、能答题”

### Requirement: 题目页保留语雀式阅读工作区
系统 SHALL 保留语雀式题目阅读工作区，确保题目详情仍然以高可读性的正文阅读体验为主。

#### Scenario: 用户进入具体题目
- **WHEN** 用户打开某道题目
- **THEN** 页面继续使用语雀式阅读布局呈现题干、答案、解析、代码示例和关联内容
- **AND** 不得回退为普通题目卡片堆叠页

### Requirement: 面面吧智能助手独立可用
系统 SHALL 保证“面面吧智能助手”在本次学习中心重构期间保持独立和稳定。

#### Scenario: 学习中心重构中出现故障
- **WHEN** 学习中心或学习管理页面、接口、数据链路出现问题
- **THEN** “面面吧智能助手”仍可按现有能力正常使用
- **AND** 本次学习中心重构不得修改其核心交互、能力边界和稳定性

### Requirement: AI 全量生成学习管理
系统 SHALL 提供以 AI 为主的学习管理后台，第一阶段即可生成完整题库内容，而不是只生成空目录或题目骨架。

#### Scenario: 管理员发起一次题库生成
- **WHEN** 管理员输入一个学习方向或课题并发起生成
- **THEN** 系统生成分类、题目、标准答案、解析、代码示例以及必要的补充内容
- **AND** 生成结果可在后台被查看、审核、发布和再次生成

### Requirement: 多 Agent 编排生成链路
系统 SHALL 将后台题库生成实现为多 Agent 协作链路，而不是单 Prompt 一次性生成全部内容。

#### Scenario: 管理员发起一次题库生成任务
- **WHEN** 管理员输入一个方向或课题并点击开始生成
- **THEN** 系统按编排顺序执行结构规划、题目生成、答案生成、质量审核、关联构建、持久化发布准备等阶段
- **AND** 每个阶段都有明确的输入输出契约、状态记录和错误日志
- **AND** 任一阶段失败时可重试、可定位、可人工介入

### Requirement: 首版多 Agent 方案必须收敛可落地
系统 SHALL 对多 Agent 方案进行首版收敛，优先保证稳定性、可调试性和前后台一致性，而不是追求过度复杂的智能体数量。

#### Scenario: 实现第一版后台生成系统
- **WHEN** 团队落地多 Agent 方案
- **THEN** 首版应优先保证主流程稳定跑通
- **AND** 非 AI 的持久化、发布、状态推进逻辑必须由代码实现，不作为模型职责
- **AND** 不要求首版一次引入过多智能角色，只要能满足结构规划、内容生成、质量审核三类核心职责即可

### Requirement: 新题库生成主链路以统一数据源为准
系统 SHALL 为前后台定义唯一主数据源，避免生成结果落库后前台不可见或前后台读写割裂。

#### Scenario: 后台生成题库并准备前台消费
- **WHEN** 后台完成一次题库生成
- **THEN** 生成结果必须写入前台可直接消费的统一数据结构
- **AND** 首版不得继续依赖“Prisma 一套、文件仓一套”的双轨正式链路
- **AND** 如果数据库模型尚未具备稳定迁移基础，首版应优先使用现有稳定主链路作为唯一正式来源

### Requirement: 多 Agent 生成必须具备内容依据与防幻觉约束
系统 SHALL 为多 Agent 生成过程补充依据约束、格式验证和质量门槛，避免生成空泛、失真或无法发布的题目内容。

#### Scenario: Agent 生成分类、题目和答案
- **WHEN** 任一 Agent 输出结构化结果
- **THEN** 系统必须对输出做 JSON 契约校验和必要字段校验
- **AND** 质量审核阶段必须能够识别空泛答案、主题串味、技术错误和无关分类
- **AND** 当内容不满足门槛时，系统应回退到重试、人工修改或标记待审核，而不是直接发布

### Requirement: 旧学习中心数据直接清空
系统 SHALL 允许本轮重构直接放弃旧学习中心前台数据，并以新结构重新开始。

#### Scenario: 新版学习中心上线前
- **WHEN** 新版学习中心与后台发布前进行数据准备
- **THEN** 旧学习中心前台历史数据不再作为新版本数据源
- **AND** 新版仅消费本轮定义的新数据结构和新生成结果

## MODIFIED Requirements
### Requirement: 学习中心信息架构
学习中心 MUST 从“知识库入口 + 知识文档浏览”的产品表述，修改为“题库入口 + 分类与题目浏览”的产品表述。题目阅读页仍保留语雀式工作区，但列表页、题库页和后台管理页全部围绕题库化结构重新定义；同时用户进入具体题库后应直接落到首题文档页，通过左侧整库目录树完成后续导航。

### Requirement: 学习管理后台目标
学习管理后台 MUST 从“维护知识库/目录/来源”的旧思路，修改为“配置 AI 生成题库、查看生成结果、审核发布内容”的新思路。后台设计应优先解决 AI 如何稳定生成高质量分类与题目，而不是继续强化旧知识工厂术语；同时必须内建多 Agent 编排、阶段状态、失败重试、人工介入和任务可观测能力，并把“知识点总结 / 面试常问 / 参考答案和解析”作为每篇正式文档的硬性结构门槛。

## REMOVED Requirements
### Requirement: 前台知识库化心智
**Reason**: 该产品心智与用户期待不符，也导致前台页面和后台管理长期割裂。  
**Migration**: 前台统一切换到题库化结构，旧“知识库”文案、入口和展示方式不再保留。

### Requirement: 旧学习中心历史数据继续沿用
**Reason**: 旧数据结构和旧页面思路会持续污染新版本判断，阻碍重做。  
**Migration**: 新版本直接启用新数据结构与新题库内容，旧前台数据不迁移为线上正式来源。

### Requirement: 进入题库后先停留在中间题库页再选分类
**Reason**: 该流程增加无意义的二次选择，打断用户直接学习的路径。  
**Migration**: 用户进入题库后直接落到首题文档页，分类切换与题目跳转统一通过左侧整库目录树完成。
