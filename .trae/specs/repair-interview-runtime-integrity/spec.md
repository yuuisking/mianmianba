# 面试运行时完整性修复 Spec

## Why
当前面试主链存在严重的运行时完整性问题：新开面试会串入旧房间历史、题目会锚到不属于候选人的项目、用户可见文案不通顺且 Markdown 渲染泄漏、面试官没有真实的结束判断，也缺失“表现积极后转算法题”的关键分支。这些问题直接破坏了“真实智能面试官”这一核心产品承诺，必须单独立项收口，而不能继续散落在宽泛蓝图中。

## What Changes
- 新增“面试房间隔离与冷启动”能力，确保新的 `planId / stageId / roundId / mode` 组合只消费当前轮次的上下文，不允许串入旧历史或旧缓存
- 新增“题目证据锚定”能力，要求首题与追问只能基于当前候选人简历、画像、计划轮次和已确认项目证据生成
- 修改面试开场与题目渲染规则，移除用户可见的 Markdown 泄漏与不通顺招呼语
- 新增“消极回答主动结束”策略，当候选人连续给出低信息量、拒答或消极回答时，面试官可以主动结束本轮并写入原因
- 新增“积极回答转算法题”策略，当候选人表现达到阈值时，系统弹出算法题入口并进入限时作答流程
- 明确禁止以本地缓存、旧会话或兜底文案伪造新面试的提问内容

## Impact
- Affected specs: `build-interview-moat-blueprint`, `fullflow-interview-core`
- Affected code: `src/app/interview/page.tsx`, `src/app/api/chat/route.ts`, `src/lib/interview/prompt.ts`, `src/app/setup/page.tsx`, `src/app/api/sessions/*`, `src/app/api/messages/*`, `src/app/api/v2/interview-plans/*`, 运行时存储与算法题触发链路

## Current State
- 当前会话并非只存浏览器，也并非只走后端正式链路，而是“数据库正式表 + 前端本地快照”双轨并存
- 老链路已有 `InterviewSession / Message`，新全流程链路已有 `InterviewPlan / InterviewPlanStage / InterviewRound / InterviewQuestionRecord / InterviewAnswerRecord / CodingSession`
- 当前面试页恢复逻辑仍大量依赖 `sessionStorage / localStorage`，因此新链接串旧历史、串错轮次的风险真实存在
- 算法题目前更像“数据模型与编排占位已存在”，但独立算法题产品页、LeetCode 风格作答体验、运行判题与提交流程未完整收口
- Agent 团蓝图与轮次职责定义相对完整，但运行时真正稳定执行的提问链主要仍是 `Planner -> Evidence -> Composer -> Guard -> 主面试官`
- Prompt 已经开始往真人面试官口径收敛，但远未达到“完美”，其表现强依赖运行时隔离、证据锚定、状态决策与渲染链路是否正确
- 当前缺少面向面试官行为质量的自动化测试体系，尚未建立“会话隔离 / 首题锚定 / 主动结束 / 算法题触发 / 文案渲染”这一组回归保障

## ADDED Requirements
### Requirement: 面试房间运行时隔离
系统 SHALL 以 `planId + stageId + roundId + mode` 作为面试房间运行时隔离主键，新的房间进入时只能恢复与该主键完全匹配的当前轮次上下文。

#### Scenario: 新开面试进入房间
- **WHEN** 用户打开一个带有明确 `planId`、`stageId`、`roundId`、`mode` 参数的新面试链接
- **THEN** 系统只读取当前轮次的消息、画像、状态和快照
- **THEN** 不得混入其他轮次、其他房间、其他候选历史消息
- **THEN** 如果本地缓存与当前主键不匹配，系统必须忽略旧缓存并执行冷启动

#### Scenario: 旧缓存存在但链接已切换
- **WHEN** 浏览器中存在上一次面试留下的 `localStorage` 或 `sessionStorage`
- **THEN** 系统必须先校验缓存绑定的房间主键
- **THEN** 主键不一致时不得恢复旧内容，不得展示旧题目或旧追问

#### Scenario: 后端与本地同时存在状态
- **WHEN** 数据库中的正式消息记录与浏览器本地快照同时存在
- **THEN** 系统必须以当前房间主键绑定的数据为准进行合并或恢复
- **THEN** 不得因为“最近一次本地快照”覆盖当前面试的正式服务端状态

### Requirement: 题目必须锚定当前候选证据
系统 SHALL 只基于当前候选人的简历、画像、岗位、公司、轮次和已确认项目证据来生成首题与追问。

#### Scenario: 首题生成
- **WHEN** 新面试的第一个问题被生成
- **THEN** 问题必须引用当前候选人真实存在的项目、经历或能力点
- **THEN** 如果证据不足，系统应要求补充信息或降级到安全的澄清问题
- **THEN** 不得引用其他候选人的项目名、历史房间项目名或虚构项目

#### Scenario: 跳过或不知道后继续追问
- **WHEN** 用户点击“跳过本题”或发送“我不知道”类低信息回答
- **THEN** 后续问题仍必须留在当前候选人的真实证据范围内
- **THEN** 不得围绕错误项目持续深挖

### Requirement: 用户可见文案与渲染正确
系统 SHALL 以自然、通顺、适合真实面试的中文输出开场和题目，并保证用户侧不暴露 Markdown 控制符或内部格式残片。

#### Scenario: 开场展示
- **WHEN** 面试房间首次展示开场语
- **THEN** 用户应看到自然中文招呼语
- **THEN** 不得出现 `**`、原始 Markdown、内部标签或格式错乱

#### Scenario: 题目渲染
- **WHEN** 系统展示首题或追问
- **THEN** 文本高亮、换行和强调应以统一渲染规则输出
- **THEN** 不得出现“句子不通顺但带格式符号”的混合产物

### Requirement: 消极回答可触发主动结束
系统 SHALL 在候选人连续给出低信息量、拒答、跳过、消极或明显失去作答意愿时，允许面试官主动结束当前面试。

#### Scenario: 连续消极回答
- **WHEN** 候选人在同一轮次连续多次给出“跳过”“不知道”“不清楚”“不想答”或其他低信息回答
- **THEN** 系统应累计消极信号并判断是否达到主动结束阈值
- **THEN** 达到阈值后，面试官可以直接结束本轮
- **THEN** 结束结果、触发原因和上下文应被结构化写入状态与报告链路

### Requirement: 积极回答可转入算法题
系统 SHALL 在候选人表现积极且满足指定阈值时，提供进入算法题环节的明确触发与交互。

#### Scenario: 触发算法题
- **WHEN** 候选人在当前轮次表现达到预设阈值，且当前流程允许进入算法题
- **THEN** 系统应弹出算法题确认入口，而不是静默跳过
- **THEN** 候选人确认后进入限时算法题流程
- **THEN** 算法题结果应被写入当前面试评估上下文

### Requirement: 算法题必须产品化落地
系统 SHALL 提供真正可用的算法题作答页面与交互，而不是只停留在 `CodingSession` 数据模型或轮次占位标记。

#### Scenario: 进入算法题页面
- **WHEN** 候选人确认进入算法题环节
- **THEN** 系统应展示独立的算法题页面或视图
- **THEN** 页面至少包含题面、语言切换、代码编辑区、运行、提交、倒计时和结果反馈
- **THEN** 体验可以不必一比一复制 LeetCode，但必须达到“真实可答、可跑、可提、可判”的产品标准

### Requirement: Agent 团运行职责必须与蓝图一致
系统 SHALL 明确区分“已定义的 Agent 蓝图”和“当前运行时真实执行的 Agent 链”，避免把编排种子或占位角色包装成已完整落地能力。

#### Scenario: 运行时提问链
- **WHEN** 系统执行一轮新的提问或追问
- **THEN** 必须能清楚说明哪些 Agent 在本轮真实参与
- **THEN** 未真实运行的角色不得被当作已交付能力对外宣称

### Requirement: Prompt 质量必须作为正式交付物管理
系统 SHALL 把主面试官与提问链 prompt 视为正式产品资产，而不是一次性试验文案。

#### Scenario: 调整 prompt
- **WHEN** 团队重写 Planner、Composer、Guard 或主面试官 prompt
- **THEN** 必须同步验证语气自然度、面试官真实感、错误项目拦截与内部词泄漏风险
- **THEN** 不得仅以“模型能返回内容”作为通过标准

### Requirement: 面试官行为必须有自动化回归
系统 SHALL 为关键面试官行为建立可重复执行的自动化校验，而不只依赖人工走查。

#### Scenario: 提交面试主链改动
- **WHEN** 提问链、恢复链、结束状态机、算法题触发链或用户可见文案被修改
- **THEN** 系统应运行针对会话隔离、首题锚定、文案渲染、主动结束和算法题触发的自动化回归
- **THEN** 任一关键校验失败时不得视为主链验收通过

## MODIFIED Requirements
### Requirement: 面试恢复逻辑
系统 SHALL 优先恢复当前房间主键对应的真实运行时状态，而不是“只要本地有历史就直接续上次面试”；当服务端正式记录与本地快照同时存在时，必须按主键严格校验并防止串房间。

### Requirement: 面试追问逻辑
系统 SHALL 以当前候选证据和当前轮次目标为边界生成追问；当候选人反复无法回答时，系统可切换验证点或主动结束，但不得继续围绕错误项目反复追问。

### Requirement: 算法题链路
系统 SHALL 从“计划里可标记 codingRequired”升级为“候选人可见、可进入、可运行、可回写结果”的完整产品链路。

### Requirement: 面试官开场输出
系统 SHALL 输出自然、简洁、像真人面试官会说的话；开场文案不得混入内部说明、Markdown 控制符或不完整格式。

## REMOVED Requirements
### Requirement: 基于宽松本地快照的房间恢复
**Reason**: 旧规则把“尽可能恢复任何最近面试”放在“当前房间真实性”之前，导致新链接也可能串到旧历史。
**Migration**: 恢复逻辑迁移为强主键校验；历史缓存仅在完全匹配 `planId + stageId + roundId + mode` 时才可恢复。

### Requirement: 候选人持续消极时只允许继续追问
**Reason**: 旧规则缺少真实面试中的“面试官主动结束”能力，导致系统在低质量对话中无限拖延。
**Migration**: 改为“继续追问 / 切换验证点 / 主动结束”三选一决策。
