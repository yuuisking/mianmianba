# 学习中心与知识工厂重构 Spec

## Why
当前学习中心只达到了“可浏览、可录入、可 AI 摘要”的基础水平，距离“学习 + 模拟面试一体化”的系统目标仍然差距很大。需要将其升级为一个对用户像语雀一样好用、对管理员像内容工厂一样高产的核心模块，真正承载系统中的“学习”能力。

## What Changes
- 重构公开的学习中心信息架构，区分“知识库大类列表页”和“知识库阅读工作区”两层体验。
- 重构知识库阅读页为语雀式三栏工作区：左侧目录、中间文档、右侧大纲，并提供大空间、低拥挤度的阅读布局。
- 将文档内容标准升级为 Markdown 主体内容，支持图片、流程图、架构图、代码示例、面试高频问答等原创内容块。
- 新增文档内 AI 功能：一键总结当前文档、基于岗位目标生成学习计划、从当前文档一键进入专项训练。
- 新增文档内机器人入口，提供接近 DeepSeek 的文档对话体验，覆盖会话管理、历史会话、上下文工程和连续追问。
- 强化 AI 总结与文档问答的真实性约束，禁止在正文信息不足时编造总结、知识点或结论。
- 重构学习计划生成逻辑，要求展示“为什么学这个、依据哪些文档、当前覆盖是否足够”，而不是只输出静态链接列表。
- 重构后台管理为“知识工厂”，覆盖知识库管理、来源采集、分类归纳、多 AI 协同创作、草稿审核与发布。
- 将知识工厂主流程重构为“输入一个知识课题 -> 自动筛选高质量来源 -> 自动生成目录 -> 自动生成草稿 -> 审核发布”的单入口自动化链路。
- 强化知识工厂的来源筛选标准，优先官方文档、权威社区资料和高 star GitHub 项目，避免无边界采集和低质量内容灌入。
- 强化知识领域边界隔离，例如输入 Java 时不得混入 Android 目录，输入 Vue/前端时不得混入无关知识体系。
- 强制统一文档产物结构：开头为总结归纳，中间必须包含流程图/架构图/代码实例等解释块，结尾为面试常考。
- 新增定时采集机制，但仅允许管理员配置的来源白名单进入自动扫描，避免无边界抓取。
- 将最终发布文档的主存储从当前 JSON 结构升级为文件优先存储，采用“本地仓库 + Git 版本化”的内容仓模式。
- **BREAKING**：当前 `data/learning-center.json` 的单文件一体化存储不再作为最终发布内容的主数据源，仅保留为迁移过渡数据。
- **BREAKING**：当前学习中心页面从“简化知识库阅读页”升级为“语雀式知识库工作区”，现有前端路由与返回结构需要整体重构。

## Impact
- Affected specs: `learning-center-system`, `learning-center-cms`, `systematic-knowledge-engine`, `standalone-targeted-practice`, `refactor-auth-and-user-management`
- Affected code: `src/app/learning/page.tsx`, `src/app/admin/learning/page.tsx`, `src/app/admin/learning/[kbId]/page.tsx`, `src/app/api/learning/route.ts`, `src/app/api/admin/learning/*`, `src/lib/db/learningDb.ts`, `src/lib/ai/summarizer.ts`, `src/lib/ai/router.ts`, `src/lib/ai/taxonomy.ts`, `data/learning-center.json`

## ADDED Requirements
### Requirement: 文档机器人对话助手
系统 SHALL 在文档阅读页提供一个显式的机器人 Logo 入口，允许用户围绕当前文档内容与 DeepSeek 风格的 AI 助手进行连续对话。

#### Scenario: 打开文档对话
- **WHEN** 用户在文档页点击机器人 Logo
- **THEN** 页面打开文档对话面板，而不是跳转到新的无关页面
- **AND** 对话面板展示当前会话消息流、输入框和历史会话入口
- **AND** 当前会话默认绑定用户正在阅读的文档上下文

#### Scenario: 连续追问
- **WHEN** 用户围绕当前文档连续提出多个问题
- **THEN** 系统保留会话上下文、最近消息和当前文档的结构化摘要
- **AND** AI 回答风格接近真实 DeepSeek 聊天，不是一次性模板化回复
- **AND** 用户可以基于上一轮回答继续追问而不丢失上下文

#### Scenario: 历史会话管理
- **WHEN** 用户多次在同一知识库或同一文档中发起对话
- **THEN** 系统保存历史会话列表，支持查看、切换和继续对话
- **AND** 历史会话至少区分标题、所属文档、最后更新时间和最近一条消息摘要

#### Scenario: 超出文档范围的提问
- **WHEN** 用户的问题超出当前文档范围，但仍与当前知识库相关
- **THEN** 系统优先使用当前文档，其次使用当前知识库内可定位的相邻文档作为补充上下文
- **AND** 若知识库中没有足够依据，AI 必须明确说明“当前知识库未提供足够信息”
- **AND** 不得伪造不存在于知识库或模型真实能力之外的细节

### Requirement: 学习中心大类入口页
系统 SHALL 提供一个面向所有用户的学习中心入口页，先展示知识库大类，再进入具体知识库阅读工作区。

#### Scenario: 查看知识库大类
- **WHEN** 用户进入学习中心首页
- **THEN** 页面展示由管理员维护的知识库大类列表，例如 Java、前端、AI
- **AND** 用户只能查看，不可修改知识库大类
- **AND** 每个知识库卡片至少展示标题、简介、标签、文档数量、最近更新时间

#### Scenario: 进入某个知识库
- **WHEN** 用户点击某个知识库大类
- **THEN** 系统进入该知识库的阅读工作区
- **AND** 工作区默认打开该知识库下的首页文档或首个推荐文档

### Requirement: 语雀式知识库阅读工作区
系统 SHALL 提供一个 1 比 1 对齐语雀核心阅读心智的三栏工作区，左侧为目录树，中间为文档正文，右侧为文档大纲。

#### Scenario: 三栏布局阅读
- **WHEN** 用户打开知识库中的具体文档
- **THEN** 左侧展示多级目录树，支持一级、二级、三级及更深层级的展开与折叠
- **AND** 中间展示大宽度 Markdown 文档阅读区
- **AND** 右侧展示根据正文标题自动生成的文档大纲
- **AND** 页面布局应以“展开、不拥挤、适合长文阅读”为原则

#### Scenario: 目录切换文档
- **WHEN** 用户在左侧目录树点击某个节点
- **THEN** 中间区域切换到对应文档
- **AND** 右侧大纲同步切换为当前文档的大纲
- **AND** 当前目录节点在左侧保持高亮

#### Scenario: 大纲跳转
- **WHEN** 用户点击右侧大纲中的任意标题
- **THEN** 中间文档滚动到对应段落
- **AND** 当前阅读位置对应的大纲项可高亮

### Requirement: Markdown 文档块能力
系统 SHALL 以 Markdown 为文档主格式，并支持教程型学习内容所需的图文和结构化块。

#### Scenario: 渲染完整文档能力
- **WHEN** 文档被发布到学习中心
- **THEN** 系统能够稳定渲染正文、标题层级、引用、表格、代码块、任务列表、图片、Mermaid 图、流程图说明与面试问答块
- **AND** 文档中的图片、流程图、架构图和代码示例均为平台生成或平台持有的原创整理结果

#### Scenario: 文档元信息展示
- **WHEN** 用户阅读文档
- **THEN** 页面可展示文档标题、更新时间、所属知识库、所属目录路径、阅读建议或学习标签

### Requirement: 文档内 AI 总结
系统 SHALL 在文档阅读页提供 AI 一键总结当前文档的能力。

#### Scenario: 总结当前文档
- **WHEN** 用户在文档页点击“AI 总结当前内容”
- **THEN** 系统基于当前文档正文生成简洁总结
- **AND** 总结结果以侧边面板、弹层或内嵌卡片形式展示
- **AND** 总结内容必须以当前文档为唯一主上下文，不得脱离文档空泛生成

#### Scenario: 文档信息不足时拒绝编造
- **WHEN** 当前文档只有标题、空正文或极少量结构信息，不足以支撑总结
- **THEN** 系统明确提示“当前文档信息不足，暂时无法生成可靠总结”
- **AND** 可以给出“建议补充正文后再总结”的引导
- **AND** 不得凭标题猜测出大量正文内容、细节结论或面试问答

### Requirement: AI 学习计划助手
系统 SHALL 在学习中心提供一个显式的 AI 入口，允许用户基于岗位 JD、目标岗位或学习目标，自动整理现有知识库并输出学习计划。

#### Scenario: 基于目标生成学习计划
- **WHEN** 用户在学习中心输入岗位 JD 或期望目标
- **THEN** 系统从现有知识库中检索相关内容
- **AND** 输出按学习顺序组织的阶段化学习计划
- **AND** 每个阶段至少包含推荐知识库、推荐文档、推荐顺序、学习目标和建议训练入口

#### Scenario: 展示学习依据
- **WHEN** 系统生成学习计划
- **THEN** 每个阶段或每个体系必须说明“为什么推荐学习这一部分”
- **AND** 明确引用触发推荐的知识库、文档主题、用户目标关键词或能力缺口
- **AND** 对学习顺序给出可读的路线说明，而不是只展示链接集合

#### Scenario: 对齐主流学习路线
- **WHEN** 学习计划针对 Java、前端、Vue 等成熟知识方向生成
- **THEN** 系统优先参考该方向的主流学习路径组织顺序
- **AND** 先基础、再核心机制、再工程化/实战、最后面试表达
- **AND** 若当前知识库覆盖与主流路线存在缺口，需要显式标出缺失模块

#### Scenario: 知识库覆盖不足
- **WHEN** 用户目标涉及的知识点在现有知识库中缺失
- **THEN** 计划中明确标注“当前知识库未覆盖”或“建议补充内容”
- **AND** 不得伪造不存在的文档条目

### Requirement: 从文档一键进入专项训练
系统 SHALL 支持用户从任意学习文档直接进入专项训练，并自动带入当前上下文。

#### Scenario: 文档到专项训练联动
- **WHEN** 用户在文档页点击“去训练”
- **THEN** 系统跳转到专项训练模块
- **AND** 自动带入当前文档标题、知识点摘要、推荐追问方向
- **AND** 用户无需手工重复输入主题即可开始训练

### Requirement: 管理员专属知识工厂
系统 SHALL 提供仅管理员可见的后台管理模块，用于创建知识库、维护目录、采集资料、生成文档、审核发布。

#### Scenario: 管理员访问后台
- **WHEN** 非管理员访问后台管理路由
- **THEN** 系统拒绝访问
- **AND** 普通用户在导航中不可见后台管理入口

#### Scenario: 管理员创建知识库
- **WHEN** 管理员创建新的知识库
- **THEN** 可设置知识库名称、简介、标签、封面、排序和是否公开
- **AND** 可定义该知识库的目录骨架和默认首页

#### Scenario: 单课题自动建库
- **WHEN** 管理员只输入一个知识课题，例如“Java”、“Vue”或“前端工程化”
- **THEN** 系统自动完成来源筛选、目录初始化、草稿生成和发布前审核准备
- **AND** 管理员不需要手工逐步驱动“采集、归类、总结”每一小步才可得到初始结果
- **AND** 后台默认提供的是单入口自动化工作流，而不是术语复杂的多面板 CMS

### Requirement: 目录优先的知识体系设计
系统 SHALL 支持由 AI 和管理员共同维护“先目录、后内容”的知识体系。

#### Scenario: AI 生成标准大纲
- **WHEN** 管理员输入一个知识体系主题，例如“Java 后端”
- **THEN** AI 先生成符合学习先后顺序的目录骨架
- **AND** 目录至少支持章节、子章节、文档三级结构
- **AND** 管理员可以对目录进行编辑、重排、删除与补充

#### Scenario: 目录体现学习顺序
- **WHEN** AI 输出目录骨架
- **THEN** 目录需要显式体现从基础到进阶的学习顺序
- **AND** 不能只做松散分类，不给出学习路径

#### Scenario: 领域边界隔离
- **WHEN** 管理员输入的课题是单一知识领域，例如“Java”
- **THEN** 系统生成的目录必须严格保持该领域边界
- **AND** 不得混入 Android、移动端、无关框架或其他独立知识体系
- **AND** 若某个主题存在歧义，系统先按主流行业语义选主领域，再将歧义项标记为候选扩展而不是直接混入目录

### Requirement: 多来源资料采集
系统 SHALL 支持管理员手动贴链接采集资料，并支持系统按计划扫描管理员批准的来源。

#### Scenario: 管理员手动采集
- **WHEN** 管理员输入 GitHub、网页、语雀或其他网页链接
- **THEN** 系统抓取正文或结构化内容
- **AND** 记录来源 URL、采集时间、来源类型和采集状态

#### Scenario: 定时自动采集
- **WHEN** 到达管理员设定的扫描周期
- **THEN** 系统仅扫描来源白名单中的站点或仓库
- **AND** 抓取结果进入待归档队列
- **AND** 不得对未登记来源进行无界抓取

#### Scenario: 自动筛选高质量来源
- **WHEN** 系统围绕某个知识课题自动采集来源
- **THEN** 优先选择官方文档、权威社区文档、经典教程站点和高 star 的 GitHub 项目
- **AND** 来源筛选必须考虑主题相关性、质量等级、更新活跃度和内容可读性
- **AND** 不得为了凑数量而抓取低质量、重复、离题或内容空洞的来源

### Requirement: 多 AI 协同造文档流水线
系统 SHALL 将文档生成设计为多 AI 协作流水线，而不是单个摘要器一次性产出成文。

#### Scenario: 目录规划 AI
- **WHEN** 一个知识体系需要初始化或重整
- **THEN** 目录规划 AI 先输出知识体系骨架、章节顺序和章节目标

#### Scenario: 路由分类 AI
- **WHEN** 系统采集到新的资料
- **THEN** 路由分类 AI 将资料匹配到已有目录节点
- **AND** 若没有匹配节点，可输出“建议新增节点”而不是强行归类

#### Scenario: 内容创作 AI 协同产出
- **WHEN** 某个文档节点进入生成阶段
- **THEN** 系统至少拆分为“总述摘要”、“正文分章节”、“图例/流程图/架构图”、“代码实例”、“面试高频问答”多个产出步骤
- **AND** 每个步骤可由独立 AI 任务负责
- **AND** 最终由合成步骤统一装配成一篇完整文档草稿

#### Scenario: 固定文档骨架输出
- **WHEN** 系统产出任意正式文档草稿
- **THEN** 文档第一部分必须是总结归纳，快速说明主题、定位和核心结论
- **AND** 文档中间必须包含正文解析，并至少覆盖流程图、架构图或代码实例中的一种，优先多种组合
- **AND** 文档最后必须包含面试常考问题与回答提示
- **AND** 若某一部分暂时无法可靠生成，系统必须明确标记缺失，而不是用空泛废话填充

#### Scenario: 避免侵权与简单抄袭
- **WHEN** AI 根据来源资料生成文档
- **THEN** 系统必须将结果产出为原创整理内容
- **AND** 不得直接大段复制来源原文
- **AND** 必须保留来源记录供管理员追溯

### Requirement: 草稿审核与发布
系统 SHALL 区分资料、生成草稿、待审核文档和已发布文档，避免未审内容直接进入学习中心。

#### Scenario: 草稿进入审核
- **WHEN** 多 AI 协同完成一篇文档草稿
- **THEN** 草稿进入待审核状态
- **AND** 管理员可预览最终阅读效果、查看来源、查看 AI 生成痕迹和修改建议

#### Scenario: 发布文档
- **WHEN** 管理员确认草稿合格
- **THEN** 系统将草稿发布到公开学习中心
- **AND** 发布后目录树、文档内容、更新时间和检索索引同步更新

### Requirement: 文件优先的内容存储
系统 SHALL 采用文件优先的内容存储策略，将最终知识库文档保存在本地内容仓中，并支持 Git 版本化。

#### Scenario: 当前最小可行内容仓基线
- **WHEN** 系统在现有学习中心实现上以最小增量落地文件优先存储
- **THEN** 发布内容主目录固定为 `data/learning-center/`
- **AND** 兼容索引文件固定为 `data/learning-center/index.json`
- **AND** 每个知识库固定写入 `data/learning-center/kbs/<kbId>/meta.json`、`tree.json` 和 `topics/<topicId>.json`
- **AND** 草稿固定写入 `data/learning-center/drafts/<kbId>/<draftId>.json`
- **AND** 当前阶段允许已发布内容继续以结构化 JSON 文档落地到内容仓，后续阅读工作区再升级为 Markdown 主体文档

#### Scenario: 发布内容存储
- **WHEN** 文档被发布
- **THEN** 系统将知识库目录结构、Markdown 正文、文档元数据和资源文件写入本地内容目录
- **AND** 文档路径与知识库目录结构保持稳定映射
- **AND** 系统保留来源、草稿、发布版本之间的关联元数据

#### Scenario: 版本追踪
- **WHEN** 文档被更新
- **THEN** 系统保留历史版本信息
- **AND** 能够追踪该版本由哪些来源和哪些 AI 任务生成

### Requirement: 兼顾本地文件与系统索引
系统 SHALL 允许使用数据库保存管理态元数据和任务状态，但最终学习内容不依赖数据库作为唯一真源。

#### Scenario: 元数据分层
- **WHEN** 系统运行采集、生成、审核与发布任务
- **THEN** 数据库可保存任务状态、权限、调度记录、来源索引与发布记录
- **AND** 发布后的文档正文以文件系统内容仓为主真源

#### Scenario: 旧 JSON 迁移兼容
- **WHEN** 新内容仓首次在已有项目中启动
- **THEN** 系统会将 `data/learning-center.json` 作为一次性迁移输入源导入 `data/learning-center/`
- **AND** 导入后公开学习中心与后台管理继续通过兼容层获取 `kbs / trees / contents / drafts` 聚合结构
- **AND** 旧 JSON 不再作为后续写入的主真源，只保留为迁移过渡输入

## MODIFIED Requirements
### Requirement: 学习中心公开体验
现有学习中心不再满足“左侧分类、右侧详情”的简化结构。系统必须升级为“大类入口页 + 语雀式阅读工作区”的双层体验，并支持多级目录、宽文档区、右侧大纲、文档 AI 辅助与文档到专项训练的联动。

### Requirement: 学习中心后台管理
现有后台管理不再仅是“新建知识库 + 贴链接 + AI 摘要挂载”的轻量 CMS。系统必须升级为“知识工厂”，但主交互必须收敛为“输入一个知识课题即可触发全自动生成”的单入口自动化工作流；目录规划、来源采集、智能路由、分步骤创作、草稿审核、发布与定时任务仍保留，但不应要求管理员手工串联所有底层步骤。

### Requirement: 内容生成策略
现有单个 `summarizer` 模型直出结构化文档的方式不再满足要求。系统必须改为多阶段、多任务的 AI 协作流水线，保证内容结构、原创表达、图文生成与面试问答质量；同时所有 AI 产出都必须遵守真实性约束，不得在依据不足时用 mock 式内容补全。

### Requirement: 学习计划输出形态
现有学习计划不能只给出推荐链接集合。系统必须解释推荐依据、学习顺序、能力缺口和知识覆盖范围，并尽量贴近主流学习路线的真实组织方式。

## REMOVED Requirements
### Requirement: 单文件 JSON 作为最终知识主存储
**Reason**: 当前 `data/learning-center.json` 将知识库、目录、正文和草稿全部堆叠在单文件中，不适合大规模文档体系、版本追踪和内容工厂化生产。
**Migration**: 保留该 JSON 作为迁移输入源；新系统将已发布内容迁移到文件内容仓，并将任务态元数据迁移到独立索引层。

### Requirement: 前台轻量卡片式知识阅读
**Reason**: 现有前台学习中心只具备轻量阅读能力，无法承载长文学习、体系导航和语雀式知识管理心智。
**Migration**: 用新的知识库入口页和阅读工作区替换旧页面，同时迁移原有知识库大类和文档内容。

### Requirement: 脱离正文的兜底总结
**Reason**: 当文档几乎没有正文时，兜底总结会制造明显的 mock 感和错误信任，破坏学习中心的真实性。
**Migration**: 当正文信息不足时，改为明确提示“信息不足，无法可靠总结”，并引导补正文或先完成知识工厂产出。

### Requirement: 管理员手工串联所有知识工厂步骤
**Reason**: 纯手工串联“采集、归类、总结、发布”会让知识工厂沦为复杂后台，而不是高质量自动化生产工具。
**Migration**: 保留必要的人工审核和兜底编辑，但默认入口改为“输入课题 -> 自动建库 -> 自动产出草稿 -> 人审发布”的自动化流程。
