# 强化面试官 Prompt 智能度 Spec

## Why
当前专项训练与面试房间里的题目质量不稳定，已经出现“题目过空、过泛、无法训练用户”的明显问题。AI 面试官的提问能力、上下文稳定性和追问质量是系统核心竞争力，必须系统性重构，而不是只修一两句开场文案。

## What Changes
- 重构专项训练与正式面试的 prompt 架构，明确“题目生成规则、追问规则、纠偏规则、上下文保持规则”。
- 为专项训练增加更强的“岗位感 + 主题感 + 训练目标感”，避免出现泛化、空泛或导师式废话题目。
- 建立持续携带的上下文工程结构，确保每一轮都带上关键事实，降低模型丢失上下文后的胡乱提问概率。
- 强化首题生成机制，要求第一题必须是可回答、可追问、能暴露真实能力差异的面试题，而不是抽象要求用户“泛泛讲一讲”。
- 为不同模式区分提问策略：专项训练、文字面试、实时面试不能共用过于宽泛的一套提问口径。
- 为知识检索与题目生成之间建立更清晰的边界，避免“检索”替代“面试设计”，也避免面试官暴露内部链路。

## Impact
- Affected specs: `iterate-launch-interview-flow`, `targeted-training-mode`, `intelligent-mentor-persona`
- Affected code: `src/lib/interview/prompt.ts`, `src/app/api/chat/route.ts`, `src/app/interview/page.tsx`, `src/app/api/interview/resolve-targeted-role/route.ts`, `src/lib/interview/config.ts`

## ADDED Requirements
### Requirement: 高质量专项训练出题
系统 SHALL 在专项训练模式下生成真正可训练用户的高质量问题，首题与后续追问都必须体现岗位、主题、目标和能力层级，而不是泛化空题。

#### Scenario: Java 后端专项训练首题
- **WHEN** 用户以 `Java 后端开发` 岗位发起专项训练，且训练目标为某个具体主题
- **THEN** 首题必须落在 Java 后端真实面试语境里
- **AND** 题目必须是可回答、可追问、可暴露能力差异的问题
- **AND** 题目不能只是要求用户泛泛讲原理、边界和经验

#### Scenario: 训练主题不足时的出题收口
- **WHEN** 训练主题较宽或用户输入较短
- **THEN** 系统仍应基于岗位与真实语境生成一个可落地的问题
- **AND** 不得退化为抽象空泛的大而化之题目

### Requirement: 持续上下文工程
系统 SHALL 在每一轮给模型的 prompt 中持续携带关键上下文，包括岗位、模式、训练主题、训练目标、用户画像、项目、风险项、当前轮次和上一轮回答结论，防止模型丢失上下文后胡乱提问。

#### Scenario: 长轮次专项训练
- **WHEN** 用户已进行多轮专项训练
- **THEN** 新一轮问题仍应保持与最初岗位、主题、目标一致
- **AND** 追问应参考前面已问过的内容、用户的薄弱点和已暴露的问题
- **AND** 不能突然跳成泛化综合面试

### Requirement: 面试官角色能力分层
系统 SHALL 为不同模式定义不同的面试官行为策略，避免所有模式共享同一种“泛导师式”问法。

#### Scenario: 专项训练模式
- **WHEN** 当前模式为专项训练
- **THEN** 提问应更聚焦、更尖锐、更强调纠偏与递进训练
- **AND** 默认优先围绕一个能力点连续打透

#### Scenario: 正式面试模式
- **WHEN** 当前模式为文字面试或实时面试
- **THEN** 提问应更像真实面试流程
- **AND** 允许从自我介绍、项目、原理、场景、追问逐步展开

### Requirement: 首题质量约束
系统 SHALL 对第一题增加质量约束，避免出现“像要求写作文”或“无法验证能力”的弱题。

#### Scenario: 首题生成
- **WHEN** 面试房间刚开始
- **THEN** 首题应优先选择以下类型之一：项目切入题、场景决策题、原理 + 应用结合题、故障排查题、设计取舍题
- **AND** 题目必须能衍生出后续连续追问
- **AND** 不能只给出空泛表述性任务

### Requirement: 题目质量自检
系统 SHALL 在生成问题前或生成后具备最小题目质量自检能力，至少识别“过泛、过空、不可追问、与岗位不符”的问题并重新生成。

#### Scenario: 弱题拦截
- **WHEN** 候选题目过于抽象、过泛或与岗位不符
- **THEN** 系统应阻止该题直接发给用户
- **AND** 改为生成更具体、更有辨识度的问题

## MODIFIED Requirements
### Requirement: 系统性 Prompt 生效
系统 SHALL 为“发起面试”链路建立统一、可追踪、可分层的 prompt 组装机制，确保面试模式、简历解析结果、JD 解析结果、用户画像、专项训练重点以及题量/时长策略都能真实进入 AI 面试官上下文；同时新增题目质量控制、模式化行为策略与上下文持续携带能力，避免模型因上下文漂移而产生低质量问题。

### Requirement: 专项训练重点进入 Prompt
系统 SHALL 不仅让专项训练重点进入 prompt，还要让岗位、专题主题、训练目标、上一轮回答结论与待强化弱点共同决定下一题；AI 面试官必须优先围绕这些真实因素出题、追问与纠偏，而不是只把它们当作背景描述。

## REMOVED Requirements
### Requirement: 专项训练通用开场题
**Reason**: 现有“先讲清主题原理、边界或实战经验”一类通用开场题过于空泛，无法稳定体现岗位感和训练价值。  
**Migration**: 迁移为“按岗位语境 + 训练主题 + 训练目标 + 题目质量约束”动态生成首题，不再使用单一固定模板。
