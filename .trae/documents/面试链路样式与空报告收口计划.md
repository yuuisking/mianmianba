# 面试链路样式与空报告收口计划

## Summary
- 目标：收口“发起面试 / 专项训练 / 面试间 / 报告页”这一条链路，解决你新提出的 5 个问题：配置样式难看、专项训练不应有限制、发起面试的实时视频开关位置不对、模型在专项训练里反复确认、空面试仍生成带分报告。
- 成功标准：
  - `专项训练` 页面不再展示题量或时长上限，默认无限训练，直到用户主动结束。
  - `发起面试` 页面将“题量 / 时长”改成二选一配置；只允许选择一个限制类别；时长类别支持固定枚举 + 自定义分钟数。
  - `实时面试` 的视频控制移入房间内，通过摄像头 icon 随时开关，而不是进入前在 `setup` 页预先决定。
  - `专项训练` 的开场 prompt 与追问规则改成直接开练，不再反复二次确认主题。
  - `刚进入就结束` 或 `没有形成有效问答` 时，不生成评分，不展示伪造的 70 分或其他分数。

## Current State Analysis
- `src/app/practice/page.tsx`
  - 当前仍保留 `questionLimit` / `durationLimitMinutes` 两套选择按钮。
  - 页面底部明确写了“有限上限会自动结束，不限不自动结束”，与“专项训练天然不限”冲突。
  - 当前启动专项训练时会把 `questionLimit` / `durationLimitMinutes` 写入 `parsedProfileData`。
- `src/app/setup/page.tsx`
  - 当前“面试模式”下方有一个 `实时模式视频开关` 区块，按钮文案为“已开视频 / 仅音频”。
  - 当前“统一题量 / 时长”是并列双卡，用户可以同时选题量和时长，不符合“二选一”的新要求。
- `src/lib/interview/config.ts`
  - 当前配置模型是双字段并存：`questionLimit` 与 `durationLimitMinutes`。
  - `QUESTION_LIMIT_OPTIONS` 与 `DURATION_LIMIT_OPTIONS` 都是固定枚举，没有“限制类型”概念，也没有自定义时长输入。
  - `buildInterviewLimitStrategy()` 当前会同时拼接“题量说明 + 时长说明”。
- `src/app/interview/page.tsx`
  - 当前房间 UI 已根据 `profile.videoEnabled` 区分是否展示本地视频区域，但切换发生在进入房间前，不是房间内工具栏控制。
  - 页面顶部目前只显示状态标签，没有明确的“腾讯会议式摄像头按钮”可随时切换。
  - 题量自动结束逻辑在 `nextCompletedRounds >= questionLimit` 时触发；时长自动结束逻辑也在同页存在。
- `src/lib/interview/prompt.ts`
  - 当前专项训练 prompt 已强调围绕主题追问，但没有强约束“禁止再次确认主题”。
  - `buildInterviewOpening()` 的专项训练开场也仍然是相对温和的引导，可继续收紧成“直接进入训练问题”。
- `src/app/api/reports/generate/route.ts`
  - 当前无有效会话判断，收到 `history` 后就直接让模型生成报告。
  - 路由里的 JSON 模板示例给了固定分数样例 `85`；虽然这是 prompt 示例，不等于真写死，但当前没有拦截“空历史仍出分”的问题。
  - 默认兜底值仍较多，例如 `questionCount || 5`、`focus || "综合面试"`、`role || getInterviewModeLabel(...)`，会让无效会话看起来像有效报告。
- `src/app/report/page.tsx`
  - 当前只要拿到 `interviewHistory` 就会请求 `/api/reports/generate`。
  - 没有“本次未形成有效面试，不生成评分”的前端分支。

## Proposed Changes

### 1. 重构限制配置模型
- 文件：`src/lib/interview/config.ts`
- 变更：
  - 新增限制类别字段，例如 `limitType: "none" | "question" | "duration"`。
  - 保留 `questionLimit`、`durationLimitMinutes` 作为最终执行值，但通过归一化逻辑保证二者互斥。
  - 调整枚举：
    - `QUESTION_LIMIT_OPTIONS` 收敛为少量固定值。
    - `DURATION_LIMIT_OPTIONS` 收敛为少量固定值。
    - 增加时长自定义值的归一化函数，例如 `normalizeCustomDurationLimit()`。
  - 更新 `buildInterviewLimitStrategy()`：只输出当前生效的单一策略，不再同时展示“题量 / 时长”双策略。
- 原因：当前双字段并列使用导致 UI 和 prompt 都默认同时生效，不符合“二选一”。

### 2. 让专项训练彻底变成不限模式
- 文件：`src/app/practice/page.tsx`
- 变更：
  - 删除专项训练页中的题量 / 时长选择 UI。
  - 启动专项训练时固定写入：
    - `limitType: "none"`
    - `questionLimit: null`
    - `durationLimitMinutes: null`
  - 调整页面提示文案，明确“专项训练默认持续进行，直到你主动结束”。
- 原因：专项训练本质是反复练习，不应该被题量或时长限制打断。

### 3. 把发起面试的配置改成“数量 / 时长二选一”
- 文件：`src/app/setup/page.tsx`
- 变更：
  - 用两级交互替换现有双卡布局：
    - 第一级：选择 `数量` 或 `时长`
    - 第二级：
      - 若选择 `数量`，展示少量固定枚举。
      - 若选择 `时长`，展示少量固定枚举 + `自定义`
  - 选择 `自定义` 时，出现分钟输入框；输入值写入 `durationLimitMinutes`。
  - 在保存到 `parsedProfileData` 时，确保只写入一个生效限制，另一个强制置空。
  - 去掉 `setup` 页里的“实时模式视频开关”区块。
- 原因：当前视觉复杂且语义冲突，同时允许两种上限并存，会让用户策略理解变差。

### 4. 把视频开关移入实时面试房间
- 文件：`src/app/interview/page.tsx`
- 变更：
  - 在实时房间顶部或底部工具栏新增摄像头 icon 按钮，支持“开 / 关视频”即时切换。
  - 初始进入房间时默认 `videoEnabled = false`，只开麦，不自动开视频。
  - 点击摄像头按钮时：
    - 首次开启：请求视频轨道并挂到当前预览。
    - 关闭时：停止并释放视频轨道，但保留音频连接。
  - 实时区域布局继续复用现有双栏能力：有视频时显示本地画面，无视频时收起本地画面。
  - 同步更新房间标签文案，不再强调“进入前已开视频”，改为“当前视频状态可随时切换”。
- 原因：这符合你明确要求的“像腾讯会议一样，在房间里随时控制摄像头”。

### 5. 强约束专项训练不要反复确认
- 文件：`src/lib/interview/prompt.ts`
- 变更：
  - 在 `buildInterviewSystemPrompt()` 的专项训练规则中新增明确约束：
    - 不要再次向用户确认已识别的主题或训练目标。
    - 进入专项训练后直接从训练问题、追问、纠偏开始。
    - 只有在后端已标记“信息不足”的情况下，才允许阻断开始；一旦开始训练，禁止模型继续确认主题。
  - 调整 `buildInterviewOpening()` 的专项训练开场语，改成直接进入第一个训练问题，而不是再做确认式开场。
- 原因：你反馈“模型会反复和我确认”，这属于 prompt 行为收口，不只是 UI 问题。

### 6. 为空面试增加“无评分”分支
- 文件：`src/app/api/reports/generate/route.ts`
- 变更：
  - 在真正请求模型前新增“有效会话”校验。
  - 建议标准：
    - 候选人有效回答数小于 1，或
    - 总消息数不足以形成至少 1 轮有效问答
    - 则直接返回 `noEffectiveInterview: true`
  - 这类返回不再调用模型，不生成 `score`，不写入正式评分。
  - 同时清理接口中的默认兜底值，避免无效会话被包装成完整面试。
- 原因：现在“进去就结束也给 70 分”本质是服务端缺少有效性判断。

### 7. 更新报告页展示逻辑
- 文件：`src/app/report/page.tsx`
- 变更：
  - 兼容 `/api/reports/generate` 返回的“无有效面试”结果。
  - 新增空报告态：
    - 展示“本次未形成有效面试内容，不生成评分与正式报告”
    - 不展示分数卡片
    - 提供“重新开始面试”与“返回首页/发起面试”的按钮
  - 保留已有正常报告渲染逻辑。
- 原因：只有服务端拦截不够，前端也需要正确承接这种合法空态。

### 8. 收口与兼容发起面试确认页
- 文件：`src/app/profile/page.tsx`
- 变更：
  - 确认页中的“题量 / 时长”摘要，改成单一限制策略展示。
  - 若来自专项训练或 `limitType: "none"`，则显示“本次不设自动结束上限，由你主动结束”。
  - 若实时模式取消了预进入视频开关，这里不再展示“已开启视频 / 默认仅音频”的静态预判文案，而改成“进入房间后可随时开启或关闭视频”。
- 原因：否则 `setup`、`profile`、`interview` 三处口径会重新不一致。

## Assumptions & Decisions
- 已确认决策：
  - 专项训练不提供题量或时长上限，固定不限。
  - 发起面试里题量上限和时长上限二选一。
  - 仅时长支持自定义输入；数量只保留少量固定枚举。
  - 空面试不生成评分，不生成正式报告。
  - 专项训练开始后禁止模型再次确认主题。
- 当前计划中的默认实现细节：
  - 实时房间默认视频关闭、音频优先。
  - “有效面试”最低门槛按“至少 1 次候选人有效回答 + 至少 1 次 AI 问题”执行。
  - 自定义时长仅接受正整数分钟，并通过前端输入约束与服务端归一化共同保证合法性。

## Verification Steps
- 配置页验证：
  - `practice` 页面不再出现题量 / 时长选择。
  - `setup` 页面只能先选“数量”或“时长”之一。
  - 选“时长”后可切换少量枚举，并可输入自定义分钟数。
  - 选“数量”后不再能同时保留时长限制。
- 房间验证：
  - 进入实时面试默认不开视频。
  - 房间内摄像头按钮可随时开启 / 关闭视频。
  - 开关视频不影响音频面试持续进行。
- 训练行为验证：
  - 专项训练一旦进入，模型直接开始训练问题，不再反复确认主题。
  - 专项训练长时间进行也不会因题量或时长自动结束。
- 报告验证：
  - 刚进入面试就结束，不应出现 70 分或任何评分。
  - 正常完成至少一轮有效问答后，仍可生成正式报告。
  - 报告页能正确区分“无评分空态”和“正常评分报告”。
- 回归验证：
  - `text` / `realtime` / `targeted` 三种模式都能正常进入。
  - 已有 session / report 查询逻辑不因空态分支而崩溃。
