# 用户画像页与面试页样式收口计划

## Summary
- 目标：修正“发起面试 -> 解析画像 -> 进入面试”这一条链路里最影响体验的两个页面：
  - `用户画像页` 信息结构错乱、阻断时机不对、展示内容过多
  - `面试页` 文本样式、头像称呼、流式反馈感与底部按钮尺寸不协调
- 成功标准：
  - 简历摘要不再放在 `profile` 页，而是在 `setup` 页解析成功后回填到上传区下方的大文本框中。
  - “关键信息不足”提示前移到解析阶段，不再等到 `profile` 页才阻断。
  - “必要信息”只围绕项目经历、技术基础和能支撑后续提问的真实内容，不再把 `JD 正文 / GPA / 排名 / 离职原因 / 求职动机` 这类项作为硬性阻断。
  - `profile` 页主体只收敛为三块：`岗位匹配提醒和优化建议`、`简历可优化点`、`本场策略`，同时保留顶部岗位编辑、底部“手动补充训练侧重点”以及“返回修改 / 开始”按钮。
  - `interview` 页中“AI”头像改为“面试官”，文本气泡与 Markdown 样式收口，底部快捷按钮尺寸明显缩小且更协调。
  - 样式调整遵循 `brand-guidelines`，做克制优化，不大改整体布局和视觉语言。

## Current State Analysis
- `src/app/setup/page.tsx`
  - 当前解析成功后直接把后端返回结果写入 `sessionStorage.parsedProfileData`，然后跳转到 `/profile`。
  - 上传区下方的大文本框当前只承载原始简历文本 `resumeText`，并未在解析后展示 `resumeSummaryMarkdown`。
  - 错误提示逻辑全部用 `submitError` 承接，但解析失败时没有把“必要信息不足”的口径前置到这个阶段。
- `src/app/api/parse/route.ts`
  - 当前 `buildMissingDataHints()` 会把很多并不适合作为硬阻断的缺失项塞进 `missingDataHints`。
  - 当前返回结构里有 `resumeSummaryMarkdown`、`persona`、`jdGapWarning`、`projects`，其中 `persona` 与 `projects` 被 `profile` 页大面积展开渲染。
  - 当前 prompt 明确要求模型输出 `persona` 和 `projects`，但没有单独的“简历可优化点”字段。
- `src/app/profile/page.tsx`
  - 当前页面同时展示：`简历摘要`、`用户画像`、`岗位匹配提醒`、`本场策略`、`高优可追问项目`、`手动补充训练侧重点`。
  - 页面顶部会显示“关键信息不足，暂不能直接进入面试”，并通过 `collectBlockingHints()` 阻断进入面试。
  - 当前阻断规则至少包含：
    - 缺少 `role`
    - 缺少 `resumeSummaryMarkdown`
    - 缺少 `persona`
  - 这与用户要求的“信息不足应在解析时提示，而且必要项要收缩到项目经历 / 技术基础”不一致。
- `src/app/interview/page.tsx`
  - 文本面试区当前左侧头像仍显示 `"AI"`，与用户期待的“面试官”不一致。
  - 文本气泡宽度、字号和留白偏大，底部 `跳过本题 / 我不知道 / 发送回答` 按钮也明显偏大。
  - 虽然后端 `chat` 是流式接口，但文本区当前表现更像整段落地，缺乏更顺滑的流式阅读感。
  - 实时房间主视觉中心头像也仍显示 `"AI"`。
- `src/app/api/chat/route.ts`
  - 已经是流式输出接口，但前端展示层没有做更精细的流式呈现优化计划，需要在 UI 计划里单独收口“显示感受”，而不是重写接口。
- `src/lib/interview/config.ts`
  - `InterviewProfileState` 目前有 `resumeSummaryMarkdown`、`persona`、`jdGapWarning`、`projects`、`missingDataHints` 等字段。
  - 若 `profile` 页改为只显示三块信息，需要同步考虑前后端存储字段是否要精简或仅调整展示。

## Proposed Changes

### 1. 前移“必要信息不足”到解析阶段
- 文件：`src/app/api/parse/route.ts`
- 变更：
  - 重写 `buildMissingDataHints()` 的判定标准，只保留与后续提问直接相关的必要信息：
    - 目标岗位可识别
    - 至少有可用的技术基础线索
    - 至少有可追问的项目/经历线索
  - 去掉将 `JD 正文`、`GPA/排名`、`离职原因`、`求职动机` 等作为阻断级缺失项的做法。
  - 当必要信息不足时，接口直接返回更明确的解析阶段错误或提示，让 `setup` 页当场提示，而不是等到 `profile` 页才拦。
- 原因：用户明确要求阻断发生在“上传简历 -> 点击解析生成画像”这个动作上，而不是确认页。

### 2. 调整解析结果结构，补足“简历可优化点”
- 文件：`src/app/api/parse/route.ts`
- 变更：
  - 在解析结果里增加一个更贴合 `profile` 页展示的新字段，例如 `resumeImprovements` 或等价结构，作为“简历可优化点”。
  - 保留 `jdGapWarning` 用于“岗位匹配提醒和优化建议”。
  - 保留 `resumeSummaryMarkdown`，但不再在 `profile` 页展示，而是用于回填 `setup` 页文本框和后续 prompt 上下文。
  - `persona` 与 `projects` 若继续保留，也转为内部上下文优先，不再在确认页大面积展示。
- 原因：用户已经明确了确认页理想展示结构，当前 `persona + 项目 + 摘要` 的大拼盘会继续制造混乱。

### 3. 让 `setup` 页承接简历摘要回填
- 文件：`src/app/setup/page.tsx`
- 变更：
  - 解析成功后，把 `resumeSummaryMarkdown` 直接回填到上传区下面的大文本框。
  - 若用户上传了文件且原始 `resumeText` 为空，解析完成后该文本框展示 AI 摘要，允许用户继续修改。
  - 若解析阶段发现必要信息不足，则在 `setup` 页直接显示解析提示，不跳转 `profile`。
  - 保持现有上传区和整体布局不大改，只做信息承接调整。
- 原因：这是用户明确指定的摘要展示位置。

### 4. 收敛 `profile` 页信息结构
- 文件：`src/app/profile/page.tsx`
- 变更：
  - 去掉 `简历摘要` 主卡片。
  - 去掉当前大面积的 `用户画像`、`高优可追问项目` 展示。
  - 页面主体收敛为三块：
    - `岗位匹配提醒和优化建议`
    - `简历可优化点`
    - `本场策略`
  - 保留顶部岗位编辑输入框。
  - 保留底部“手动补充训练侧重点”。
  - 保留“返回修改”和“开始面试/开始训练”按钮。
  - 移除当前确认页上的阻断大提示区；是否允许进入房间改为更多依赖解析阶段结果。
- 原因：按用户要求简化信息结构，同时不误删其认可的岗位编辑与重点输入能力。

### 5. 收口 `profile` 页的必要进入条件
- 文件：`src/app/profile/page.tsx`
- 变更：
  - 重写 `collectBlockingHints()` 或取消它的“广泛阻断”职责。
  - `profile` 页只保留极少数真正无法开始的情况，例如：
    - 完全没有解析结果
    - 完全没有岗位
  - 项目经历 / 技术基础类不足尽量在 `setup -> parse` 阶段解决，不在这里继续堆提示。
- 原因：避免解析和确认两个阶段同时做阻断，导致体验重复且混乱。

### 6. 收口面试页文本样式与称呼
- 文件：`src/app/interview/page.tsx`
- 变更：
  - 文本模式中将左侧 `"AI"` 头像改为 `"面试官"`，并优化成更协调的圆形/标签样式。
  - 实时房间中心主视觉中的 `"AI"` 也改成 `"面试官"`。
  - AI 文本气泡的字号、行高、最大宽度、内边距和时间标签做克制收口，减少“乱”和“臃肿”。
  - 对 Markdown 内容增加更统一的段落间距、标题/列表样式约束，避免 AI 输出看起来散乱。
- 原因：用户明确指出当前称呼和文本样式都乱。

### 7. 收口底部动作区按钮尺寸
- 文件：`src/app/interview/page.tsx`
- 变更：
  - 缩小 `跳过本题`、`我不知道`、`发送回答` 的高度、横向 padding 和字号。
  - 保持按钮语义不变，但视觉上做分层：
    - 次级快捷动作更轻
    - 主按钮 `发送回答` 更聚焦，但不夸张
  - 不改变现有操作布局，只做克制尺寸与间距收口。
- 原因：用户明确指出这几个按钮过大、审美不协调。

### 8. 优化“流式输出感”，但不重写后端接口
- 文件：`src/app/interview/page.tsx`, `src/app/api/chat/route.ts`
- 变更：
  - 先复核当前文本模式消息落地方式，明确是“已流式但显示感弱”还是“前端聚合后再渲染”。
  - 如果前端当前是整段插入，则计划改为更明显的逐段/逐 token 更新。
  - 如果后端已稳定流式、前端也逐步接收，则只做显示层优化，例如：
    - typing 状态更自然
    - 气泡底部时间与 loading 状态更统一
    - 避免一次性大段跳变的视觉感受
- 原因：用户明确指出“AI 输出的文本也不是流式输出的”，这需要先基于真实代码判断是接口问题还是显示问题。

### 9. 用 `brand-guidelines` 做克制视觉收口
- 文件：`src/app/profile/page.tsx`, `src/app/interview/page.tsx`
- 变更：
  - 统一使用当前系统已有的橙 / 蓝 / 绿点缀与浅灰底，不引入新的强烈色板。
  - `profile` 页用更清晰的卡片层级，但保留现有整体宽度、留白和布局骨架。
  - `interview` 页重点优化头像标签、文本气泡、底部操作区，不改整体页面结构。
- 原因：用户强调“样式！样式！样式！”但同时明确不要大改瞎改。

## Assumptions & Decisions
- 已确认决策：
  - 简历摘要直接覆盖回填 `setup` 页上传区下面的大文本框。
  - `profile` 页主体只展示三块核心内容。
  - `profile` 页仍保留：顶部岗位编辑、底部训练重点输入、返回与开始按钮。
  - 解析必要信息只围绕“项目经历、技术基础、真实岗位上下文”，不再扩大到 GPA、离职原因、JD 正文等。
  - 面试页的样式优化只做克制收口，遵循 `brand-guidelines`，不做大改版。
- 当前默认实现选择：
  - `resumeSummaryMarkdown` 继续保留在解析结果和面试上下文里，但从确认页展示层移除。
  - “简历可优化点”优先作为新的解析字段返回，而不是继续复用 `persona.risks` 直接硬套。
  - “流式输出”优先判断为显示层问题；只有在读代码后确认前端不是逐步渲染时，才纳入交互层实现调整。

## Verification Steps
- 解析链路验证：
  - 上传简历并点击解析，若必要信息不足，应在 `setup` 页直接看到提示，不再跳到 `profile` 后才提示。
  - 缺少 `JD`、`GPA/排名`、`离职原因/动机` 时，不应再被当作硬阻断。
- `setup` 页验证：
  - 解析成功后，上传区下方的大文本框被 AI 简历摘要直接回填。
  - 用户仍可在该文本框中继续修改内容。
- `profile` 页验证：
  - 页面主体只剩三块核心信息。
  - 不再展示当前那种大段“简历摘要 / 用户画像 / 高优项目”拼盘。
  - 顶部岗位编辑、底部训练重点输入、“返回修改 / 开始”仍然保留。
- `interview` 页验证：
  - 文本模式和实时模式中的 `"AI"` 均替换为 `"面试官"`。
  - AI 气泡 Markdown 样式更统一，阅读层级更稳定。
  - `跳过本题 / 我不知道 / 发送回答` 按钮明显变小且更协调。
  - 若文本模式当前可做更明显流式呈现，则确认更新后回复是逐步出现的；若只能改善显示感，则确认跳变感明显降低。
