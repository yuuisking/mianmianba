# 面试结束策略、算法题系统与实时体验增强计划

## 摘要
- 目标是把当前文字面试从“规则驱动的问答 demo”升级成更接近真实技术面试的系统：面试官先标准化开场并要求自我介绍，再基于更强的多 Agent 规划与追问链推进问答；在准备结束前，由结束裁决 Agent 主动判断是否收口，并在技术轮统一插入算法题；算法题系统升级为 LeetCode Top 200 题库 + Monaco 高保真编辑器 + 真实代码执行/判题链；思考态增加呼吸气泡动画，增强“实时思考中”的反馈。
- 本计划覆盖四条主线：
  - 主动结束策略从关键词阈值升级为 Agent 裁决 + 结束前提示 + 技术轮先转算法题。
  - 算法题从两道模板题升级为 Top 200 题库、LeetCode 风格布局和真实运行/提交能力。
  - 面试页“思考中”状态升级为带呼吸动画的气泡组件，而不是纯文本。
  - Planner / Evidence / Composer / Guard 链做增量增强，并新增结束裁决链，让提问与追问更贴近真实模拟面试。

## 当前状态分析

### 1. 结束策略现状
- 当前决策完全由规则函数 [turnPolicy.ts](file:///Users/yangyu/Desktop/resumer/src/lib/interview/turnPolicy.ts) 控制，只支持 `continue / end_interview / offer_coding` 三种动作。
- 主动结束条件是“最近连续 3 次用户回答都属于低信息量或命中固定负向关键词”，没有“答非所问”判断，也没有“面试官觉得信息已足够、主动收口”的正向结束逻辑。
- `/api/chat` 在命中 `end_interview` 时直接输出结束话术，没有“即将结束”的过渡提示，也没有把“结束前统一先给算法题”作为标准流程，见 [route.ts](file:///Users/yangyu/Desktop/resumer/src/app/api/chat/route.ts)。

### 2. 算法题系统现状
- 当前算法题只有两套硬编码模板，逻辑集中在 [codingSessionService.ts](file:///Users/yangyu/Desktop/resumer/src/lib/interview-v2/codingSessionService.ts)：
  - 默认岗位：`两数之和`
  - 测试岗：`区间合并`
- 当前编辑器组件 [CodingPanel.tsx](file:///Users/yangyu/Desktop/resumer/src/components/interview/CodingPanel.tsx) 本质是 `textarea + 运行/提交按钮 + 结果文本`，并非 LeetCode 风格编辑器。
- 支持语言当前仅 `javascript / typescript / python`，不符合目标的 `Java / C++ / JavaScript / Python / Go`。
- `run/submit` 后端接口 [run/route.ts](file:///Users/yangyu/Desktop/resumer/src/app/api/v2/coding-sessions/[id]/run/route.ts) 与 [submit/route.ts](file:///Users/yangyu/Desktop/resumer/src/app/api/v2/coding-sessions/[id]/submit/route.ts) 最终仍走 `executeCodingSessionAction()` 的静态规则匹配，不是真实编译/执行/判题。
- 仓库内未发现现成沙箱或代码执行基建，当前“真实可用，不可 mock”的要求意味着必须补齐执行链，而不是只换前端皮肤。

### 3. 思考态现状
- 面试页 [page.tsx](file:///Users/yangyu/Desktop/resumer/src/app/interview/page.tsx) 已有 `isThinking`、`thinkingStatus` 状态与多种文案：
  - `正在同步你的回答`
  - `面试官思考中`
  - `进一步检索资料中`
  - `正在生成下一轮追问`
  - `正在恢复上次对话`
- 但当前是文案级反馈，没有独立的“思考气泡 / 呼吸动画 / 实时流动感”组件，表现力较弱。

### 4. 出题链与面经利用现状
- `/api/chat` 已接入 Planner、Evidence、Composer、Guard 四段链路，且确实会读取 `experienceInsights` 和检索结果 `searchResults`，见 [route.ts](file:///Users/yangyu/Desktop/resumer/src/app/api/chat/route.ts#L522-L999)。
- 但现有 Planner / Composer prompt 仍偏“结构化 JSON 规划”，缺少“像真人面试官一样根据回答判断是否继续深挖、换题、收口”的现场感。
- 当前结束动作不在 Agent 链内，仍由固定规则函数兜底，因此“答非所问但很长”“回答质量高但已经足够结束”这类真实面试场景无法自然覆盖。

### 5. 开场语现状
- 正式模拟开场当前由 [buildInterviewOpening()](file:///Users/yangyu/Desktop/resumer/src/lib/interview/prompt.ts#L1520-L1585) 返回：
  - `你好，欢迎进入【公司】一面面试，我是你的面试官。`
  - 然后直接 `第一个问题：...`
- 目前没有统一的“同学您好，欢迎参加【公司】的【轮次】面试，我是你本次的面试官，请问你准备好了嘛，准备好了请先做一个自我介绍。”这条标准开场，也没有“自我介绍后再开始基于介绍和简历追问”的状态切换。
- 已确认范围：该欢迎语流程只应用到“正式模拟”，不改专项训练模式。

## 方案决策与约束
- 结束前算法题范围：`技术轮都要`
  - 只要是技术轮，面试官准备结束时统一先转算法题；算法题完成后再正式结束本轮。
- 题库来源：`完全按站点文案`
  - 计划按高保真题面结构设计题库与展示；执行阶段需要同步评估文本来源、数据导入方式和版权风险。
- 编辑器方案：`Monaco 高保真`
  - 第一版即引入 Monaco，按 LeetCode 布局风格做真实代码编辑区。
- Monaco 依赖决策：`@monaco-editor/react + monaco-editor`
  - 当前仓库没有现成编辑器依赖，执行阶段直接按 React 场景最稳的 Monaco 组合落地，不再二选一。
- Agent 策略：`增量增强`
  - 保留 Planner / Evidence / Composer / Guard 基础骨架，新增结束裁决 Agent、答非所问判断链，并整体重写 prompt。
- 欢迎语范围：`仅正式模拟`
  - 全流程模拟与标准正式技术面统一先欢迎 + 自我介绍；专项训练仍保留直切主题。
- 真实判题部署决策：`同机独立 Judge Runner 服务`
  - 当前 ECS 部署形态是 `Next.js + systemd`，没有现成 Docker/判题服务编排；执行阶段按“同一台 ECS 机器新增独立 `judge-runner` 服务 + 主应用通过内网 HTTP 调用”实现，避免把多语言编译执行直接塞进 Web 进程。
- 题库存放决策：`版本化静态数据 + 数据库导入`
  - Top 200 题库以版本化 JSON/SQL seed 进入仓库，再通过导入脚本落库，避免题库内容散落在代码逻辑中。

### A. 主动结束策略重构

#### A1. 新增结束裁决 Agent 与结束前过渡状态
- 文件：
  - `src/app/api/chat/route.ts`
  - `src/lib/interview/turnPolicy.ts`
  - `src/app/interview/page.tsx`
  - `src/lib/interview/config.ts`
- 改动内容：
  - 将当前纯规则 `evaluateInterviewTurnPolicy()` 改为“规则信号 + Agent 裁决”双层结构：
    - 规则层保留快速信号统计，但仅作为输入，不再直接决定最终结束。
    - 新增 `EndJudge Agent`，输入最近 3-5 轮问答、当前轮次类型、面经证据、是否已进入算法题、候选人是否答非所问、回答信息量是否已足够完成评估。
  - 新策略动作扩展为至少：
    - `continue`
    - `offer_coding_before_end`
    - `end_after_coding`
    - `end_interview_directly`（仅非技术轮或已完成算法题时可用）
  - 新增“结束前提示消息”：
    - 不是直接终止，而是由面试官先给收口提示，例如“我这边核心点基本确认了，接下来我们在结束前补一道算法题。”
  - 支持“答非所问”累积：
    - 不是只看关键词，而是让 Agent 判断“候选人是否持续偏离问题主题、没有回答问点”。
  - 支持“正向结束”：
    - 如果候选人已经给出足够证据，面试官也可以主动收口，不再机械继续追问。
- 为什么这样改：
  - 用户要求结束逻辑更像真实面试官，不应只因“差”才结束，也应因“信息足够”而主动结束。
  - 固定关键词无法判断“答非所问但篇幅很长”的场景。
- 具体实现方式：
  - 保留 [turnPolicy.ts](file:///Users/yangyu/Desktop/resumer/src/lib/interview/turnPolicy.ts) 的负向/正向统计函数，重命名为“signal extractor”层。
  - 在 `/api/chat` 新增 `judgeInterviewTurnClosure()`，由 DeepSeek 返回结构化 JSON：
    - `shouldEnd`
    - `endType`
    - `needsCodingBeforeEnd`
    - `offTopicCount`
    - `confidence`
    - `reason`
    - `candidateFacingTransition`
  - 响应头和前端状态扩展：
    - 将当前 `x-interview-action` 扩展为可表达“先转算法再结束”。
    - 前端增加“即将收口”的消息态和 UI 提示，不再让用户误以为系统 bug。

#### C2. Agent Prompt 设计标准必须结构化明确
- 文件：
  - `src/app/api/chat/route.ts`
  - `src/lib/interview/prompt.ts`
  - 如有必要新增 `src/lib/interview/agentPrompts.ts` 或同类集中管理文件
- 改动内容：
  - 所有核心 Agent 的 prompt 都必须显式写清楚以下 5 类信息，不能只给一句模糊的 system prompt：
    - `你是什么角色`
    - `你有哪些能力`
    - `你主要用来做什么`
    - `你的输入是什么、输出必须是什么`
    - `你不能做什么 / 不能泄漏什么`
  - 适用 Agent 至少包括：
    - `Planner Agent`
    - `Evidence Agent`
    - `Composer Agent`
    - `Guard Agent`
    - `EndJudge Agent`
    - `OffTopicJudge / FollowUpJudge`
    - 必要时补 `Opening Agent` 或首轮追问专用 Agent
  - 每个 Agent prompt 都要具备“角色身份 + 能力清单 + 核心职责 + 约束边界 + 输出契约”五段式结构。
- 为什么这样改：
  - 用户明确要求不要怕麻烦，要在 prompt 上下功夫，避免 Agent “看起来有链路，实际上很蠢”。
  - 当前部分 prompt 更像任务说明，而不是完整角色定义，容易导致能力边界模糊、输出风格漂移。
- 具体实现方式：
  - 为每个 Agent 统一采用以下设计骨架：
    - `角色定义`：例如“你是经验很深的技术面试规划官 / 结束裁决官 / 现场追问助手”
    - `能力说明`：明确它能做哪些判断、能基于什么证据做判断
    - `主要职责`：这一轮它唯一应该完成的目标
    - `输入上下文`：它能依赖哪些字段，哪些字段必须优先
    - `输出契约`：JSON 字段、字段含义、字段约束
    - `禁止事项`：不能泄漏内部策略、不能代替别的 Agent 做决策、不能输出给候选人的最终台词等
  - 在计划执行时优先把 prompt 收敛到统一文件或统一 builder，避免 prompt 分散在多个函数里难以持续优化。
  - 将 prompt 验收加入测试与走查：
    - 是否能从 prompt 本身读出角色、能力、职责、边界
    - 是否会因 prompt 过短导致泛化输出
    - 是否能稳定产出贴近真实面试的结构化结果

    - `CodingProblemPane`
    - `CodingEditorPane`
    - `CodingResultPane`
  - 保持 room 级恢复逻辑，确保刷新后恢复题面、语言、代码、最近一次结果。

#### D3. 真实运行 / 提交 / 判题链
- 文件：
  - `src/lib/interview-v2/codingSessionService.ts`
  - `src/app/api/v2/coding-sessions/[id]/run/route.ts`
  - `src/app/api/v2/coding-sessions/[id]/submit/route.ts`
  - `src/app/api/v2/coding-sessions/route.ts`
  - 新增 `src/lib/coding/*`
  - 新增 `scripts/judge-runner/*`
  - 新增 `scripts/ops/install-judge-runtime.sh`
  - 新增 `deploy/judge-runner.service` 或同类 systemd 服务定义
- 改动内容：
  - 去掉当前 `static-pattern-review` 作为主评估方式，改成真实编译/运行/判题。
  - 支持每题多组隐藏/公开测试用例。
  - `RUN` 返回编译结果、运行输出、报错、样例通过情况。
  - `SUBMIT` 返回隐藏用例通过数、总用例数、耗时、内存、最终状态。
- 为什么这样改：
  - 用户明确要求“真实可用，不可 mock”；仅 Monaco 无法满足。
- 具体实现方式：
  - 执行层单独封装，明确采用“同机独立 Judge Runner 服务”方案：
    - `resumer` 主应用只负责鉴权、题目分配、会话状态和结果回写
    - `judge-runner` 独立服务只负责代码编译、执行、比对测试用例
    - 二者通过本机内网 HTTP 调用，避免把长时编译执行阻塞在 Web 请求进程中
  - `judge-runner` 运行约束明确为：
    - 使用独立临时目录
    - 单次执行超时
    - 输出大小限制
    - 进程级并发限制
    - 仅开放 Java / C++ / Node.js / Python / Go 五类语言入口
  - 部署前置条件明确为：
    - 远端 ECS 机器安装 `openjdk`、`g++`、`python3`、`node`、`go`
    - 增加 judge 服务的 systemd 配置与启动脚本
  - 运行结果结构统一为：
    - `compileStatus`
    - `runStatus`
    - `passedCount`
    - `totalCount`
    - `stdout`
    - `stderr`
    - `timeMs`
    - `memoryKb`
    - `failedCases`
  - 判题结果写回 `CodingSession` 与 `CodingSubmission`。

### F. 边界场景与失败策略

#### F1. 结束链与算法题边界
- 文件：
  - `src/app/interview/page.tsx`
  - `src/app/api/chat/route.ts`
  - `src/lib/interview-v2/codingSessionService.ts`
- 需要明确覆盖的场景：
  - 用户在“结束前算法题”弹窗中拒绝进入
  - 用户进入算法题后长时间不作答
  - 用户运行多次但从未提交
  - 用户刷新页面后恢复到算法题中
  - 算法题执行服务不可用时，面试如何收口
- 决策：
  - 拒绝算法题或超时未完成，不阻塞本轮结束；系统记录 `codingSkipped` 或 `codingAborted`，然后由面试官正常结束本轮。
  - Judge Runner 不可用时，不回退为 mock 判题；页面明确提示“当前算法题服务不可用，本轮算法环节记为未完成”，随后结束面试。

#### F2. 自我介绍与追问边界
- 文件：
  - `src/lib/interview/prompt.ts`
  - `src/app/api/chat/route.ts`
- 需要明确覆盖的场景：
  - 用户自我介绍极短
  - 用户不按要求自我介绍，直接问待遇/流程
  - 用户自我介绍与简历矛盾
- 决策：
  - 自我介绍过短时，不直接进入技术深挖，先由面试官补一个最小必要澄清问题。
  - 自我介绍与简历矛盾时，首轮优先走真实性核验，而不是直接进入项目细节。

## 验证步骤

- 正式模拟开场必须固定为：

- 候选人回答质量很好且核心点已覆盖时，面试官也能主动准备结束，而不是无限追问。

## 风险与假设
- 用户已确认题库按“完全按站点文案”高保真落地；执行阶段需要同步处理数据来源与版权/合规风险。
- 当前仓库没有现成代码沙箱；若执行环境不能直接承载多语言编译运行，需在实施时补独立执行服务或受控沙箱。
- Monaco 与真实判题链会明显扩大交付范围；计划按完整方案设计，不做“只换皮肤”的降级。
