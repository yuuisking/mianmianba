# 实时语音豆包对齐技术方案

## Summary
目标是在域名与 HTTPS 就绪前，先把实时语音专项的核心设计收口清楚，确保后续实现不会偏离。最终方向不是“参考豆包”，而是：

- 语音交互层一比一还原豆包语音通话体验
- 面试官大脑继续固定为 `DeepSeek v4`
- 语音供应商继续统一使用火山引擎
- 房间 UI 主结构不大改，重点改实时链路、状态机、协议和语音体验

一句话技术路线：

`DeepSeek v4` 负责面试官思考与文本生成，火山引擎负责语音识别与语音播报，浏览器和服务端之间通过实时状态机与流式会话协议，把整套交互编排成豆包式体感。

## 当前实现盘点

### 1. 浏览器采集链路
- 当前在 [page.tsx](file:///Users/yangyu/Desktop/resumer/src/app/interview/page.tsx#L1274-L1426) 中通过 `getUserMedia` 获取麦克风和可选摄像头。

- 当前分段条件是：

  - 或单次发言超过 `10000ms`

  - 没有 partial transcript
  - 用户说话中几乎没有即时反馈
  - 识别结果出来后才进入下一轮 `handleSendMessage`

### 3. DeepSeek v4 文本生成链路

- 这意味着：面试官大脑和文本侧已经不是瓶颈，真正的瓶颈在“文本流和语音流没有被同一个实时协议打通”。

### 4. TTS 与播放链路
- 当前前端会在 [page.tsx](file:///Users/yangyu/Desktop/resumer/src/app/interview/page.tsx#L1031-L1146) 中请求 `/api/speech/tts`。
- 服务端入口在 [tts route](file:///Users/yangyu/Desktop/resumer/src/app/api/speech/tts/route.ts#L1-L38)。
- 当前底层 TTS 来自 [volc.ts](file:///Users/yangyu/Desktop/resumer/src/lib/speech/volc.ts#L161-L242)。

- 现状问题：
  - 已经比“整段朗读”好，但本质还是“文本切句 -> 单句 HTTP TTS -> 播放队列”
  - 还不是豆包那种连续接话感
  - 首字等待仍受句子切分和 TTS 往返时延影响
  - 打断虽然已有雏形，但不是完整会话级中断

### 5. 当前打断机制
- 打断逻辑在 [page.tsx](file:///Users/yangyu/Desktop/resumer/src/app/interview/page.tsx#L1148-L1171)。
- 当前做法：
  - `abortControllerRef` 中止 `/api/chat`
  - `ttsAbortControllerRef` 中止 TTS 请求
  - 停止当前音频 source
  - 清空队列
- 当前优势：
  - 已有最关键的“用户一说话就可打断 AI”雏形
- 当前不足：
  - 没有正式的“中断状态机”
  - 没有定义“被打断后保留哪些文本上下文”
  - 字幕、状态、播放、模型流中断仍靠分散的前端状态拼接

## 与豆包一比一还原的差距

### 豆包体感的核心特征
当前要对齐的不是一个抽象概念，而是以下可观察特征：

1. 首字快
2. 接话自然
3. 用户插话后停止非常快
4. 停止后不会有明显拖尾
5. 字幕会跟着说话同步推进
6. 状态切换不“机械”，而像真的在对话
7. 长回答会按自然意群往外吐，而不是朗读稿件

### 当前实现和目标的差距
- 当前用户输入是 `final-only`，豆包更接近 `partial + final`
- 当前 AI 语音是“切句后逐句 TTS”，豆包更接近“增量接话”
- 当前打断是“前端强停”，豆包更接近“会话级中断”
- 当前字幕与状态依赖多个本地状态拼接，豆包更接近“统一实时协议驱动”
- 当前房间里“AI 大脑”和“语音壳层”没有显式解耦约束，后续实现很容易跑偏

## 硬约束

### 1. 交互目标
- 必须以“豆包语音通话体验一比一还原”为唯一交互目标
- 不是“类似”
- 不是“尽量接近”
- 而是每个关键体感都要拿豆包做对齐基准

### 2. 大脑模型
- 面试官内容生成固定使用 `DeepSeek v4`
- 具体包括：
  - 出题
  - 追问
  - 纠偏
  - 承接上下文
  - 面试判断
- 不允许因为要还原豆包交互而切换到底层豆包模型

### 3. 语音供应商
- 继续统一使用火山引擎
- 不混用阿里语音
- 不使用浏览器原生 TTS/ASR 作为正式主链路

## 目标技术架构

### 架构分层
分成三层：

1. 大脑层：`DeepSeek v4`
2. 语音层：火山引擎 ASR / TTS / 可用实时能力
3. 编排层：房间状态机 + 会话协议 + 字幕策略 + 打断策略

### 数据流
目标态数据流如下：

1. 浏览器持续采集音频
2. 浏览器将音频增量发送到服务端实时会话
3. 服务端驱动火山识别，持续产出 `partial_user_transcript`
4. 用户停顿或达到提交条件后，形成 `final_user_transcript`
5. 服务端将 final 用户文本送入 `DeepSeek v4`
6. `DeepSeek v4` 以增量 token 返回
7. 服务端按意群切片成 `assistant_text_segment`
8. 每个 segment 进入火山 TTS
9. 客户端边收边播，边播边更新字幕
10. 如果用户重新开口，立即触发 barge-in，中断当前 AI 会话

### 解耦原则
- `DeepSeek v4` 只管“说什么”
- 火山只管“怎么听”和“怎么说”
- 状态机只管“什么时候听、什么时候打断、什么时候播、字幕怎么推进”

## 豆包对齐基准表

### 1. 首字节奏
- 对齐目标：用户说完后，AI 尽快进入“接话中”体感
- 当前阻塞点：
  - 要等 final ASR
  - 要等首句切分
  - 要等 TTS 返回
- 设计策略：
  - 用户 final 文本一形成就立即发给 `DeepSeek v4`
  - `DeepSeek v4` 增量 token 不等整段
  - 达到最小可播报意群后立即进入首段 TTS

### 2. 接话感
- 对齐目标：像真人接话，不像播报器
- 设计策略：
  - 使用短意群播报，而不是大段完整句
  - 第一段尽量短
  - 第一段优先是“回应性起句 + 核心问题切入”

### 3. 打断体感
- 对齐目标：用户一插话，AI 立即停
- 设计策略：
  - 前端检测到有效人声即触发 `barge_in_requested`
  - 服务端立刻标记当前 assistant turn 失效
  - 客户端立刻停止当前 source 和后续队列
  - 字幕立刻冻结，不再继续增长

### 4. 字幕推进
- 对齐目标：字幕跟着当前语音自然推进
- 设计策略：
  - 用户侧：`partial -> final`
  - AI 侧：`segment draft -> speaking -> spoken`
  - 被打断后停止当前 segment 的后续增长

### 5. 状态反馈
- 对齐目标：状态不是“系统提示”，而是对话现场感
- 设计策略：
  - 只保留最必要状态
  - 状态切换与真实链路事件绑定
  - 避免同时显示多个矛盾状态

## 实时状态机设计

### 房间级状态
- `idle`：未连接
- `connecting_media`：请求设备中
- `ready`：设备就绪，等待发言
- `user_speaking`：检测到用户说话
- `user_processing`：正在形成 final 用户文本
- `assistant_thinking`：`DeepSeek v4` 正在生成
- `assistant_speaking`：AI 正在播报
- `assistant_interrupted`：AI 被用户打断
- `error`：链路异常

### 用户发言子状态
- `vad_idle`
- `vad_hot`
- `partial_user_transcript`
- `final_user_transcript`

### AI 发言子状态
- `llm_stream_open`
- `assistant_segment_buffering`
- `tts_requesting`
- `tts_playing`
- `tts_draining`
- `assistant_cancelled`

## `DeepSeek v4` 与火山解耦协议

### 文本侧
- `DeepSeek v4` 返回的是文本增量
- 编排层负责把文本增量切成适合播报的意群段
- 不要求 `DeepSeek v4` 直接理解音频协议

### 语音侧
- 火山负责：
  - 用户语音识别
  - AI 文本播报
- 火山不负责：
  - 面试问题设计
  - 面试轮次策略
  - 上下文承接判断

### 编排层
- 编排层是必须新增的一层
- 职责包括：
  - turn 管理
  - segment 管理
  - barge-in 取消
  - 字幕冻结与恢复
  - 播放状态同步
  - partial/final 聚合
