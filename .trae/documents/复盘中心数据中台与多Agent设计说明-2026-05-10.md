# 复盘中心数据中台与多 Agent 设计说明

## 1. 这份文档解决什么问题

这份文档不是在讨论“复盘页怎么更好看”，而是在定义：**复盘中心必须从“结果展示页”升级成“用户成长数据中心”**。

你的目标不是再做一个报告列表，而是做一个真正能回答下面这些问题的系统：

1. 我最近到底变好了还是变差了？
2. 我变差的核心原因是什么？
3. 这个结论是基于哪些真实证据得出的？
4. 这些问题更像知识问题、表达问题，还是策略问题？
5. 我下一步到底该练什么，为什么先练这个？
6. 我照着建议练完以后，问题有没有真的被压下去？

如果复盘中心回答不了这 6 个问题，那它就不是数据中心，只是一个包装过的总结页。

## 2. 当前真实现状判断

基于对公网 `http://47.95.233.109/review` 的真实查看，以及当前实现文件：

- `src/app/review/page.tsx`
- `src/app/api/v2/review/dashboard/route.ts`
- `src/lib/interview-v2/reviewDashboard.ts`
- `src/lib/interview-v2/reviewAgents.ts`

当前版本已经比旧版“只有报告入口”要好，但本质上仍然更接近**结构化展示页**，还不是**分析中台**。

### 2.1 现在已经有的东西

- 有顶部 KPI：本周训练次数、平均得分、薄弱维度数、7 日复训率
- 有核心结论 headline
- 有“多 Agent 洞察链”展示
- 有关键洞察卡片
- 有问题归因卡片
- 有建议看板
- 有历史训练记录列表
- 有“专项突破”跳转动作

### 2.2 现在最核心的问题

#### 问题 1：它现在展示的是“结论”，不是“证据链”

用户能看到：

- 当前最影响面试表现的是「结构化表达 (STAR)」
- 连续出现 4 次

但用户看不到：

- 这 4 次分别是哪 4 场
- 每场里具体是哪一句回答暴露了这个问题
- 它为什么被判成 STAR 问题而不是“项目表达不清”
- 这个问题在不同面试类型下是否稳定出现

这会让复盘中心失去说服力。用户会觉得它像一个“会写总结的模型”，而不是一个“会分析数据的系统”。

#### 问题 2：当前归因逻辑过度依赖旧报告里的文本维度和风险文案

`src/lib/interview-v2/reviewDashboard.ts` 当前的核心归因逻辑，本质是：

- 读 `report.dimensions`
- 读 `report.risks`
- 把低分维度计数
- 把风险文本计数
- 再生成薄弱点卡片

这意味着现在的“问题归因”更像：

- 对历史报告做二次聚合

而不是：

- 对原始对话、题目、评分事件、追问表现、岗位目标做一手分析

也就是说，当前复盘中心分析的是“别的报告已经加工过的数据”，不是“训练本身的数据”。

#### 问题 3：多 Agent 只是说明文案，不是可追溯的工作流

当前页面里“信号采集 / 诊断归因 / 建议策略 / 洞察叙事”四块，表达方向是对的，但它们目前更像 UI 概念层的解释卡片。

用户看不到：

- 每个 Agent 的输入是什么
- 每个 Agent 的输出是什么
- 当前这轮结论是谁给的
- 哪些结论是确定性的，哪些是模型判断
- 每个 Agent 的置信度是多少

如果多 Agent 不能对应到真实工作流，它最后会退化成“页面上的四张介绍卡”。

#### 问题 4：从洞察到行动的链路不完整

当前“立即补强”会跳到：

- `/interview?mode=targeted&topic=...&desc=...`

真实点击后可以进入专项训练，但当前训练页会出现：

- “等待补充岗位信息”

这说明复盘中心虽然给了动作入口，但没有把**岗位、级别、目标公司、最近失败样本、建议训练策略**一起带过去。

这不是闭环，只是“跳过去了”。

#### 问题 5：历史记录层价值偏低

当前历史训练记录里出现多条：

- `0 轮对话`
- `报告未生成`

这会直接拉低复盘中心的专业感。因为用户会感觉这里不是在帮我成长，而是在堆日志。

历史区当前缺少：

- 有效样本筛选
- 无效样本标记原因
- 关键节点回放
- 同题复练对比
- 同维度前后改善曲线

#### 问题 6：当前页面更像“静态总览”，缺少“钻取分析”

一个真正的数据中心，必须允许用户从“概览 -> 维度 -> 样本 -> 证据 -> 行动”不断下钻。

当前复盘中心主要是平铺：

- headline
- 指标卡
- 洞察卡
- 问题卡
- 计划卡

它缺少真正的数据中心能力：

- 时间窗口切换
- 岗位/公司/面试类型筛选
- 单维度趋势图
- 样本拆分
- 对比分析
- 追溯证据
- 改善结果验证

## 3. 我认为复盘中心应该被定义成什么

复盘中心应该被定义为：

> 一个围绕“求职者能力提升”构建的个人成长数据中台。

它不是报告仓库，不是弱项列表，不是建议集合页，而是：

- 把训练、面试、学习、专项练习的数据统一接起来
- 把这些数据转成可解释的能力画像
- 把能力画像转成问题诊断
- 把问题诊断转成个性化训练策略
- 把训练结果再反馈回画像，形成闭环

复盘中心最终要让用户形成一个感受：

> “这个系统真的知道我哪里差、为什么差、接下来该怎么练，而且它能证明我有没有变好。”

## 4. 复盘中心的正确产品结构

我建议把复盘中心重构为 6 个核心区块，不再只是单页平铺。

## 4.1 总览区：今天先告诉用户最重要的事

总览区回答三个问题：

1. 当前最该解决的问题是什么？
2. 当前整体状态是在变好还是变坏？
3. 现在最值得做的下一步动作是什么？

总览区应该包含：

- 一个主结论卡：当前最大拖分项 + 影响范围 + 优先级
- 一个趋势卡：最近 7/14/30 天总评分、稳定性、复训完成率变化
- 一个动作卡：今天最值得做的一件事
- 一个可信度卡：当前结论基于多少有效样本、覆盖哪些场景

这里不能只是文案，要带上：

- 有效样本数
- 最近时间范围
- 涉及场景
- 是否达到“可下结论”的样本门槛

## 4.2 诊断区：把“差在哪”讲清楚

诊断区不应该只是薄弱点列表，而应该是一个问题树。

建议展示成三层：

1. 能力层
   - 结构化表达
   - 技术深度
   - 量化证明
   - trade-off 判断
   - 项目拆解
   - 业务理解

2. 原因层
   - 不会说
   - 没素材
   - 没结构
   - 理解不深
   - 不会举证
   - 一紧张就散

3. 影响层
   - 追问里容易失分
   - 项目题说不深
   - 行为题不成体系
   - 系统设计题没有权衡
   - 一旦追到指标就断

这个区块的目标不是告诉用户“你差”，而是告诉用户：

> 你的问题是怎么形成的，它在面试里怎么表现出来，它会带来什么后果。

## 4.3 证据区：每个结论都要能往下挖

证据区是复盘中心和普通“AI 总结页”拉开差距的关键。

每条洞察都应该支持下钻到证据：

- 命中的会话
- 命中的问题
- 对应回答片段
- 对应追问片段
- 评分维度变化
- Agent 判定原因

建议每条证据至少包含：

- 来源会话
- 面试类型
- 岗位 / 公司 / 轮次
- 问题标题
- 用户回答摘录
- 命中问题标签
- 为什么命中
- 改写建议

这一区是“证明系统不是瞎说”的关键。

## 4.4 策略区：从洞察变成训练方案

策略区不能只给“去专项突破”四个字，而要形成完整动作单。

每条策略至少回答：

1. 练什么
2. 为什么练这个
3. 练到什么程度算达标
4. 推荐用什么训练形式
5. 练完怎么验证是否有效

建议一个策略卡包含：

- 目标问题
- 推荐训练方式
  - 专项问答
  - 行为题 STAR 复练
  - 项目深挖
  - 模拟面试
  - 学习中心补知识
- 推荐样本数
- 通过标准
- 预计压降的风险
- 训练后的验证入口

## 4.5 改善区：告诉用户是否真的变好了

这是当前页面最缺的部分。

复盘中心不能永远只诊断问题，还要告诉用户：

- 这个问题上周很严重，这周下降了多少
- 哪些问题压下去了
- 哪些问题反弹了
- 哪些建议已经执行了
- 哪些建议有效，哪些建议无效

改善区应该具备：

- 维度改善趋势
- 同类题前后对比
- 历史行动完成情况
- “建议执行后是否有效”的闭环反馈

## 4.6 数据探索区：像数据中心一样允许筛选和对比

至少支持：

- 时间范围：7 天 / 14 天 / 30 天 / 全部
- 面试类型：模拟面试 / 专项训练 / 学习测验
- 岗位：后端 / 前端 / Java / 运营等
- 公司：字节 / 阿里 / 腾讯 / 自定义
- 维度：表达 / 深度 / 量化 / trade-off / 项目 / 行为题
- 样本状态：有效 / 无效 / 中断 / 未生成报告

有了筛选之后，复盘中心才会真正像中台。

## 5. 复盘中心的正确多 Agent 设计

我建议不要只保留现在的 4 个 Agent，而是扩成**7 个真实 Agent + 1 个调度器**。

## 5.1 Orchestrator Agent（调度编排 Agent）

职责：

- 决定这次复盘应该跑哪些 Agent
- 判断当前样本是否足够支撑结论
- 控制哪些输出可以直接展示，哪些需要降级
- 统一产出最终 dashboard snapshot

输入：

- 用户 ID
- 查询条件
- 最近样本范围

输出：

- 一次复盘任务的执行图
- 各 Agent 调用顺序
- 最终合并后的视图数据

## 5.2 Signal Agent（信号采集 Agent）

职责：

- 从所有原始数据源拿干净信号

输入：

- interviewSession
- messages
- report
- weakness
- learningProgress
- documentInterviewSession
- userAnswerScore

输出：

- 有效样本列表
- 无效样本列表
- 基础统计指标
- 样本标签

重点：

- 要区分“有效样本”和“脏样本”
- `0 轮对话`、中断、报告缺失不能和正常样本混算

## 5.3 Evidence Agent（证据抽取 Agent）

职责：

- 从原始对话和评分记录中抽“能证明结论”的证据

输入：

- 问题文本
- 用户回答
- 追问链
- 维度评分
- 风险标签

输出：

- 证据片段
- 证据所属问题
- 证据命中的能力维度
- 证据置信度
- 原始摘录 + 解释 + 建议改写

重点：

- 这个 Agent 决定复盘中心有没有说服力

## 5.4 Diagnosis Agent（诊断归因 Agent）

职责：

- 把“表象问题”归因成“根因问题”

输入：

- 有效样本
- 证据片段
- 维度低分记录
- 岗位目标

输出：

- 核心问题列表
- 每个问题的成因
- 问题的影响范围
- 问题稳定性

重点：

- 不能只报“STAR 差”
- 要区分：
  - 不会 STAR
  - 有 STAR 结构但缺少量化证据
  - 有 STAR 但没突出个人作用
  - 会讲经历但不适配岗位语境

## 5.5 Strategy Agent（策略生成 Agent）

职责：

- 把问题变成训练动作单

输入：

- 核心问题
- 用户目标岗位
- 已做过的训练
- 可用训练资源

输出：

- 训练动作
- 推荐训练形式
- 顺序
- 完成标准
- 复验方式

重点：

- 不是“给建议”，而是“给动作设计”

## 5.6 Drill Agent（专项生成 Agent）

职责：

- 根据策略动态生成下一次训练内容

输入：

- 目标问题
- 岗位信息
- 过去失败样本
- 练习目标

输出：

- 下一个专项训练入口
- 训练脚本
- 重点追问
- 评分规则

重点：

- 复盘页点“立即补强”时，不应该只传一个 topic 文本
- 应该生成一条真正定制过的训练任务

## 5.7 Progress Agent（改善验证 Agent）

职责：

- 判断建议执行后是否有效

输入：

- 策略执行记录
- 执行后的新样本
- 旧问题基线

输出：

- 是否改善
- 改善幅度
- 是否反弹
- 是否需要继续追击

重点：

- 没有这个 Agent，复盘中心永远只是“告诉用户下一步做什么”
- 有了它，系统才真正开始“陪用户变好”

## 5.8 Narrative Agent（洞察叙事 Agent）

职责：

- 把结构化分析结果组织成用户看得懂的结论

输入：

- 指标
- 问题
- 证据
- 策略
- 改善状态

输出：

- headline
- 洞察卡片
- 总结文案
- 用户侧说明

重点：

- Narrative Agent 只能负责“说清楚”
- 不能负责“决定事实”

## 6. 正确的 Agent 协作顺序

复盘中心每次刷新，不应该是一坨同步逻辑直接拼页面，而应该是这样的流水线：

1. Orchestrator 收到请求
2. Signal Agent 拉取并清洗样本
3. Evidence Agent 抽取可追溯证据
4. Diagnosis Agent 生成问题树
5. Strategy Agent 输出动作单
6. Drill Agent 生成下一次训练入口
7. Progress Agent 判断历史建议是否有效
8. Narrative Agent 把结果转成页面展示
9. Orchestrator 统一合并成 dashboard snapshot

页面只负责渲染，不负责推理。

## 7. 复盘中心应该使用的数据模型

当前最大问题之一是：复盘中心高度依赖旧 `report` 的字符串字段。

要变成数据中台，必须把中间结果结构化。

建议新增或沉淀出下面几类实体：

## 7.1 ReviewSnapshot

表示某次打开复盘中心时的聚合结果。

建议字段：

- `id`
- `userId`
- `timeRange`
- `sampleCount`
- `validSampleCount`
- `headline`
- `confidence`
- `generatedAt`

## 7.2 ReviewIssue

表示一个被稳定识别的问题。

建议字段：

- `id`
- `snapshotId`
- `name`
- `category`
- `severity`
- `frequency`
- `stability`
- `impactScore`
- `rootCause`
- `description`
- `status`

## 7.3 ReviewEvidence

表示支撑问题的证据。

建议字段：

- `id`
- `issueId`
- `sessionId`
- `questionId`
- `messageId`
- `excerpt`
- `reason`
- `dimension`
- `confidence`

## 7.4 ReviewAction

表示一条可执行建议。

建议字段：

- `id`
- `issueId`
- `title`
- `description`
- `actionType`
- `targetPath`
- `payload`
- `successMetric`
- `expectedOutcome`

## 7.5 ReviewActionExecution

表示建议是否被执行，以及执行效果。

建议字段：

- `id`
- `actionId`
- `userId`
- `startedAt`
- `completedAt`
- `resultStatus`
- `improvementScore`
- `notes`

## 8. 页面应该怎么重做

## 8.1 页面布局建议

建议从现在的“单页长列表”改为：

- 顶部筛选条
- 第一屏：全局结论 + 可信度 + 今日建议
- 第二屏：问题树 + 关键趋势
- 第三屏：证据台
- 第四屏：行动板
- 第五屏：改善验证
- 第六屏：历史与对比

## 8.2 第一屏一定要回答的内容

第一屏建议固定回答：

- 当前最大问题
- 当前最明显进步
- 当前建议优先动作
- 当前结论可信度

例如：

- 最大问题：STAR 结构稳定性不足
- 最大进步：追问承接能力较上周提升 22%
- 今日最值得练：项目型行为题量化表达
- 可信度：高（最近 14 天 18 个有效样本）

## 8.3 问题卡不能只有描述，要有结构

每张问题卡建议固定结构：

- 问题名
- 严重程度
- 最近出现次数
- 最近一次出现时间
- 影响范围
- 根因判断
- 典型证据
- 推荐动作

## 8.4 证据台要像“分析台”

建议支持：

- 按问题筛选证据
- 按岗位筛选证据
- 按题型筛选证据
- 打开原始对话上下文
- 对比“原回答 vs 推荐改写”

## 8.5 行动板必须和训练系统打通

行动板每个按钮都不该只是跳页面，而应该携带：

- 当前问题 ID
- 训练目标
- 推荐模式
- 推荐题型
- 推荐难度
- 岗位上下文
- 最近失败样本摘要
- 验证标准

这样进入下一页后，系统才能真正无缝接上。

## 9. 我对当前实现的直接建议

这里不谈阶段，不谈先后，只谈应该一次性做到什么样。

## 9.1 不要再把“复盘结果”只建立在旧 Report 文本上

当前 `reviewDashboard.ts` 的逻辑更多是在聚合旧 `report.dimensions` 和 `report.risks`。

建议改成：

- 以原始会话、问题、回答、追问、评分事件为主
- `report` 只作为二级参考
- 让复盘中心直接面对一手训练数据

## 9.2 “多 Agent 洞察链”必须变成真实可追踪的执行结果

不要只展示：

- 信号采集 Agent
- 诊断归因 Agent

而要展示这次运行的结果，例如：

- 信号采集：本次纳入 18 个有效样本，剔除 6 个无效样本
- 证据抽取：提取到 23 条有效证据，覆盖 4 个问题
- 诊断归因：确认 2 个核心根因问题
- 策略生成：生成 3 条训练建议
- 改善验证：其中 1 条建议已验证有效

## 9.3 “问题归因”要升级成问题树，不是问题列表

当前的问题列表还太扁平。

建议改成：

- 能力维度
- 根因
- 影响
- 样本证据
- 建议动作

## 9.4 “建议看板”必须从文案卡升级成任务卡

每条建议卡必须包含：

- 为什么做
- 做什么
- 怎么做
- 做到什么算完成
- 完成后怎么验证

## 9.5 历史记录不能继续堆“0 轮对话 / 报告未生成”

建议：

- 默认只展示有效训练记录
- 无效记录单独归档为“无效样本”
- 给出无效原因：
  - 提前退出
  - 回答轮次不足
  - 报告生成失败
  - 样本不够

## 9.6 复盘中心需要“可信度”表达

所有洞察都要带：

- 样本数
- 时间范围
- 覆盖场景
- 置信等级

没有这个，用户很难信任系统。

## 9.7 复盘中心必须支持岗位语境

当前“立即补强”进入专项后还会提示“等待补充岗位信息”，这是明显断链。

复盘中心生成动作时，必须把下面这些上下文一起带过去：

- 目标岗位
- 目标公司
- 当前级别
- 对应薄弱点
- 推荐题型
- 训练目标

## 10. 面向模型的一次性交付要求

如果把这份文档交给模型执行，我建议直接按下面的目标实现，不要再做“展示优化型复盘页”。

### 目标一：把复盘中心升级成真正的数据中台

要求：

- 不再只是展示旧报告聚合
- 直接分析一手训练数据
- 支持筛选、下钻、对比、追溯

### 目标二：建立真实多 Agent 工作流

要求：

- 至少具备 Orchestrator、Signal、Evidence、Diagnosis、Strategy、Drill、Progress、Narrative
- 每个 Agent 都有明确输入输出
- 页面可以看到 Agent 的真实工作结果，而不是静态说明

### 目标三：每条洞察都必须带证据

要求：

- 任意“问题”都能展开看到证据
- 任意“建议”都能回到问题来源
- 任意“改善结果”都能看到前后对比

### 目标四：从洞察到训练形成闭环

要求：

- 从复盘中心点动作进入训练时，必须自动带上完整上下文
- 训练完成后，能自动回流复盘中心
- 复盘中心能识别建议是否有效

### 目标五：把“变好”可视化

要求：

- 能看到问题是否下降
- 能看到建议是否执行
- 能看到执行后是否改善
- 能看到哪类训练最有效

## 11. 最后一句判断

你现在这个复盘中心，方向已经对了，尤其是“多 Agent 洞察链”和“别再只是报告入口”这两个判断是对的。

但当前版本本质上还是：

> 一个把历史结果重新包装成 dashboard 的页面

而你真正要做的应该是：

> 一个围绕用户成长闭环构建的分析中台，既能诊断问题，也能证明问题，还能推动行动，并验证行动是否有效。

只要把“证据链、问题树、动作闭环、改善验证、多 Agent 真工作流”这五件事做实，复盘中心才会从“看起来很聪明”变成“真的能帮用户变好”。
