# Requirements: 面面吧 V2.0 重设计

**Defined:** 2026-05-13
**Core Value:** 围绕目标岗位与目标公司，让用户在真实轮次模拟中获得可信证据、明确改进动作和是否 ready 的判断。

## v1 Requirements

### 建模与配置

- [ ] **MODEL-01**: 用户可以基于目标公司、岗位、级别、JD 和简历创建统一候选画像。
- [ ] **MODEL-02**: 系统可以根据候选画像输出岗位能力权重、目标轮次和公司流程假设。
- [ ] **MODEL-03**: 用户在阶段面试、全流程面试和专项训练之间可以共享同一份候选画像与上下文。

### 编排与状态机

- [ ] **ORCH-01**: 系统可以使用统一的 `plan / stage / round` 状态机编排所有面试流程。
- [ ] **ORCH-02**: 系统可以为每轮记录主目标、通过条件、当前面试官角色和后续风险。
- [ ] **ORCH-03**: 系统可以在正常结束、淘汰、中断、超时等场景下稳定收口状态并支持恢复。

### 阶段面试

- [ ] **STAGE-01**: 用户可以发起阶段面试并获得与目标岗位匹配的首题与追问。
- [ ] **STAGE-02**: 系统可以在阶段面试中实时生成证据切片、维度评分和下一问策略。
- [ ] **STAGE-03**: 用户刷新或重新进入后可以恢复当前房间、消息、计时和进度。

### 全流程面试

- [ ] **FLOW-01**: 用户可以创建符合目标公司和岗位的全流程面试计划并查看轮次列表。
- [ ] **FLOW-02**: 每个阶段都可以绑定独立面试官角色、重点维度、评审团规则和通过条件。
- [ ] **FLOW-03**: 每轮结束后系统可以给出通过或未通过结果、加一项、优化点和下一步动作。
- [ ] **FLOW-04**: 系统可以根据前一轮风险与结果决定推进、补强或终止后续阶段。

### 算法题与代码面

- [ ] **CODE-01**: 系统可以在满足条件时触发独立算法题环节并跳转到专用 coding room。
- [ ] **CODE-02**: 用户可以运行、提交、结束算法题，并把结果写回当前流程。
- [ ] **CODE-03**: 算法题题面与引导语只能引用经过证据校验的真实项目、岗位和公司上下文。

### 评估与报告

- [ ] **EVAL-01**: 系统可以对每轮回答抽取证据片段并映射到明确维度。
- [ ] **EVAL-02**: 多个 Judge 可以独立评分，并经过校准输出最终分数、理由与置信度。
- [ ] **EVAL-03**: 当证据不足或分歧过大时，系统可以明确降置信或拒绝伪造正式结论。
- [ ] **EVAL-04**: 面试结束后系统可以立即返回骨架报告，并继续补齐证据与长文本解释。

### 复盘与行动闭环

- [ ] **LOOP-01**: 复盘中心可以聚合多场记录并识别稳定弱项、波动问题和已修复问题。
- [ ] **LOOP-02**: 系统可以把复盘问题直接回流成专项训练、同题重答、学习卡片或项目举证动作。
- [ ] **LOOP-03**: 学习中心可以按岗位与弱项输出高分案例、反例和知识卡，而不是泛内容浏览。

### 成长与 Ready 判断

- [ ] **GROW-01**: 成长档案可以持续展示维度趋势、已修复问题和回退风险。
- [ ] **READY-01**: 系统可以给出目标岗位的 readiness、轮次风险和最短补强路径。

### 平台与可观测性

- [ ] **PLAT-01**: 系统可以记录关键 Agent 调用、输入输出摘要、延迟、成本和置信度。
- [ ] **PLAT-02**: 系统可以回放一次计划、轮次或报告生成链中的关键决策依据。

## v2 Requirements

### 实时与通知

- **REAL-01**: 全流程面试默认支持语音加视频的实时形态，并保留低等待感知。
- **REAL-02**: 系统在用户长时间未输入时可以提供温和提示与状态解释。
- **NOTI-01**: 系统可以覆盖邀约、提醒、淘汰和下一步动作的站内或站外通知。

### 面经与内容运营

- **EXP-01**: 面经采集链可以支持多来源清洗、去重、审核与质量评分。
- **KNOW-01**: 知识中心建设评审团可以半自动生产岗位化知识卡、反例和学习路径。

### 商业化与后台

- **OPS-01**: 管理后台可以管理岗位适配器、流程模板、知识资产和面经质量。
- **BIZ-01**: 订阅、服务版和冲刺营等商业化能力纳入统一权限模型。

## Out of Scope

| Feature | Reason |
|---------|--------|
| 一次性接入大量非技术岗位 | 当前先打穿技术岗通用能力内核和岗位适配器 |
| 先做独立移动端 App | Web brownfield 仍未稳定，不适合同时分散资源 |
| 全量微服务拆分 | 当前更需要清晰边界与迁移护栏，而不是更多服务部署复杂度 |
| 没有证据支撑的自动高分报告 | 与产品信任和 V2 护城河方向冲突 |
| 社区、社交 feed、泛内容平台化扩展 | 不属于当前 V2 核心交付范围 |
